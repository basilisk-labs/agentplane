AgentPlane started as a deterministic CLI workflow layer for repository-native task execution. This roadmap defines how it evolves from baseline discipline to autonomous multi-agent execution while preserving auditability and policy control.

<!-- truncate -->

## Why Now

Three constraints shape the roadmap:

- Teams want stronger automation without losing control of repository state.
- Prompt quality and execution quality must be treated as system design, not ad-hoc tuning.
- Extensibility must be explicit so teams can evolve agents, tools, and scenarios without forking core behavior.

The roadmap is structured to reduce risk by sequencing foundation, contracts, extension, and autonomy in that order.

## Version-by-Version Roadmap

## 0.1 Foundation and Baseline Workflow

- Establish the initial AgentPlane CLI baseline and project structure.
- Define the first end-to-end task lifecycle for local execution.
- Introduce core task tracking, verification logging, and export conventions.

Operational outcome:

- Teams can run a deterministic local workflow with explicit task state and auditable transitions.

## 0.2 Architecture and Codebase Formalization

- Formalize architecture boundaries and core codebase contracts.
- Modularize subsystems to keep scaling simple and predictable.
- Formalize operational processes at the CLI level.

Operational outcome:

- Lower coupling and clearer responsibility boundaries across CLI, policy, and backend layers.

## 0.3 Prompt System Improvements

- Improve agent prompts for better determinism and execution quality.
- Formalize process contracts at the agent-prompt level.
- Deliver a dedicated Claude Code-compatible version.

Operational outcome:

- More stable agent behavior under real repository constraints and clearer role-level execution contracts.

## 0.4 Recipes and Plugin Extension Model

- Support custom agents, tools/SKILLS, and scenarios through recipes.
- Define recipe packaging, versioning, and compatibility rules.
- Add recipe discovery, installation, and validation flows in CLI.

Operational outcome:

- Extension becomes first-class: teams can introduce domain-specific capability without patching core internals.

## 0.5 Agentplane Runner

- Build Agentplane Runner for autonomous task execution.
- Enable coordinated swarm-style execution by custom agents.
- Provide orchestration primitives for reliable multi-agent runs.

Operational outcome:

- Autonomous execution moves from manual orchestration to controlled, policy-aware runtime behavior.

## Operational Impact for Users

Across versions, users should expect:

- Faster execution loops with clearer policy boundaries.
- Better reproducibility for planning, implementation, and verification paths.
- Lower overhead when introducing custom automation patterns.
- Stronger traceability from intent to repository mutation.

## Risks and Open Questions

Primary delivery risks:

- Over-complex extension surfaces can reduce adoption if plugin contracts are not strict enough.
- Autonomous execution can increase failure blast radius if guardrails are not enforced at runtime.
- Prompt formalization must remain aligned with real execution constraints, not only prompt style.

Open questions that will shape implementation detail:

- Runner execution model under mixed local/remote tool constraints.
- Cross-recipe compatibility guarantees and version pinning strategy.
- Default observability contract for multi-agent runs.

## What to Expect Next

Near-term focus remains on roadmap execution through explicit contracts:

- finalize editorial and technical contract surfaces for roadmap communication,
- ship incremental milestones with deterministic verification,
- publish follow-up release notes as each roadmap segment lands.

The direction is straightforward: stronger determinism, safer extensibility, and autonomous execution without sacrificing governance.
