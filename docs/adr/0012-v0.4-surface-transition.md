# ADR 0012: v0.4 Surface Transition

## Status

Accepted

## Date

2026-04-30

## Context

ADR 0011 froze the public `0.3.x` surface in `FREEZE.v0.3.md` so patch releases
could harden internals without drifting command, config, task-document, or
package contracts.

The `0.4.0` line intentionally leaves that patch line. Keeping the root
`FREEZE.v0.3.md` artifact in a `0.4.x` checkout makes release parity ambiguous:
the package version is no longer in the frozen `0.3.x` range, but the checkout
still appears to claim the v0.3 freeze as its active surface contract.

The final `0.3.x` state is preserved on the `0.3` branch and by the `v0.3.29`
tag.

## Decision

Retire the root `FREEZE.v0.3.md` artifact from the `0.4.x` development and
release line.

The v0.3 freeze remains authoritative only for:

1. the `0.3` branch,
2. the `v0.3.29` tag, and
3. ADR 0011 as historical context for upgrade and migration work.

The v0.4 line uses release notes, ADRs, tests, and CLI/runtime compatibility
checks as its active surface record until a new freeze artifact is explicitly
created.

## Consequences

1. `release:parity` can distinguish active `0.4.x` releases from the frozen
   `0.3.x` line.
2. v0.4 can intentionally change modular prompt assembly and recipe-extension
   surfaces without pretending to remain inside the `0.3.x` freeze.
3. Upgrade documentation still has a stable v0.3 reference point through the
   `0.3` branch and `v0.3.29` tag.

## Follow-up

Create a new freeze artifact only after the v0.4 surface stabilizes enough that
patch releases need the same kind of explicit contract guard used for `0.3.x`.
