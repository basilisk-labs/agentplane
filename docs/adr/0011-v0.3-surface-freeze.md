# ADR 0011: v0.3 Surface Freeze

## Status

Accepted

## Date

2026-04-24

## Context

The `0.3.x` line has accumulated the task lifecycle, git workflow, docs workflow, package
boundary, and config schema needed for the next development line. The remaining risk before v0.4 is
not a missing feature; it is contract drift:

1. CLI commands, `.agentplane/` task documents, and config schema fields can drift independently.
2. Packaging can accidentally publish implementation artifacts outside the intended runtime bundle.
3. Lifecycle commands can keep working while their close/finish vocabulary becomes ambiguous.
4. Internal cleanup can accidentally look like public API removal when it is not tied to an explicit
   compatibility decision.

`FREEZE.v0.3.md` records the current package version, command inventory, lifecycle map, close
taxonomy, config schema pointer, package boundary, CLI bundle hash, and the checkout used to
generate the artifact.

## Decision

Freeze the public v0.3 surface for the rest of the `0.3.x` line.

Frozen surfaces:

1. Public CLI command names and their command-spec entry points under
   `packages/agentplane/src/commands/**/*.command.ts`.
2. Project-local `.agentplane/` task document shape and lifecycle command route.
3. Config schema fields exposed through `@agentplaneorg/core/config` and
   `@agentplaneorg/core/schemas`.
4. Package artifact boundary for `agentplane`, including the bundled CLI runtime and explicitly
   whitelisted bin/assets/readme/license files.
5. Task lifecycle and close taxonomy documented in `docs/user/task-lifecycle.mdx` and
   `docs/developer/close-taxonomy.mdx`.
6. Repository-local schema/example contracts in the private `@agentplane/spec` package
   (`packages/spec/schemas/**` and `packages/spec/examples/**`). This package remains
   `private: true` and `0.0.0` in v0.3; freezing it here documents compatibility intent inside the
   repository and does not make it a published package API.

Allowed during `0.3.x`:

1. Internal refactors that preserve the frozen public surface.
2. Test, lint, routing, and packaging guard improvements.
3. Documentation clarifications that describe already-shipped behavior.
4. Bug fixes that keep command names, config fields, task document shape, and package boundary
   compatible.

Not allowed during `0.3.x` without a new ADR or explicit compatibility exception:

1. Removing or renaming public CLI commands.
2. Removing config fields or narrowing public schema compatibility.
3. Changing `.agentplane/` task document semantics without a migration path.
4. Publishing additional runtime implementation files as package API by accident.
5. Reclassifying task close commands and hosted PR close commands as interchangeable lifecycle
   actions.
6. Publishing or versioning `@agentplane/spec` as a public package without a separate package-boundary
   decision.

## Consequences

### Positive

1. v0.3 patch releases can continue hardening internals without changing the user-facing surface.
2. v0.4 work has a concrete migration pointer instead of relying on memory or audit notes.
3. Package publication, config schema, and lifecycle docs now have one freeze artifact to check.

### Negative

1. Some compatibility shims may remain until v0.4 even when the implementation no longer needs them.
2. Patch releases must distinguish public-contract fixes from internal cleanup before merging.
3. The freeze artifact must be updated or retired when the project leaves the `0.3.x` line.

## Migration Pointer

Use `FREEZE.v0.3.md` as the starting inventory for v0.4 planning. A v0.4 migration ADR should say
which frozen surfaces are intentionally changed, which shims are removed, and which checks prove that
existing `0.3.x` projects can upgrade safely.

## Follow-up

Reopen this decision when one of these becomes true:

1. A `0.4.0` release candidate is planned.
2. A patch release needs to change a frozen CLI/config/task/package surface.
3. `FREEZE.v0.3.md` no longer matches the package version or command inventory.
