# Hermes Agentplane recipe

## What does it do?

The Hermes Agentplane recipe adds prompt policy for repositories that dispatch Agentplane work
through Hermes Kanban worker lanes.

It keeps the boundary explicit:

- Hermes owns dispatch/run lifecycle: queue, assignee, claim, run id, retries, crash detection,
  comments, dashboard, and human handoff.
- Agentplane owns engineering task lifecycle: task README/frontmatter, plan approval,
  branch/worktree, verification, PR/integration, finish, and ACR evidence.

## When should I use it?

- Hermes Kanban is the durable operational board for a repository.
- Multiple agents or lanes may work against the same repository.
- `branch_pr` should protect checkout, branch, verification, and integration boundaries.
- Hermes cards should close only after Agentplane terminal evidence validates.

## When should I not use it?

- The repository does not run Hermes Kanban.
- The task is a single local direct-mode helper run.
- The Hermes runtime cannot provide lifecycle callbacks or an external worker lane adapter.

## Command to apply

```bash
agentplane recipes install hermes-agentplane
agentplane recipes add hermes-agentplane
agentplane recipes explain hermes-agentplane
```

For new repositories that should avoid Codex as the default task runner, initialize with the Hermes
tool profile:

```bash
agentplane init --quick --tool hermes
```

That profile keeps the normal Agentplane policy gateway and configures `agentplane task run` through
the `custom` runner adapter with `runner.custom.command=["hermes","agentplane","run"]`.

## Expected adapter commands

```bash
agentplane init --tool hermes --workflow branch_pr --yes
agentplane hermes doctor --json
agentplane hermes enqueue <task-id> --board <board> --assignee agentplane-coder --json
agentplane hermes supervise <task-id> --execute-step --json
agentplane task run <task-id>
agentplane hermes lifecycle comment --body '<structured projection>' --json
agentplane hermes lifecycle complete --body 'Agentplane terminal evidence validated' --json
```

`--tool hermes` configures `runner.default_adapter=hermes`, disables IDE-specific sync, and writes
`runner.custom.command=["hermes","agentplane","run"]`. The Hermes runtime image must provide that
command; Agentplane does not fall back to Codex for Hermes task execution.

## Runtime environment

Hermes worker lanes must pass these variables to Agentplane-side commands:

```bash
AGENTPLANE_HERMES_LANE_REGISTRY=/opt/agentplane/lane-registry.json
HERMES_KANBAN_TASK=<hermes-card-id>
HERMES_KANBAN_BOARD=<board>
HERMES_KANBAN_RUN_ID=<run-id>
HERMES_KANBAN_WORKSPACE=<repo-workspace>
HERMES_KANBAN_CLAIM_LOCK=<claim-lock>
```

`agentplane hermes doctor --json` reports missing Hermes environment values and lane-registry
state. The Agentplane side uses `AGENTPLANE_HERMES_LANE_REGISTRY` as the registry input.
Task-runner invocations also export `AGENTPLANE_RUNNER_TASK_ID` when the run target is an Agentplane
task, so Hermes launchers can bind the external worker run back to the canonical task id.

## Visibility contract

`agentplane hermes enqueue` and `agentplane hermes supervise` return a
`hermes_comment_projection` packet. Hermes plugins should post that packet before block or
complete transitions. The packet includes:

- Agentplane task id, revision, status, verification state, and current route phase.
- Route blockers and the next typed action.
- Runner visibility commands: `agentplane task run status`, `inspect`, and event `logs`.
- Evidence references for the task README, ACR, and runner artifacts.
- The authority boundary that keeps Hermes status projection-only.

## Expected artifacts

```text
.agentplane/tasks/<task-id>/README.md
.agentplane/tasks/<task-id>/acr.json
.agentplane/recipes/packages/hermes-agentplane/
```

## Safety model

The recipe does not make Hermes a task backend for Agentplane. Hermes board state is a projection.
Agentplane task state remains canonical and Git-visible.
