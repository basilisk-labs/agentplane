# Use with coding agents

Agentplane is agent-agnostic and LLM-agnostic. Any repository-capable coding agent can use the same
repository harness when it can read files, run commands, edit code, and respect `AGENTS.md`.

The split is simple:

| Actor      | Responsibility                                                                      |
| ---------- | ----------------------------------------------------------------------------------- |
| Human      | Set the task, approve guardrails, observe progress, and review evidence.            |
| Agent      | The agent writes code, uses the CLI, follows `AGENTS.md`, and records verification. |
| Agentplane | Preserve task state, context, verification, PR artifacts, ACR, and trace.           |

## The default handoff

Use this pattern for terminal agents, editor agents, and hosted coding agents:

1. Install and initialize Agentplane once.
2. Give the agent a specific task and scope.
3. Tell the agent to follow the repo policy gateway.
4. Review the task README, diff, verification evidence, and ACR before merge.

The human does not need to drive every lifecycle command manually. The agent should use the CLI as
part of the harness.

```text
Use Agentplane as the harness. Read AGENTS.md, create or continue the task, keep edits inside the
approved scope, run the verification contract, record the result, and stop for review when scope or
risk changes.
```

## If your agent runs in a terminal

This covers terminal agents and custom local wrappers.

- Start the agent from the repository root.
- Keep the task boundary in Agentplane, not in terminal scrollback.
- Ask the agent to run focused checks and record verification.
- Review Git diff plus `.agentplane/tasks/<task-id>/`.

Use the terminal agent's normal invocation. Agentplane does not require a provider-specific command.

## If your agent runs in an editor

This covers editor agents that can work inside an initialized repository.

- Open the repository after Agentplane is initialized.
- Put the approved task and scope in the agent prompt.
- Keep long-lived project rules in `AGENTS.md` or `CLAUDE.md`, not only in editor chat.
- Ask the agent to record verification before asking for review.

Editor private context is not durable evidence. The reviewable state is the Git diff and Agentplane
artifacts.

## If you use `branch_pr`

Use `branch_pr` when implementation should stay isolated until review and hosted checks pass.

- The task branch or worktree owns implementation commits.
- PR artifacts live under `.agentplane/tasks/<task-id>/pr/`.
- Integration happens after verification and hosted checks are stable.
- The final review should point to the implementation commit, verification note, and PR result.

`branch_pr` adds process overhead. Use it when isolation, review, or branch protection matter.

## If GitHub Actions is the hosted gate

GitHub Actions should remain the CI authority. Agentplane gives CI a repo-local explanation of the
agent work:

- approved task scope;
- verification evidence;
- PR artifacts;
- Agent Change Record;
- final finish state.

For reusable team workflows, add an ACR check around your repository's task-id convention. Keep
branch protection and required checks in GitHub.

## Review checklist

Before merging AI-written work, confirm:

- The task exists under `.agentplane/tasks/<task-id>/`.
- The task scope matches the diff.
- Verification evidence names the checks that passed.
- Unexpected scope changes are called out and re-approved.
- ACR exists when the review flow requires it.
- The implementation commit or PR is linked from task evidence.
