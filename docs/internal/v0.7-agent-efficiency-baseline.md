# AgentPlane 0.7 compatibility and agent-efficiency baseline

This baseline is the pre-refactor control for the 0.7 program. It has two distinct provenance
anchors and must not collapse them into one claim:

- **Published compatibility reference:** Git tag `v0.6.24` at
  `30f62b82dff28909dcb3ccc2ace2bf3e356203bb`, plus the exact npm metadata and package inventories
  captured in `scripts/baselines/v0.6.24-compatibility-contract.json`.
- **Efficiency reference:** task-parent `main` at
  `1a702e160ba9f0efe7067f2a22fc008defc89ffb`. This commit contains post-`v0.6.24` changes, so
  `scripts/baselines/agent-efficiency-pre-v0.7-main.json` is a pre-0.7 main baseline, not a claim
  about the published package.

The compatibility check is intentionally network-free. Registry evidence is captured once with
explicit provenance and then reviewed as an immutable historical input. The check compares the
working tree against the committed exact-main contract and exposes the already-existing delta from
the published release.

## Golden scenarios

`scripts/bench/agent-efficiency-fixtures.json` defines exactly ten scenarios:

1. direct code task;
2. `branch_pr` task;
3. context assimilation;
4. stale `HEAD`;
5. approval required;
6. missing knowledge;
7. evaluator rework;
8. out-of-scope write;
9. adapter crash;
10. Hermes one-step supervision.

Each scenario carries expected quality, safety, lifecycle, episode, and evidence controls. These
controls define the comparison case; they are not presented as observed telemetry. Cost comparisons
are valid only when the controls remain equivalent. A cheaper failed or unsafe run is not an
improvement.

## Measurements

The harness defines a provenance cell for every required measurement and keeps structural values
separate from timing diagnostics:

- input, output, and reasoning tokens by role;
- number of LLM episodes and agent lifecycle calls;
- prompt boilerplate bytes and duplicated serialized bytes;
- preparation latency and context-search latency;
- knowledge retrieval hits and gaps;
- success, rework, block, context-gap, and scope-violation outcomes;
- observed versus agent-claimed evidence.

Numerical values are accepted only when a committed source artifact supplies them. Expected episode
and lifecycle traces remain controls; their lengths are not counted as observed calls. A missing
token, retrieval, outcome, or evidence signal is represented as `null` plus an
`observability_gap`; it is never estimated from bytes and never coerced to zero. This preserves the
difference between “measured zero” and “not observable at the pre-0.7 main anchor”.

The primary optimization target is cost per verified result at equivalent quality and safety. Token
count alone is diagnostic. Timing is also diagnostic: repeated runs report distributions rather
than enforcing a hard threshold from one machine.

## Determinism and comparison rules

For the same fixture registry and tracked source artifacts:

- normalized structural output must be byte-identical across repeated runs;
- timestamps, absolute paths, host details, and repeated `duration_ms` samples stay outside the
  structural digest;
- a known baseline value becoming unknown is a regression;
- an unknown baseline value becoming measured is an observability improvement;
- a structural cost increase above the committed 10% tolerance fails only for a quality-equivalent
  scenario;
- a safety or verified-success regression fails regardless of cost.

Expected protocol changes and quality/evidence improvements are reported as non-comparable rather
than being forced through the cost ratchet. Preparation, context-search, and wall-clock latency stay
diagnostic and do not receive a single-machine ceiling.

The tolerance is a ratchet guard for later 0.7 slices, not evidence that the baseline itself is
better or worse than `v0.6.24`.

## Commands

Capture or inspect the exact compatibility surface:

```bash
bun run bench:compatibility:capture
bun run bench:compatibility:check
```

Measure or check the agent-efficiency control:

```bash
bun run bench:agent-efficiency
bun run bench:agent-efficiency:check
```

Run the measurement twice before accepting a baseline change. Review both the normalized structural
digest and the per-scenario provenance. Do not update either frozen baseline automatically in CI.

## Compatibility surface

The frozen compatibility contract covers all 243 command entries in the structural
`help --json --all` projection, positional arguments, options, global machine-output and error
envelopes, workflow schema, agent-facing context/extraction contracts, package manifests and
exports, runtime engine constraints, and publishable tarball inventories. It intentionally does not
claim to freeze every command-specific JSON payload. The published-to-main delta is explicit: the
pre-0.7 main changed the agent-facing context contract while the CLI topology remained compatible.
A deliberate change requires an explicit baseline update and review; accidental drift fails the
offline check.

## Known limitations

- Current runner artifacts do not expose complete input/output/reasoning token usage by role.
- Agent-reported changed paths and test claims are not equivalent to authoritative Git, filesystem,
  or check-run evidence.
- Legacy context lookup does not provide an authoritative retrieval-recall measure.
- Latency varies by machine, filesystem cache, process scheduling, and adapter availability.

These gaps are baseline facts. Later 0.7 tasks may close them, but must not rewrite the historical
control as if the signals had always existed.
