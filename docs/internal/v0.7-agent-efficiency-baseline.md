# AgentPlane 0.7 compatibility and agent-efficiency baseline

This baseline is the pre-refactor control for the 0.7 program. It has two distinct provenance
anchors and must not collapse them into one claim:

- **Published compatibility reference:** Git tag `v0.6.24` at
  `30f62b82dff28909dcb3ccc2ace2bf3e356203bb`, plus the exact npm metadata and package inventories
  captured in `scripts/baselines/v0.6.24-compatibility-contract.json`.
- **Efficiency reference:** task-parent `main` at
  `1a702e160ba9f0efe7067f2a22fc008defc89ffb`. This commit contains post-`v0.6.24` changes, so
  `scripts/baselines/agent-efficiency-pre-v0.7-main.json` is a pre-0.7 main baseline, not a claim
  about the published package.
- **Additive replay reference:** the same exact main commit, executed in disposable repositories by
  a separately hashed replay harness. Replay envelopes and
  `scripts/baselines/agent-efficiency-pre-v0.7-replay.json` are created only after all authoritative
  provider and supervisor evidence is available; they never replace the historical baseline.

The compatibility check is intentionally network-free. Registry evidence is captured once with
explicit provenance and then reviewed as an immutable historical input. The check compares the
working tree against the committed exact-main contract and exposes the already-existing delta from
the published release.

## Golden scenarios

`scripts/bench/agent-efficiency-fixtures.json` defines exactly ten scenarios:

1. direct code task;
2. `branch_pr` task;
3. context assimilation;
4. stale `HEAD`;
5. approval required;
6. missing knowledge;
7. evaluator rework;
8. out-of-scope write;
9. adapter crash;
10. Hermes one-step supervision.

Each scenario carries expected quality, safety, lifecycle, episode, and evidence controls. These
controls define the comparison case; they are not presented as observed telemetry. Cost comparisons
are valid only when the controls remain equivalent. A cheaper failed or unsafe run is not an
improvement.

## Measurements

The harness defines a provenance cell for every required measurement and keeps structural values
separate from timing diagnostics:

- input, output, and reasoning tokens by role;
- exact LLM episode ledger and the registry lifecycle control;
- prompt boilerplate bytes and duplicated serialized bytes;
- preparation latency and context-search latency;
- knowledge retrieval hits and gaps;
- success, rework, block, context-gap, and scope-violation outcomes;
- observed versus agent-claimed evidence.

`evidence_observed_to_claimed_ratio` is the observed share of machine-checkable plus agent-claimed
evidence: `observed / (observed + agent_claimed)`. Human-supplied controls are excluded. The three
evidence telemetry fields must become observable together; a newly observed share below the golden
evidence control is a quality regression, not a cost improvement.

Numerical values are accepted only when a committed sanitized evidence payload supplies them. Each
run has a separate canonical evidence bundle. The builder recomputes the digest of every payload
from its committed bytes, recomputes the whole bundle digest, and resolves every telemetry cell's
`artifact_id` plus dotted `source` path back to the exact payload value. A digest copied into an
envelope without matching committed bytes is rejected. Every provider-backed replay has an episode
ledger whose order exactly equals `expected_episode_trace`; every episode contains exactly one
final `turn.completed` event, one resolved schema-validated final status, and a bounded
full-fixture effect snapshot. Repeated schema-valid status events are collapsed only when their
normalized enum values agree; conflicting or malformed events fail closed. Registry
`expected_lifecycle_trace` remains a synthetic control, is
persisted with `fixture_control` provenance, and is never confused with the separately recorded
number of anchor-preparation CLI calls. A missing token, retrieval, outcome, or evidence signal is
represented as a typed gap or `not_applicable` cell; it is never estimated from bytes and never
coerced to zero. This preserves the difference between “measured zero” and “not observed by this
contract”.

Replay prompt-cost definitions are explicit. `duplicate_input_bytes` reuses
`repeated_utf8_string_leaves_v1` over each canonical prepared bundle. When the anchor does not emit
`prepared_metadata.prompt_boilerplate_bytes`, `prompt_boilerplate_bytes` uses
`bundle_base_prompt_content_utf8_v1`: the sum of UTF-8 bytes in
`bundle.base_prompts[].content`. The prepared bootstrap must carry the anchor's exact
`bundle.json` reference. Supervisor-added fixture instructions are not counted as anchor
boilerplate; their bytes and the complete prompt-input byte count remain separate evidence fields.

The anchor does not expose an authoritative deterministic context-search or retrieval-recall
receipt. The replay therefore marks `context_search_latency_ms`, `retrieval_hits`,
`retrieval_gaps`, and `retrieval_recall_proxy` as `not_applicable`. Fixture existence checks used to
produce deterministic quality controls remain separately labeled fixture diagnostics and are not
presented as AgentPlane search telemetry.

The primary optimization target is cost per verified result at equivalent quality and safety. Token
count alone is diagnostic. Timing is also diagnostic: repeated runs report distributions rather
than enforcing a hard threshold from one machine.

## Determinism and comparison rules

For the same fixture registry and tracked source artifacts:

- normalized structural output must be byte-identical across repeated runs;
- timestamps, absolute paths, host details, and `duration_ms` samples stay outside the structural
  digest; the historical fixture may provide only one diagnostic sample, and comparison requires a
  separately collected multi-run distribution;
- a known baseline value becoming unknown is a regression;
- an unknown baseline value becoming measured is an observability improvement;
- a structural cost increase above the committed 10% tolerance fails only for a quality-equivalent
  scenario;
- a safety or verified-success regression fails regardless of cost.

Expected protocol changes and quality/evidence improvements are reported as non-comparable rather
than being forced through the cost ratchet. Preparation, context-search, and wall-clock latency stay
diagnostic and do not receive a single-machine ceiling.

The tolerance is a ratchet guard for later 0.7 slices, not evidence that the baseline itself is
better or worse than `v0.6.24`.

## Commands

Capture or inspect the exact compatibility surface:

```bash
bun run bench:compatibility:capture
bun run bench:compatibility:check
```

Measure or check the agent-efficiency control:

```bash
bun run bench:agent-efficiency
bun run bench:agent-efficiency:check
```

Capture or rebuild the additive replay baseline:

```bash
bun run bench:agent-efficiency:replay:capture -- \
  --pilot \
  --driver scripts/bench/run-agent-efficiency-codex-replay.mjs
bun run bench:agent-efficiency:replay:capture -- \
  --anchor 1a702e160ba9f0efe7067f2a22fc008defc89ffb \
  --runs 5 \
  --driver scripts/bench/run-agent-efficiency-codex-replay.mjs
bun run bench:agent-efficiency:replay:check
```

The default command deterministically rebuilds from the canonical envelope and evidence
directories. An initial live capture additionally passes `--driver` with a reviewed repo-local
`.mjs` implementation under `scripts/bench/`; the CLI intentionally does not infer a driver from
local credentials. The exact driver bytes, contract version, and digest are part of both the anchor
and harness digest. The capture process passes only a fixed safe process environment and the exact
reviewed `AGENTPLANE_RF04_REPLAY_*` contract fields. Parent `PATH`, loader/preload variables, Git
configuration, proxy settings, temporary directories, credentials, and arbitrary opt-in fields are
never forwarded. The fixture registry is copied after commit/tree verification into a canonical
untracked `fixture_control_overlay_v1`; it is not misrepresented as content from the historical
anchor.

The reviewed driver pins the absolute ChatGPT-bundled Codex CLI `0.145.0-alpha.18`, visible model
`gpt-5.6-terra`, and reasoning effort `low`. The exact anchor AgentPlane CLI performs repository
initialization, task lifecycle preparation, and `task run --dry-run`; the hardened driver then
feeds the exact prepared bootstrap once to Codex. This is an
`anchor CLI preparation / Codex CLI execution` benchmark, not a claim that the stock 0.6 runner
executed every scenario or hosted `branch_pr` transition. Zero-agent outcomes are deterministic
fixture-supervisor observations. Registry lifecycle traces are fixture controls.

The pilot executes only `direct/run-01` and persists nothing. A complete capture runs ten scenarios
five times: 50 isolated run envelopes and 55 provider turns from the declared episode ledgers. Each
exact detached anchor checkout uses a local shared object store, disables inherited Git config and
hooks, removes its remote, and verifies both commit and tree before setup. The nested work fixture is
initialized and committed before snapshot-sensitive work. Harness compilation and fixture
prerequisites are measured separately as `harness_setup_latency_ms`; they are excluded from
preparation and time-to-result comparisons. `preparation_latency_ms` accumulates only the exact
AgentPlane subprocess durations for task lifecycle and `task run --dry-run`, excluding Git commits,
fixture preparation, and provider execution. The harness digest covers the transitive local module
closure and links two dependency claims. The portable claim hashes the Bun lock graph and relevant
workspace manifests, so CI can recompute it across operating systems. The capture-platform claim
walks each package's resolved `dependencies`, `optionalDependencies`, and available peers from its
real installation directory, including Bun sibling packages and platform binaries. It records the
capture OS, architecture, libc class, and Node ABI, then links those fields and both digests through
a capture receipt. Same-platform checks recompute both layers; foreign-platform checks recompute
only the portable layer and validate the captured receipt without substituting local binary bytes.
The complete same-host closure is rechecked after capture and inside the publication transaction.
The anchor build also rechecks HEAD, tree, and tracked-clean status after compilation.
`time_to_first_scoped_mutation_ms` comes from a live filesystem watch on `work/allowed.txt`; if that
event is not observed, the diagnostic is `not_applicable` rather than a post-turn estimate. The
parent timeout is scenario-aware and remains above the sum of all per-turn budgets while retaining
a hard 25-minute cap.

Each driver invocation must emit one canonical sanitized envelope plus one canonical
content-addressed evidence bundle. Provider `reasoning_output_tokens` maps directly to baseline
`reasoning_tokens`; no estimate is accepted. Codex receives a fixed minimal environment with no
`HOME`, `CODEX_HOME`, or `TMPDIR`, no inherited shell environment, and network-disabled workspace
sandboxing. Raw JSONL, prompts, final text, hidden reasoning, and stderr are streamed under byte
caps and never persisted; only the validated `blocked | done | reviewed` status enum survives.
Read-only episodes reject every write. Scoped-write episodes allow only `work/allowed.txt`; ignored
files, symlinks, and same-content rewrites are still detected before any exact-path commit. The
capture rejects non-canonical or unreferenced evidence, recursive
POSIX or Windows host paths, credential-like strings and sensitive keys, mixed model/runtime
profiles, missing run indices, and cells that are not linked to an exact recomputed evidence value.

Each completed replay retains all 50 actual run outcomes, including failure, rework, blocked, and
safety-violation values; it summarizes true/false counts and reports every mismatch against the
golden control instead of discarding non-matching runs. It also resolves 70 outcome cells,
27 provider token cells, and 170 scalar cells. Scalars are either observed or carry a typed
`not_applicable` rule; unknown is never encoded as zero. Structural cost and safety fields receive a
deterministic digest. Timestamps, host
fingerprints, token distributions, and latency distributions remain diagnostic, so changes to
timestamp, host, or latency samples do not change the structural digest. The offline check rebuilds
the baseline twice and rejects anchor, usage, outcome, provenance, run-count, or artifact-hash
tampering.

Envelope, evidence, and baseline targets are fixed canonical repository paths with component-wise
`lstat`/`realpath` checks and no symlink traversal or path overlap. A complete capture stages and
validates all three artifacts, then publishes them as one rollback transaction with the baseline
installed last. `--replace` covers the complete three-artifact generation. Controlled failures
restore every prior artifact; an uncatchable process kill leaves a transaction marker and requires
explicit recovery instead of silently accepting a mixed generation.

Run the measurement twice before accepting a baseline change. Review both the normalized structural
digest and the per-scenario provenance. Do not update either frozen baseline automatically in CI.

## Compatibility surface

The frozen compatibility contract covers all 243 command entries in the structural
`help --json --all` projection, positional arguments, options, global machine-output and error
envelopes, workflow schema, agent-facing context/extraction contracts, package manifests and
order-sensitive conditional exports, runtime engine constraints, the shared semantic release
tarball policy, policy-eligible source inventory, and the authoritative published npm tarball
inventories.
It intentionally does not claim to freeze every command-specific JSON payload. The
published-to-main delta is explicit: the
pre-0.7 main changed the agent-facing context contract while the CLI topology remained compatible.
A deliberate 0.7 change updates the reviewed cumulative candidate ledger in
`scripts/baselines/v0.7-compatibility-candidate.json`, including exact source-task provenance and
additive/removal/mutation evidence. The frozen 0.6.24 baseline remains byte-identical; unreviewed
candidate drift fails the offline check.

## Known limitations

This is a provenance-safe observability and structural-cost baseline, not yet a complete
quantitative cognitive/token benchmark. At the frozen main anchor it contains 10 known scalar cells
out of 170, no known observed-outcome cells out of 70, no known token cells out of 27, one diagnostic
duration sample, and two scenarios with source artifacts. The other cells remain explicit gaps.
AgentPlane must add anchored replay/instrumentation and populate the mandatory telemetry before an
`alpha.1` gate may claim improved total cost per verified result.

- The frozen anchor runner artifacts do not expose complete role token usage. The reviewed live
  driver closes this gap only after its pilot and full 55-turn capture succeed. Until the resulting
  50 envelopes, evidence bundles, and replay baseline are committed, CI and prepublish replay
  checks remain intentionally fail-closed and the branch is not mergeable.
- Agent-reported changed paths and test claims are not equivalent to authoritative Git, filesystem,
  or check-run evidence.
- Legacy context lookup does not provide an authoritative retrieval-recall measure.
- Latency varies by machine, filesystem cache, process scheduling, and adapter availability.

These gaps are baseline facts. The replay schema, isolated capture driver contract, deterministic
builder, and offline tamper checks close the infrastructure gap but do not manufacture the missing
measurements. A later explicitly authorized capture may close the evidence gap; it must not rewrite
the historical control as if the signals had always existed.
