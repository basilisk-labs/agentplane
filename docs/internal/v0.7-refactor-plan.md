# AgentPlane 0.7 refactor execution plan

Status: approved program; executable task graph created by `202607221838-SD1W93`.

Baseline: `main@026a4db26e7e541f36ef6652274ff3cefa1feccb` (AgentPlane 0.6.24).

## Outcome

AgentPlane 0.7 makes the CLI mechanically authoritative and semantically blind:

- the CLI owns project/task/Git/provider truth, policy, authority, lifecycle transitions, retrieval primitives, schemas, transactions, observation, evidence, and reproducibility;
- role agents own interpretation, planning, architecture, implementation, semantic evaluation, knowledge synthesis, and ambiguity resolution;
- every agent episode receives a versioned, bounded, reproducible work order;
- every observed process/check/Git/artifact fact is recorded by the supervisor, never trusted from an agent claim;
- every side effect is a typed operation bound to a current state fingerprint and authority record.

The existing AgentPlane context subsystem remains the durable knowledge plane. Work orders contain digest-addressed refs plus prepared bounded excerpts and omission receipts; they do not create a second context pack or wiki.

## Release strategy

The stable `0.7.0` release is the terminal gate. Intermediate builds are optional stabilization points, not permission to ship partial semantics as stable 0.7:

1. `0.7.0-alpha.1`: trust boundary, compatibility baseline, workflow contract repair.
2. `0.7.0-alpha.2`: typed work contracts, fingerprinted workflow steps, authority, real evaluator.
3. `0.7.0-beta.1`: common supervisor, resumable context lifecycle, direct execution.
4. `0.7.0-beta.2`: FTS/incremental retrieval, canonical candidates, bounded knowledge requests.
5. `0.7.0-rc.1`: phase-scoped tools and supervised `branch_pr`.
6. `0.7.0-rc.2`: granular command capabilities, typed render boundary, measured caches, zero architecture violations, migration matrix and docs.
7. `0.7.0`: exact-SHA release only; no new architecture scope.

A compatibility-safe correctness fix may be backported as `0.6.25+` only when it is self-contained, passes the full patch-release gates, and does not introduce a new default contract or persisted format. A prerelease is published only when external integration testing benefits from it.

## Wave gates

A milestone may advance only when all of the following are true:

- every included task is merged through `branch_pr`, has recorded verification, evaluator evidence, pre-merge closure, hosted close, and clean worktree cleanup;
- task-state, policy routing, lifecycle invariants, schemas, types, focused tests, and milestone-specific benchmarks pass;
- the frozen success/rework/safety controls do not regress while orchestration cost is measured;
- compatibility changes have a versioned schema, fixtures, migrator, idempotency/rollback evidence, and installed-package smoke;
- no task widens agent lifecycle authority or treats agent-reported mechanics as observed truth.

Wave approvals are sequential. Only ready leaves are approved and started; later plans remain pending so discoveries can refine acceptance criteria without silently changing active work.

## Integration policy

- Use one task branch/worktree and PR per real verification boundary.
- A related-task batch is permitted only when all included task IDs are listed, independently planned/verified, and a single PR reduces coordination risk.
- Context assimilation is the first production supervisor slice because its semantic SGR and transactional apply boundary already exist.
- Direct supervision must stabilize before `branch_pr`; `branch_pr` must preserve provider waits, protected-main checks, user-attributed merge authority, hosted close, and cleanup.
- Caches are forbidden until fingerprint and preparation instrumentation prove a deterministic, worthwhile node.
- Full evaluation-platform/benchmark product scope remains outside 0.7; the task-level EVALUATOR episode is a correctness primitive.

## Measurement contract

Golden scenarios are held constant across 0.6.24 and 0.7:

- direct code task;
- `branch_pr` task;
- context assimilation;
- stale HEAD;
- approval required;
- missing knowledge;
- evaluator rework;
- out-of-scope write;
- adapter crash;
- Hermes one-step supervision.

Record input/output/reasoning tokens per role, episode count, agent lifecycle calls, prompt boilerplate and duplicate bytes, preparation and retrieval latency, time to first scoped mutation and verified result, first-pass verified success, rework/block/context-gap rate, scope violations, stale rejects, retrieval hits/gaps, and observed-versus-claimed evidence.

An optimization is accepted only when total orchestration cost improves without lower verified success or weaker safety.

## Re-approval triggers

Request user approval again if any of the following occurs:

- scope expands beyond the reviewed RF-00 through RF-27 program and the verified contract-drift prerequisites;
- security, sandbox, network, merge, publish, or authority policy is weakened;
- a mandatory verification or migration gate is removed or its pass criterion changes;
- more than five additional implementation files are required by an active leaf beyond its approved plan;
- a stable intermediate release would expose an incomplete or incompatible contract not covered by the release strategy.

Defects discovered inside an approved contract are handled by a bounded follow-up task and do not change the program outcome.
