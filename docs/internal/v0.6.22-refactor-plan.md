# v0.6.22 Refactor Plan

## Goal

Ship the deferred maintainability and context-extraction work in patch release `v0.6.22` without changing public CLI, task lifecycle, context artifact, schema, or backend contracts.

The first patch tranche landed through PR `#4552`. It restored the clone gate, aligned the architecture check with the supported Node range, made route repairs exhaustive, preserved route probe diagnostics, and made ambiguous wiki targets explicit. The context-extraction tranche then added transactional artifact writes, graph-integrity checks, compact typed extraction packs, byte-aware batching, full-source fingerprints, and rebase-aware release-task reconciliation.

## Verified baseline

- 16 production modules exceed the 400-line hotspot warning; none exceeds the 600-line error threshold.
- 11 test files exceed 1000 lines, totaling 12,438 lines; none exceeds the 1300-line error threshold.
- The Knip baseline accepts 574 findings across 158 files.
- Context extraction now writes derived artifacts transactionally and validates graph integrity; its remaining release risk is lifecycle closeout around protected-main rebases, not extraction output consistency.

## Release gates

Before publication:

1. `bun run release:parity`, `bun run ci:contract`, and `bun run test:fast` pass on the merged `main` commit.
2. `ap release plan --patch` resolves the next version to exactly `v0.6.22`.
3. The hotspot and oversized-test baselines decrease; the Knip baseline drops below 574 findings and 158 files.
4. No task widens public behavior while splitting modules or tests.
5. Context extraction preserves granular SGR v2 entities, resolvable links, byte-budget metadata, stable fingerprints, and transactional derived artifacts under focused and full regression tests.
6. Included batch tasks, integration-queue entries, doctor diagnostics, and evaluator evidence remain correct when GitHub lands a protected-main PR by rebase.
7. The GitHub tag, release notes, published package versions, and merged `main` commit are verified after publication.

## Stop conditions

- A refactor changes a public contract or requires a migration.
- The patch requires a new dependency or architecture boundary not covered by the approved task.
- A baseline can only be reduced by hiding findings, deleting assertions, or weakening thresholds.
- Release verification cannot tie published artifacts to the exact merged commit.

Any stop condition requires a new task or explicit scope approval instead of being absorbed into `v0.6.22` silently.
