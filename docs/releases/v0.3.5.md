# Release Notes — v0.3.5

## Summary

- Finalized the README v3 task-document contract so active repositories, templates, config defaults, and migration tooling now point at the same `Summary / Scope / Plan / Verify Steps / Verification / Rollback Plan / Findings` shape.
- Hardened the public docs shell and narrowed patch-critical test contours so the documentation surface is less theme-fragile and the local validation loop is more structured.
- Completed the install-first backend work by making Redmine projection-first, teaching `doctor` about backend-aware migration state, and adding a real env-gated live Redmine validation suite.

## Added

- Added formal backend capabilities and explicit projection-versus-snapshot contracts so task backends can declare whether they read from projection by default, write local task READMEs, refresh from a canonical source, or export projection snapshots.
- Added a canonical projection-snapshot export path that lets backends write `.agentplane/tasks.json` from the current local projection without conflating that snapshot with the canonical task source.
- Added backend-aware migration behavior so task-doc migration now refreshes the exported task snapshot through backend capabilities instead of relying on a `local`-only special case.
- Added projection-aware `doctor` checks that prefer live backend task projection when deciding whether task READMEs are still on the legacy v2 contract.
- Added projection-first Redmine reads so ordinary task commands now use the local cached projection instead of silently refreshing from the network.
- Added install-first backend docs and regression coverage for projection-based export behavior, so external backends are described and tested as first-class runtime surfaces.
- Added a warning for `DONE` task README archives that exist on disk but are missing from the git index, so task-archive drift no longer hides behind a clean `.agentplane/tasks.json`.
- Added capability-driven close-commit handling so projection-backed backends that write local task READMEs receive the same deterministic task-archive treatment as the local backend.
- Added a repo-local `skills/` directory and vendored the `humanizer` skill into the repository as the first managed development skill.
- Added a new public blog post for `0.3.4` and backfilled the `0.3.3` release post so the release line remains visible on the public site.
- Added a canonical env-gated live Redmine contract suite that refreshes the local projection from a real Redmine instance, reads tasks from that projection, and exports a projection snapshot without performing remote writes.
- Added canonical backend-matrix scripts so the hermetic backend regressions and the live Redmine suite can be run together through one named entrypoint.

## Improved

- Improved repository config, JSON schema defaults, migration output, and task-facing docs so README v3 is now the canonical active contract instead of one option among multiple live shapes.
- Improved docs-shell spacing so documentation pages have larger horizontal gutters and less cramped reading space on wide layouts.
- Improved docs-shell chrome by removing decorative icons from the documentation frame, hiding the right-hand subsection map, and fixing the sticky offset so the left navigation does not slip under the navbar while scrolling.
- Improved the docs-shell implementation to rely on repo-owned theme hooks and stable wrappers instead of Docusaurus hash-based CSS-module selector patterns.
- Improved the design checker so minimal chrome radii are enforced through explicit design tokens rather than brittle literal-value matching.
- Improved patch-critical integration coverage by splitting the critical and platform-critical contours into canonical named sub-suites instead of one large implicit bucket.
- Improved the fast local CI selector with a backend-focused bucket so backend-only changes stop falling back to the broad full-fast contour.
- Improved workflow and developer docs so the new fast buckets, critical sub-suites, and backend validation surfaces are explained in one canonical place.
- Improved remote-backend setup and troubleshooting docs so install-first semantics are described as `canonical source -> local projection -> explicit sync -> export snapshot`.
- Improved `task migrate-doc` by auto-exporting the task snapshot after migration for the local backend, eliminating the extra manual `task export` step before a clean `doctor`.

## Fixed

- Fixed the last mixed-contract gaps between repo config, schema examples, migration order, templates, tests, and docs for README v3 task documents.
- Fixed docs-shell fragility that depended on theme-generated CSS-module class names, which made the layout vulnerable to upstream naming drift.
- Fixed the false distinction between `local` and projection-backed backends when deciding whether `finish` should create a deterministic close commit for task README artifacts.
- Fixed silent task-archive drift where task README archives could exist on disk but fail to land in git without any explicit diagnostic.
- Fixed Redmine task reads so installed users no longer trigger hidden network refreshes during ordinary `task list`, `task show`, and `doctor` flows.
- Fixed backend migration diagnostics so `doctor` now evaluates migration state from the right backend-aware projection path instead of trusting a stale export snapshot.
- Fixed external-backend validation coverage by adding a real read-only Redmine contract run instead of relying only on hermetic fixtures.

## Upgrade Notes

- `0.3.5` does not introduce a breaking workflow or configuration migration for installed repositories.
- Repositories already on README v3 continue to work without manual task-doc migration.
- Repositories that still contain legacy README v2 task docs can continue to use `agentplane task migrate-doc --all`; migration now keeps the projection snapshot synchronized through the active backend path.
- Redmine remains projection-first: ordinary reads stay local, while `agentplane backend sync redmine --direction pull` remains the explicit network refresh boundary.
- The new live Redmine backend suite is env-gated and intended for repositories that already have Redmine credentials configured; it is not a mandatory part of ordinary local pre-push loops.

- Confirmed that the live Redmine suite passes against the configured test backend without performing remote writes.
- Publish, tag, npm, and GitHub release checks still run after the release commit and tag are pushed by the release task.
