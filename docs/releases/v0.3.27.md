# Release Notes - v0.3.27

## Summary

- Hardens the v0.3 release hygiene path with explicit gates for repository artifacts, task-state drift, package tarball contents, sanitized build manifests, and local install smoke coverage.
- Makes clean-repository initialization repo-neutral by changing the default workflow scope from `packages/**` to `**`, with service-path exclusions handled by policy/runtime guards.
- Updates current documentation for the 0.3.26 codebase, removes stale legacy guidance from active docs, and records the 0.6 roadmap milestone for the eval system.
- Fixes init recipe materialization tracking so cached recipe files are included in the initial install commit.

## Added

- Added `scripts/check-agentplane-artifacts.mjs` to block new volatile `.agentplane` artifacts such as tracked task snapshots, raw logs, JSONL traces, run directories, and repro directories.
- Added task-state validation to the release gate so stale release TODO/DOING work and invalid task metadata are caught before publication.
- Added package tarball assertions for `agentplane`, `@agentplaneorg/core`, and `@agentplaneorg/recipes`, including allowlist and denylist checks against real `npm pack --json` output.
- Added sanitized build-manifest checks that distinguish framework checkout manifests from installed package manifests.
- Added local tarball install smoke coverage for the published CLI path.
- Added git archive artifact checks backed by `.gitattributes export-ignore` rules for volatile `.agentplane` paths.
- Added release gate coverage for the repo-neutral workflow default and docs cleanup checks.
- Added the 0.6 roadmap milestone for eval definitions, scenario-based runs, scoring/report artifacts, and regression gates for agents, recipes, and runner behavior.

## Improved

- Improved clean init behavior by making `.agentplane/WORKFLOW.md` default to `in_scope_paths: ["**"]` instead of assuming a monorepo `packages/**` layout.
- Improved workflow runtime fallback and fix logic so generated workflow guidance is suitable for arbitrary user repositories.
- Improved package metadata safety by keeping the published `.build-manifest.json` sanitized while preserving richer framework checkout manifest behavior for stale-dist guard checks.
- Improved release documentation alignment by rewriting the active roadmap around the actual v0.3 git-native task control plane and later modular prompt, runner, and eval milestones.
- Improved active docs and templates by removing obsolete legacy names and stale init/UI wording while leaving historical release notes intact.
- Improved init install commit fidelity by returning materialized recipe paths from cached recipe setup and adding them to the init commit path set.

## Fixed

- Fixed the clean-install workflow scope regression where newly initialized repositories could receive `packages/**` even when they were not package workspaces.
- Fixed release artifact drift by preventing new tracked `.agentplane/tasks.json`, `runs/`, `repro/`, `.log`, and `.jsonl` artifacts from entering source or archive outputs.
- Fixed package proof gaps where `npm pack --dry-run` visibility alone did not assert tarball contents or deny volatile/dev files.
- Fixed build-manifest metadata leakage risk by sanitizing the manifest shape published in npm packages.
- Fixed cached recipe init commits so materialized recipe files are represented in the initial install commit.
- Fixed documentation drift around legacy command names, root workflow-file fallbacks, and old init UI terminology in current user/developer docs.

## Upgrade Notes

- No manual migration is required for existing `0.3.26` projects.
- Newly initialized repositories now get a repo-neutral workflow scope. Existing `.agentplane/WORKFLOW.md` files are not rewritten unless the user runs the relevant workflow repair path.
- Release maintainers should run the full `release:ci-check` before publishing because it now includes artifact, package, manifest, local install, workflow-default, and docs-cleanup gates.
- Historical release notes remain historical records and may still mention older behavior; current docs and templates should describe the active CLI surface.
