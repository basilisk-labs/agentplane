# Release Notes — v0.3.3

## Summary

- Hardened framework-development runtime behavior so contributors can see which CLI/runtime is actually active, avoid stale-dist false positives, and default to the repo-local binary inside the framework checkout.
- Completed the README v3 rollout for task docs, including compatibility with legacy task READMEs, explicit migration tooling, clearer verification structure, and safer task-doc write paths.
- Tightened release and recovery discipline with immutable preflight checks, release recovery tooling, generated-doc sync during release mutation, and repository-owned CLI version expectations.
- Reduced contributor friction with path-aware local CI buckets, faster `doctor` behavior on large task archives, and shared workflow test scripts that keep local and CI runners aligned.

## Added

- Added `agentplane runtime explain` plus shared runtime-source resolution, so contributors can inspect the active binary path, framework source, and repo-version expectations directly.
- Added repository-owned CLI version diagnostics via `framework.cli.expected_version`, together with `doctor` and runtime reporting that point to the shortest recovery path when the active CLI is too old.
- Added README v3 as the default task-doc contract for new tasks, with `Findings` as the task-local observation section and acceptance-oriented `Verify Steps`.
- Added dual-read support for legacy README v2 tasks and a real `agentplane task migrate-doc` path for upgrading older task archives to README v3.
- Added explicit recovery guidance for legacy project upgrades where framework-managed files and task-doc formats are partially out of date.
- Added release recovery tooling for partial publish states, including read-only recovery checks before maintainers attempt a follow-up release.
- Added targeted fast-CI buckets for docs-only, task, doctor, CLI help, release, upgrade, guard, CLI core, and CLI runtime change scopes.
- Added shared workflow test scripts for platform-critical and coverage-sensitive suites so local development and GitHub Actions use the same runner contracts.
- Added a developer-facing CLI bug ledger for the `v0.3.x` stabilization cycle and promoted only the strongest incidents into the incident log.

## Improved

- Improved stale-dist enforcement by comparing watched runtime snapshots against the build manifest instead of using git dirtiness as a proxy for runtime freshness.
- Improved inspection ergonomics so read-only commands such as `doctor`, `runtime explain`, and related help surfaces warn and continue on dirty runtime trees instead of failing like mutating commands.
- Improved framework-checkout behavior so `agentplane` resolved from `PATH` hands off to the repo-local runtime by default, including from nested directories.
- Improved upgrade, release, finish, and doctor diagnostics so they describe the detected state, likely cause, and concrete recovery path more clearly.
- Improved direct workflow onboarding by compressing the happy path into shorter canonical docs, aligning startup surfaces, and expanding onboarding parity checks.
- Improved task lifecycle ergonomics by making plan/verify approvals profile-driven defaults instead of asking separate interactive questions in the normal happy path.
- Improved `doctor` performance by extracting modular check runners, bounding default archive scans, and batching historical commit lookups.
- Improved local contributor feedback loops by making `pre-push` path-aware and by splitting `ci:local` into fast and full contours.
- Improved public docs and blog surfaces for the `0.3.x` line, including humanized release posts and a release-focused blog landing order.

## Fixed

- Fixed the release regression where `release apply --push` could recurse into local `pre-push` hooks and appear to stall after creating the release commit and tag.
- Fixed release mutation drift so version-sensitive generated docs are regenerated and staged as part of the release commit itself.
- Fixed release/config drift so repository-owned `framework.cli.expected_version` is kept in sync with the released version during `release apply`.
- Fixed stale-dist false positives that blocked diagnostics or rebuilt framework checkouts even when the runtime build matched the current source snapshot.
- Fixed framework-development invocation drift where contributors inside the framework repository could unintentionally execute an older global binary from `PATH`.
- Fixed historical `doctor` noise by compacting archive-only findings and separating current actionable failures from legacy metadata anomalies.
- Fixed literal `\\n` rendering bugs in task README sections and normalized multiline task-doc serialization.
- Fixed task-doc write correctness by serializing concurrent README updates and by making `task doc set` report whether it actually updated the requested section, applied a full-doc payload, or made no effective change.
- Fixed README v3 persistence so migrated tasks keep `doc_version: 3` through updates and export instead of snapping back to the legacy version.
- Fixed the legacy README upgrade scenario in CI by raising the timeout budget for the real integration path and by unifying the platform-critical test runner across local and CI execution.

## Upgrade Notes

- No breaking API or workflow-mode changes are introduced in `0.3.3`.
- Repositories with older task docs can continue operating without an immediate rewrite, but if `agentplane doctor` reports legacy or mixed README state, run `agentplane task migrate-doc --all`.
- Repositories upgraded from older framework versions should follow the two-step recovery path when indicated by diagnostics: `agentplane upgrade --yes` (or `--remote --yes`) and then `agentplane task migrate-doc --all` if legacy task docs remain.
- Framework contributors using a global `agentplane` install should reinstall or update the global CLI to pick up the repo-local handoff wrapper and the newer runtime diagnostics.
- Release maintainers no longer need to manually sync generated package reference docs or `framework.cli.expected_version` after `release apply`; both are now part of the release mutation path.

## Verification

- Release scope reviewed against `v0.3.2..HEAD` and grouped into framework runtime hardening, README v3 migration, release/recovery discipline, and contributor-loop performance work.
- Repo-local checks across the shipped scope covered `agentplane doctor`, docs-site checks, policy routing checks, targeted `vitest` suites, shared workflow test scripts, and the standard `pre-push` gate.
- README v3 migration was validated on both the current repository archive and a simulated legacy project recovery path.
- Platform-critical CI and prepublish suites now run through shared scripts, which keeps timeout and runner contracts aligned between local execution and GitHub Actions.
- npm and GitHub publication checks still run after the release commit and tag are pushed by the release task.
