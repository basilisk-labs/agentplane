# Release Notes — v0.3.7

## Summary

- Hardened backward compatibility for older initialized repositories by turning legacy task-doc recovery into a more explicit and partially one-step upgrade path, instead of leaving operators to infer when `task migrate-doc` still needs to run.
- Brought Redmine-backed repositories much closer to the canonical one-file task-state model by adding canonical-state plumbing, revision-aware round trips, readiness checks, migration tooling, and conflict/live coverage.
- Continued the task-state cleanup by making one-file canonical sections and revision semantics real operational state, reducing stale-snapshot lifecycle races, and fixing remaining derive/verify artifact gaps around README v3.
- Tightened the release line itself with GitHub-aware publish gates, release-ready and publish-result artifacts, exact-SHA recovery flows, and stronger local release validation so patch publication is harder to fake and easier to recover.

## Added

- Added Redmine canonical-state custom-field plumbing, revision/section round-trip behavior, lifecycle metadata preservation, readiness diagnostics, migration commands, and live sync/conflict coverage so Redmine-backed repositories can adopt the same canonical task-state model without relying on lossy projection-only assumptions.
- Added one-file canonical task schema fields (`revision`, canonical `sections`), generated README-body projection, intent-based mutation APIs, projection-drift detection, and canonical migrate-doc behavior so task state converges on one source of truth instead of split rendered-body semantics.
- Added stronger release-publish artifacts and controls: Core CI release-ready manifests, publish-result artifacts, exact-SHA workflow dispatch input, release recovery that reads those artifacts, release-identity keyed workflow concurrency, and a local release E2E harness.
- Added smoother README v3 operator flows around task creation and verification: `task derive` now accepts `--verify` and seeds implementation `Verify Steps`, while verification details keep multiline formatting instead of collapsing into literal escaped text.

## Improved

- Improved compatibility for older repositories by making `upgrade` explicitly hand off to task README migration when needed and by adding `upgrade --migrate-task-docs` as a supported bridge for legacy README v2 recovery.
- Improved mixed-state recovery coverage so incomplete managed policy trees and legacy task docs are exercised together in upgrade and release-smoke regressions instead of being treated as separate happy-path cases.
- Improved lifecycle correctness by closing stale-state races in plan approval, start, status transitions, and retry-capable task writes, so the README state used for enforcement matches the freshest task state more reliably.
- Improved the release pipeline so local prepublish checks and GitHub publish gates validate the same release-critical surface, including red Core CI blocking for the release SHA and better recovery diagnostics when a publish has to be resumed.

## Fixed

- Fixed derived implementation tasks landing in weaker scaffolding than `task new`; they now preserve verify commands and seed the same acceptance-oriented `Verify Steps` contract for verify-required primary tags.
- Fixed verification artifacts that previously rendered literal `\n` sequences in README output when `--details` carried multiline content.
- Fixed upgrade commit flows that could still trip protected-path enforcement while repairing mixed config/policy drift as part of a legitimate upgrade.
- Fixed multiple local task mutation paths that still depended on stale snapshots or older rendered-body assumptions, reducing overwrite and drift classes that were especially risky in patch-level workflow tooling.

## Upgrade Notes

- `0.3.7` does not introduce a breaking workflow-mode or configuration migration for installed repositories.
- Repositories already on `0.3.6` and README v3 remain on the same user-facing task-doc contract; the main changes are in recovery behavior, backend state modeling, and release/runtime strictness.
- Older initialized repositories that still carry legacy README v2 docs or partial managed-tree drift now have a clearer supported path: run `agentplane doctor`, then `agentplane upgrade --migrate-task-docs --yes` when diagnostics call for it. If legacy task docs still remain, `upgrade` now emits an explicit handoff instead of silently leaving the repo in a mixed state.
- Redmine-backed repositories gain canonical-state migration and readiness tooling in this release. Ordinary reads remain projection-first and local; sync boundaries stay explicit.

## Verification

- Generated the `v0.3.7` release plan from `v0.3.6` with `agentplane release plan --patch` and used its `changes.md` inventory as the source record for this note.
- Passed targeted backward-compatibility suites with `bun x vitest run packages/agentplane/src/cli/run-cli.core.upgrade.test.ts packages/agentplane/src/cli/release-smoke.test.ts packages/agentplane/src/backends/task-backend.local.test.ts packages/agentplane/src/backends/task-backend.redmine.test.ts`, covering legacy README v2 recovery, combined managed-tree drift plus legacy docs recovery, and backend projection mappings.
