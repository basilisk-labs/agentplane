# Release Notes — v0.3.6

## Summary

- Hardened the patch-release line around task lifecycle clarity, commit diagnostics, and release recovery so `commit`, `finish`, `verify`, `doctor`, and `help` surfaces explain the real state more directly.
- Introduced semantic task-document patching and safer task README mutation behavior so parallel task writes stop silently clobbering one another and same-section collisions fail explicitly instead of overwriting newer content.
- Continued the cleanup phase by splitting large CLI and backend regression buckets, deriving fast-CI routing from actual suite topology, and removing duplicated harness code that kept broad test contours expensive to maintain.
- Refined the public documentation surface again with calmer desktop docs spacing, a published `0.3.5` release article, and a more stable docs shell that aligns better with the current release line.

## Added

- Added a semantic `TaskStore.patch()` layer with section-aware conflict detection and append-safe retry semantics so task README writes can be expressed as explicit document operations instead of whole-file rewrites.
- Added command-level adoption of semantic task patches across doc, plan, verify, comment, start, block, status, finish, and close flows so lifecycle commands mutate the freshest task state rather than stale in-memory payloads.
- Added explicit same-section and full-doc conflict regressions for the new patch layer so silent overwrite behavior is replaced with deterministic failure modes.
- Added namespace help roots for `agentplane help task` and `agentplane help task plan` so lifecycle discovery works through the obvious command path instead of falling through to `unknown command`.
- Added an explicit full-document mode to `task doc set`, allowing canonical README replacement without overloading section writes.
- Added phase-visible lifecycle progress output so long-running `start` and `finish` flows surface their internal stages instead of collapsing into one opaque status line.
- Added state-oriented release recovery summaries and narrower release-recovery fast-CI coverage so release failure diagnostics are more actionable before reruns.
- Added deterministic, topology-derived fast-CI suite discovery for split CLI and guard suites so future suite decomposition does not require synchronized manual registry edits.
- Added split suites for init, upgrade, backend sync, task lifecycle, guard, workflow, PR flow, task backend, recipes, branch meta, and task doc-write coverage so each regression family now maps to a clearer contract boundary.
- Added a published `0.3.5` release-journal entry and corresponding blog tags so the public release line remains chronological and complete.
- Added a framed OpenClaw-style init welcome dialog block after the ASCII art so remote OAuth onboarding now has a distinct visual callout.

## Improved

- Improved `commit` and `finish` diagnostics so the CLI distinguishes real failure phases and preserves meaningful formatter/linter signals from hook output.
- Improved `--allow-tasks` semantics so active-task artifacts are covered consistently in policy evaluation, auto-staging, help text, and regressions without opening blanket access to unrelated task directories.
- Improved `verify` output so successful verification reports a status-oriented summary instead of printing only a README path.
- Improved reconcile failures for malformed task READMEs so the CLI names the actual broken frontmatter condition and points operators at a concrete recovery path.
- Improved `doctor` by dropping satisfied CLI version expectations from informational findings, leaving only genuinely notable runtime context in the clean path.
- Improved stale-build detection by treating test-only churn as non-runtime drift while preserving guards for actual runtime changes.
- Improved multiline `task doc set --text` handling so escaped newlines are normalized consistently across doc and plan mutation paths.
- Improved docs-shell spacing and top alignment on desktop so the main reading column has more breathing room from the divider and the sidebar starts higher under the navbar.
- Improved release-facing and developer-facing docs so the bug ledger, troubleshooting notes, generated CLI docs, and public release surfaces match the shipped CLI behavior more closely.
- Improved the integration-test harness surface by centralizing repeated `AGENTPLANE_HOME`, stdio, and task-backend stubs into shared helpers across previously duplicated suites.

## Fixed

- Fixed stale CLI release smoke coverage by replacing the ad-hoc recipe archive fixture with the shared recipe archive helper, so `recipes install` in the release gate now tracks the real manifest contract instead of a drifting minimal archive.
- Fixed misleading lifecycle error reporting where `commit` and `finish` could attribute failures to the wrong phase.
- Fixed empty-index `agentplane commit` ergonomics by allowing the top-level command to auto-stage allowlisted paths while keeping `guard commit` intentionally staged-only.
- Fixed compressed hook summaries that previously dropped the decisive `Prettier` or `ESLint` lines from the middle of a failed hook log.
- Fixed task-artifact allowlisting so `.agentplane/tasks/<active-task-id>/...` paths are accepted through `--allow-tasks` without requiring duplicate manual prefixes.
- Fixed stale expectations in workflow and lifecycle regressions after the task-doc outcome surface changed to explicit `section-updated`, `full-doc-updated`, and `no-change` reporting.
- Fixed broad mixed regression buckets that hid the actual contract under monolithic test files, reducing the blast radius of future patch work.
- Fixed task README write races where retries could recompute from stale payloads and undo a newer section update.
- Fixed doc/plan text handling where literal escaped newlines were preserved instead of turning into real multiline task content.
- Fixed over-eager stale-dist blocking for pure test and snapshot churn.
- Fixed the doctor clean path so a satisfied repository CLI version no longer produces a misleading always-on informational finding.
- Fixed task commit emoji enforcement so non-`DONE` task commits can use agent-chosen emojis instead of system-derived ones, while `DONE` still requires `✅`.
- Fixed the public release surface gap by adding the missing `0.3.5` blog entry to the `0.3.x` release block and aligning docs-column spacing with the current layout intent.

## Upgrade Notes

- `0.3.6` does not introduce a breaking workflow-mode, task-doc, or configuration migration for installed repositories.
- Repositories already using README v3 continue to work without manual migration; the semantic task patch layer changes how writes are applied internally, not the user-facing task-doc contract.
- Non-`DONE` task status commits may now use agent-chosen emojis when explicitly provided; `DONE` close commits still require the `✅` marker.
- Release and publish flow remains GitHub-first: local `release apply --push --yes` prepares and pushes the release commit and tag, while npm publication still happens through the publish workflow.

## Verification

- Generated the `v0.3.6` release plan from `v0.3.5` with `agentplane release plan --patch` and used its `changes.md` inventory as the source record for this note.
- Confirmed that the generated plan resolves the next patch target to `v0.3.6` and covers the entire post-`v0.3.5` change range now documented below.
- Passed `bun run release:prepublish` for the prepared `v0.3.6` state, including parity, builds, generated docs freshness, lint, release-critical tests, and npm dry-run packing for both publishable packages.
