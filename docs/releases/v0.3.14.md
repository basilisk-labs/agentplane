# Release Notes — v0.3.14

## Summary

- Completed the large post-`v0.3.13` refactor wave on protected `main`, including Zod contract consolidation, backend capability routing, command output normalization, and branch-pr workflow cleanup.
- Fixed the release workflow gap that left `npm install -g agentplane` broken by treating `@agentplaneorg/recipes` as a required published package in release detection, manifests, smoke checks, and publish execution.
- Tightened branch-pr ergonomics again so task branches publish themselves, close tails materialize without dirtying base, merged cleanup becomes idempotent, and delete-only cleanup pushes avoid the full slow pre-push path.
- Standardized more of the developer surface around git identity, commit message rules, release documentation, schema strategy, and test harness structure so the next release cycle starts from a narrower operational baseline.

## Added

- Added canonical Zod-backed config validation as the new contract source for the shipped configuration surface.
- Added canonical Zod-backed task artifact validation so task exports, PR metadata, handoff data, and task frontmatter now share one runtime contract style.
- Added a first-class `@agentplaneorg/testkit` facade for CLI harness usage instead of forcing tests to reach into package-private helpers.
- Added an explicit ADR for schema runtime strategy so the repository now records why the validation stack converges around generated schemas from code-first contracts.
- Added backend capability helpers in command-layer shared code, giving task flows a central place to ask what a backend can do.
- Added task-store service modules so task persistence logic is no longer trapped inside one shared monolith.
- Added a typed `CommandResult` flow for release and task commands, reducing direct output coupling in those paths.
- Added release-apply pipeline modules so release preparation now has explicit phases instead of one large orchestration block.
- Added branch-pr documentation for the optimized route, including one-pass `pr open`, worktree-local execution, and merged cleanup expectations.
- Added publish workflow coverage for `@agentplaneorg/recipes`, including detection, smoke checks, manifest outputs, and hosted release publication.
- Added an explicit patch-release preparation task for `v0.3.14`, freezing the version, tag, and hosted publication route in task metadata.

## Upgrade Notes

- No manual migration is required when upgrading from `v0.3.13` to `v0.3.14`.
- The main operational change is in release publication: this patch expects `@agentplaneorg/recipes` to be published together with `@agentplaneorg/core` and `agentplane`, and the release workflow now enforces that parity.
- Repositories using the branch-pr route should see fewer manual cleanup steps after merge, but the protected-`main` hosted publish route remains the canonical way to ship a release.

## Verification

- Release preparation for `v0.3.14` is validated from a fresh release plan generated off current `main` with the `v0.3.13..v0.3.14` diff frozen in task metadata.
- Candidate validation must cover release notes, version/parity checks, packaging/publish gates, and the package install path with `@agentplaneorg/recipes` present in the release payload.
- Final release publication remains a hosted action from merged `main`; the patch is only considered complete after npm-registry smoke confirms that the published `agentplane` install path works end-to-end.
