# Release Notes — v0.3.8

## Summary

- This patch is primarily a preparation release for the upcoming 0.4 and 0.5 lines: the codebase was simplified, split into smaller modules, and given more shared execution paths instead of shipping a single marquee feature.
- The CLI, task-mutation, workflow-transition, backend, and runner internals were converged around shared contracts so the next minor releases can build on cleaner seams with less duplicated logic.
- Release and verification coverage stayed strict throughout the refactor wave, with every landing step tracked through task records, integration commits, and release-plan inventory.
- The repository also drops the now-obsolete `REFACTOR.md`, reflecting that the previous refactor program was completed and folded back into normal release flow.

## Added

- Added shared mutation bridges, transition services, output helpers, and test builders that turn previously repeated orchestration code into reusable building blocks for the next release lines.
- Added smaller dedicated modules around CLI commands, task lifecycle handling, upgrade planning, backend adapters, and runner lifecycle branches so future feature work can land on narrower files with clearer ownership.
- Added release inventory coverage for the completed refactor wave, including traceable task closeouts and notes generation for the new `v0.3.8` patch target.

## Improved

- Improved maintainability across the repository by collapsing parallel implementations into shared paths, especially in task mutation, workflow transitions, CLI output, backend projection, and test fixtures.
- Improved patch-release readiness by keeping parity checks, release planning, and artifact validation aligned while the internal architecture was being reworked.
- Improved the forward path to 0.4 and 0.5 by reducing oversized files, trimming bespoke helpers, and making the code map easier to navigate for deeper product changes.

## Fixed

- Fixed multiple internal duplication patterns that previously forced similar behavior to be maintained in several places, which lowered confidence for larger follow-up releases.
- Fixed stale project hygiene around the completed refactor backlog by removing the obsolete planning document from the shipping tree.
- Fixed several test and artifact maintenance gaps by standardizing the way release-relevant changes are described, verified, and integrated.

## Upgrade Notes

- `0.3.8` is a patch release with no intended user-facing workflow break, but it deliberately reshapes internals to make the 0.4 and 0.5 trains easier to ship.
- Operators upgrading from `0.3.7` should expect the same top-level CLI contract while internal implementation paths are significantly cleaner and more unified.
- If you maintain custom tooling around repository internals rather than the public CLI surface, review the refactor-heavy inventory below before depending on file layout or helper locations.

## Verification

- Generated the patch release plan with `agentplane release plan --patch`, locking the release target to `v0.3.8` from the `v0.3.7` baseline.
- Confirmed package parity before release work with `bun run release:parity` and a direct package-version check for `packages/core`, `packages/agentplane`, and the pinned `@agentplaneorg/core` dependency.
- Checked release recovery state locally and against GitHub so the previous `v0.3.7` publication is clean before applying the next patch.
- Authenticated GitHub CLI access is available for the publish path, which is required by the repository release workflow.
