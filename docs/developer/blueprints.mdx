Blueprints are AgentPlane's typed route contracts for task-specific harness behavior.

v0.5 rc1 ships the model, built-in registry, validation, resolver, explain output, recipe hint
bridge, task verification visibility, and ACR summary extension. It does not ship blueprint node
execution or a runner workflow engine.

## Definition

A blueprint is a typed execution-route contract for a task. It describes the harness route that
should be followed, the evidence that must be produced, and the stop rules that prevent the wrong
route from being used.

A blueprint is not:

- a prompt;
- a recipe;
- a task status;
- a CI pipeline;
- a general workflow engine;
- a replacement for `direct` or `branch_pr`;
- a requirement that every task uses git, PRs, CI, or code checks.

The purpose is to let AgentPlane select a task-specific route. Analysis, content, documentation,
code, release, and operational tasks should not inherit the same code-oriented PR flow.

## Design Premises

Blueprints are useful only if they preserve the difference between task classes.

Facts:

- AgentPlane already has an outer task lifecycle, workflow modes, role contracts, policy modules,
  and verification records.
- Recipes already provide domain-specific guidance.
- Some tasks mutate repositories and need git topology, checks, PR artifacts, or release gates.
- Some tasks are read-only analysis or content generation and need evidence, assumptions, and
  final output, not CI.

Inferences:

- The missing layer is not another prompt. It is a structured route between the task lifecycle and
  recipe guidance.
- The route must be inspectable before work starts, otherwise it cannot explain why context,
  policies, tools, or checks were loaded.
- The first implementation should model and validate routes before it tries to execute them.

Hard constraint:

- Do not make one code PR pipeline the default route. `analysis.light` must stay lightweight:

```text
intake -> scope -> context_resolve -> work_unit -> evidence -> final
```

In the v0 node catalog, `evidence -> final` is the human shorthand for
`artifact_write -> verify_record -> finish`.

No CI, PR, worktree, hosted checks, or merge gates appear unless the task meaning requires them.

## Methodology References

Blueprints should reuse proven workflow and governance ideas without copying their full complexity.

| Reference                                                                                                           | Useful idea                                                                 | How AgentPlane should use it                                                   |
| ------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| [BPMN](https://www.omg.org/spec/BPMN/2.0/About-BPMN/)                                                               | Explicit processes, gateways, events, and handoffs                          | Treat blueprints as visible process contracts, not prose-only policy text      |
| [Workflow Patterns](https://www.workflowpatterns.com/patterns/control/)                                             | Named control-flow patterns such as sequence, split, join, and cancellation | Keep node graphs simple, explicit, and testable                                |
| [Amazon States Language](https://docs.aws.amazon.com/step-functions/latest/dg/concepts-amazon-states-language.html) | State-machine definitions as structured data                                | Represent routes as JSON-like data that can be validated before execution      |
| [Temporal](https://docs.temporal.io/)                                                                               | Deterministic workflow history and resumable execution                      | Keep future execution replayable, with durable evidence and explicit decisions |
| [Open Policy Agent](https://www.openpolicyagent.org/docs/latest/)                                                   | Policy-as-code and separated authorization decisions                        | Keep route selection and permission gates explainable and auditable            |
| [CUE](https://cuelang.org/)                                                                                         | Schema, validation, and data constraints in one model                       | Use schema-first validation and reject invalid route definitions early         |

The v0 implementation should be closer to a typed state-machine registry than to BPMN or Temporal.
Those systems justify the shape of the model, but AgentPlane should keep the first surface small:
model, built-in registry, validator, resolver, and explain output.

## Layer Model

| Layer          | Owns                                                | Example                                      |
| -------------- | --------------------------------------------------- | -------------------------------------------- |
| Task lifecycle | External task state and traceability                | `TODO -> DOING -> DONE`                      |
| Workflow mode  | Repository mutation topology                        | `direct`, `branch_pr`                        |
| Blueprint      | Route inside task execution                         | `analysis.light`, `code.branch_pr`           |
| Recipe         | Domain context, checks, templates, and output hints | `code-map`, release blog companion           |
| Runner         | Execution mechanics for a resolved route            | Codex, Claude, local runner                  |
| ACR            | Post-execution evidence                             | blueprint id, checks, artifacts, stop reason |

The task lifecycle remains the outer contract. A blueprint is the internal trajectory for a task
while it is being planned, executed, verified, and closed.

## Package Boundary

The current implementation adds types, built-in blueprints, validation, resolution, explanation, and
read-only CLI surfaces. It does not add command execution or a workflow executor.

Current module layout inside the AgentPlane CLI source package:

```text
blueprints/
  model.ts
  builtins.ts
  registry.ts
  validate.ts
  resolve.ts
  explain.ts
  recipe-hints.ts
  index.ts
  *.test.ts
```

Initial responsibilities:

| File              | Responsibility                                                                 |
| ----------------- | ------------------------------------------------------------------------------ |
| `model.ts`        | Pure types, string unions, and small helper predicates                         |
| `builtins.ts`     | Built-in blueprint definitions and node catalog                                |
| `registry.ts`     | Registry lookup, duplicate detection, and built-in export                      |
| `validate.ts`     | Structural validation and invariant checks                                     |
| `resolve.ts`      | Select a blueprint from task metadata, workflow mode, mutation, risk, recipes  |
| `explain.ts`      | Convert resolved blueprint data into stable human-readable explanation objects |
| `recipe-hints.ts` | Convert normalized recipe blueprint extensions into resolver hints             |
| `index.ts`        | Public internal exports for CLI and verification integrations                  |

Do not place route logic inside command handlers. Commands should call the resolver and formatter.

## Core Model

The v0 model should be schema-first and small enough to validate before any runner sees it.

```ts
export type BlueprintId =
  | "analysis.light"
  | "content.light"
  | "docs.change"
  | "code.direct"
  | "code.branch_pr"
  | "release.strict"
  | "ops.approval";

export type TaskKind = "analysis" | "content" | "docs" | "code" | "release" | "ops";

export type WorkflowMode = "direct" | "branch_pr";

export type BlueprintNodeKind =
  | "intake"
  | "scope"
  | "context_resolve"
  | "approval_gate"
  | "worktree_start"
  | "work_unit"
  | "deterministic_check"
  | "fast_local_checks"
  | "artifact_write"
  | "pr_artifact"
  | "hosted_checks"
  | "publish_or_integrate"
  | "verify_record"
  | "handoff"
  | "finish";

export type BlueprintNodeMode = "deterministic" | "agentic" | "approval" | "record";

export type EvidenceKind =
  | "sources"
  | "assumptions"
  | "context_manifest"
  | "changed_paths"
  | "check_result"
  | "artifact"
  | "approval"
  | "external_link"
  | "commit"
  | "final_output"
  | "weak_links"
  | "rollback";

export type Blueprint = {
  id: BlueprintId;
  version: 1;
  title: string;
  description: string;
  taskKinds: TaskKind[];
  workflowModes?: WorkflowMode[];
  nodes: BlueprintNode[];
  edges: BlueprintEdge[];
  requiredEvidence: EvidenceRequirement[];
  stopRules: StopRule[];
  recipeExtensionPoints?: RecipeExtensionPoint[];
};

export type BlueprintNode = {
  id: string;
  kind: BlueprintNodeKind;
  mode: BlueprintNodeMode;
  required: boolean;
  protected?: boolean;
  inputs?: string[];
  outputs?: string[];
  evidence?: EvidenceKind[];
  recipeExtensions?: RecipeExtensionKind[];
};

export type BlueprintEdge = {
  from: string;
  to: string;
  condition?: string;
};

export type EvidenceRequirement = {
  id: string;
  kind: EvidenceKind;
  required: boolean;
  producedBy: string;
  description: string;
};

export type StopRule = {
  id: string;
  reason: string;
  severity: "stop" | "approval_required" | "warn";
};

export type RecipeExtensionKind =
  | "context_hint"
  | "evidence_requirement"
  | "check_suggestion"
  | "output_schema"
  | "artifact_template"
  | "risk_hint"
  | "preferred_blueprint";

export type RecipeExtensionPoint = {
  nodeKind: BlueprintNodeKind;
  allowed: RecipeExtensionKind[];
  rejected: string[];
};
```

These names are design targets, not committed public API. The important invariant is that the route
is structured data that can be explained, validated, and later executed.

## Resolved Model

The resolver should not mutate tasks. It should return a decision object that command and
verification surfaces can display.

```ts
export type BlueprintResolveInput = {
  taskId?: string;
  title?: string;
  description?: string;
  tags: string[];
  owner?: string;
  workflowMode?: WorkflowMode;
  mutation: "none" | "docs" | "code" | "release" | "ops" | "unknown";
  touchedPaths?: string[];
  recipeHints?: RecipeHint[];
  riskFlags?: RiskFlag[];
  explicitBlueprintId?: BlueprintId;
};

export type RecipeHint = {
  recipeId: string;
  recipeVersion?: string;
  recipeName?: string;
  extensionId?: string;
  summary?: string;
  kind: RecipeExtensionKind;
  targetNodeKind?: BlueprintNodeKind;
  value: unknown;
};

export type RiskFlag =
  | "network"
  | "credentials"
  | "deploy"
  | "publish"
  | "merge"
  | "security"
  | "external_system";

export type ResolvedBlueprint = {
  blueprint: Blueprint;
  activeNodes: BlueprintNode[];
  skippedNodes: SkippedNode[];
  requiredEvidence: EvidenceRequirement[];
  selectionReasons: string[];
  acceptedRecipeExtensions: AcceptedRecipeExtension[];
  rejectedRecipeExtensions: RejectedRecipeExtension[];
  stopReasons: StopReason[];
};

export type SkippedNode = {
  node: BlueprintNode;
  reason: string;
};

export type AcceptedRecipeExtension = {
  recipeId: string;
  recipeVersion?: string;
  recipeName?: string;
  extensionId?: string;
  nodeKind: BlueprintNodeKind;
  kind: RecipeExtensionKind;
  summary?: string;
  value?: unknown;
  reason: string;
};

export type RejectedRecipeExtension = {
  recipeId: string;
  recipeVersion?: string;
  recipeName?: string;
  extensionId?: string;
  nodeKind?: BlueprintNodeKind;
  kind: RecipeExtensionKind;
  summary?: string;
  value?: unknown;
  reason: string;
};

export type StopReason = {
  id: string;
  severity: "stop" | "approval_required" | "warn";
  reason: string;
};
```

The resolver may return stop reasons. A stopped resolution is still useful because it explains why a
runner must ask for approval, narrow scope, or choose a different route.

## Node Catalog

Blueprint nodes are reusable building blocks. They are not task statuses. A task can remain in
`DOING` while many blueprint nodes run inside that state.

| Node kind              | Purpose                                                           | Mode          | Evidence                                  |
| ---------------------- | ----------------------------------------------------------------- | ------------- | ----------------------------------------- |
| `intake`               | Classify user intent, task kind, owner, risk, and expected output | deterministic | task kind and classification reasons      |
| `scope`                | Fix boundaries, assumptions, constraints, and stop rules          | deterministic | approved scope and unresolved ambiguity   |
| `context_resolve`      | Select relevant policy, docs, files, recipes, and tools           | deterministic | context manifest and selection reasons    |
| `approval_gate`        | Require explicit approval for risky actions                       | approval      | approval source and approved action       |
| `worktree_start`       | Create branch/worktree when `branch_pr` mutation requires it      | deterministic | branch and worktree path                  |
| `work_unit`            | Perform the main analysis, writing, coding, or operational work   | agentic       | produced artifact, diff, or answer        |
| `deterministic_check`  | Run a fixed command or validation step                            | deterministic | command, result, output summary           |
| `fast_local_checks`    | Run cheap local checks before expensive hosted checks             | deterministic | command bundle and result                 |
| `artifact_write`       | Write a task artifact, report, PR artifact, or generated output   | deterministic | artifact path and content summary         |
| `pr_artifact`          | Prepare PR title, body, labels, and review context                | deterministic | PR metadata or draft artifact             |
| `hosted_checks`        | Observe hosted checks when required                               | deterministic | provider, status, URL                     |
| `publish_or_integrate` | Push, publish, merge, release, or integrate when required         | deterministic | target, result, external link or commit   |
| `verify_record`        | Record verification evidence in the task artifact                 | record        | verification state and evidence reference |
| `handoff`              | Transfer unresolved work to a human or another owner              | record        | reason, next owner, remaining work        |
| `finish`               | Close the task through the active AgentPlane lifecycle            | deterministic | close result and final task state         |

## Built-in Blueprints v0

The first implementation should ship a small registry. The goal is route clarity, not a complete
workflow language.

| Blueprint        | Default use case                                       | Workflow modes        | Must not include by default               |
| ---------------- | ------------------------------------------------------ | --------------------- | ----------------------------------------- |
| `analysis.light` | Read-only analysis, research, review synthesis         | any                   | worktree, PR, CI, merge, publish          |
| `content.light`  | Drafts, launch text, release-note drafts, editorial UX | any                   | code checks, PR, CI, merge, publish       |
| `docs.change`    | Repository docs and policy-adjacent prose              | `direct`, `branch_pr` | hosted checks unless repo policy requires |
| `code.direct`    | Code mutation in direct mode                           | `direct`              | branch PR lifecycle                       |
| `code.branch_pr` | Code mutation in branch PR mode                        | `branch_pr`           | release publishing unless task is release |
| `release.strict` | Version, package, publish, distribution                | `direct`, `branch_pr` | silent publish, skipped approval          |
| `ops.approval`   | External systems, deploys, credentials, config changes | any                   | unapproved external mutation              |

### `analysis.light`

For read-only analysis, research, review synthesis, conceptual framing, and evidence-backed
answers.

```text
intake -> scope -> context_resolve -> work_unit -> artifact_write -> verify_record -> finish
```

Expected evidence:

- source list;
- assumptions;
- confidence or uncertainty notes;
- weak links;
- final answer or report.

No git, PR, CI, or code checks are required by default.

### `content.light`

For content drafts, copy, launch text, release-note drafts, and editorial artifacts that do not
change product code.

```text
intake -> scope -> context_resolve -> work_unit -> deterministic_check -> artifact_write
-> verify_record -> finish
```

Expected evidence:

- audience and purpose;
- source or product facts used;
- style or editorial check;
- final artifact.

### `docs.change`

For repository documentation changes.

```text
intake -> scope -> context_resolve -> work_unit -> deterministic_check -> artifact_write
-> verify_record -> finish
```

Expected evidence:

- changed docs paths;
- canonical links updated;
- docs IA check when navigation changes;
- policy routing and doctor when policy or gateway-adjacent text changes.

### `code.direct`

For code changes in repositories configured for direct workflow mode.

```text
intake -> scope -> approval_gate -> context_resolve -> work_unit -> deterministic_check
-> verify_record -> finish
```

Expected evidence:

- changed paths;
- focused tests or checks;
- task verification;
- close commit.

### `code.branch_pr`

For code changes in repositories configured for branch PR workflow mode.

```text
intake -> scope -> approval_gate -> context_resolve -> worktree_start -> work_unit
-> fast_local_checks -> pr_artifact -> hosted_checks -> publish_or_integrate
-> verify_record -> finish
```

Expected evidence:

- task branch and worktree;
- changed paths;
- fast local checks;
- focused tests;
- PR artifacts;
- hosted checks when required;
- integration or close commit.

### `release.strict`

For release, publish, version, package, and distribution work.

```text
intake -> scope -> approval_gate -> context_resolve -> work_unit
-> deterministic_check -> publish_or_integrate -> verify_record -> finish
```

Expected evidence:

- release plan;
- version and tag checks;
- package or distribution checks;
- hosted workflow evidence;
- post-publish verification.

### `ops.approval`

For operational changes that may touch external systems, credentials, deployment targets, or
configuration.

```text
intake -> scope -> approval_gate -> context_resolve -> work_unit
-> deterministic_check -> artifact_write -> verify_record -> handoff | finish
```

Expected evidence:

- approval source;
- target system or environment;
- commands or actions taken;
- rollback or recovery path.

## Selection Rules

An agent may propose a blueprint, but the AgentPlane resolver must produce the final resolved
blueprint.

Selection inputs:

- explicit user intent;
- structured task intent fields;
- task title and description;
- task tags;
- owner role;
- workflow mode;
- mutation scope;
- touched paths, when known;
- recipe hints;
- risk flags;
- required evidence;
- explicit blueprint id, when supplied.

Structured task intent is preferred over weak text inference. `title`, `description`, and `tags`
remain fallback signals so existing tasks still resolve, but new tasks should declare intent when
route choice affects mutation, risk, or evidence.

Task README frontmatter fields:

```yaml
task_kind: analysis # analysis | content | docs | code | release | ops
mutation_scope: none # none | docs | code | release | ops | unknown
risk_flags: [network] # optional: network, credentials, deploy, publish, merge, security, external_system
blueprint_request: analysis.light # optional request, still validated by the resolver
```

CLI creation fields:

```bash
agentplane task new \
  --title "Market analysis note" \
  --description "Analyze the current market context without repository mutation" \
  --owner CODER \
  --tag content \
  --task-kind analysis \
  --mutation-scope none \
  --risk network \
  --blueprint-request analysis.light
```

Resolver precedence:

1. risk flags that force safer routes;
2. explicit blueprint request, with compatibility stop reasons when unsafe;
3. structured `task_kind` and `mutation_scope`;
4. compatible recipe `preferred_blueprint`;
5. tags, title, description, and verification fallback.

Resolver rules:

1. Prefer the least powerful route that satisfies the task.
2. Do not resolve consumer analysis or content tasks to code PR routes unless code mutation is
   explicitly in scope.
3. If blueprint ambiguity changes mutation, network, publish, merge, deployment, or security risk,
   require approval or a narrower task.
4. Workflow mode constrains mutation topology. A blueprint cannot ignore `direct` or `branch_pr`.
5. Recipes can influence selection but cannot override protected route decisions.
6. Explicit blueprint ids are requests, not absolute commands. The resolver still validates
   compatibility.
7. Unknown mutation scope may resolve to `analysis.light` only for read-only tasks. For repository
   mutation it must stop or request scope clarification.

Example:

```text
User request: "Analyze this article and say what AgentPlane should adopt."
Resolved blueprint: analysis.light
Route: intake -> scope -> context_resolve -> work_unit -> artifact_write -> verify_record -> finish
Not required: PR, CI, code checks, release gates.
Evidence: sources, assumptions, conclusions, weak links.
```

## Commit Scope Synchronization

Commit messages are traceability evidence, not the initial blueprint selector. A task's blueprint is
resolved before work starts, while a commit is produced after work. The commit subject therefore
must not be overloaded with every routing field.

Keep the subject compact:

```text
<emoji> <TASK_SUFFIX> <scope>: <summary>
```

Use task-intent-aligned root scopes when the task declares structured intent:

| Task intent       | Preferred commit scope examples        |
| ----------------- | -------------------------------------- |
| `analysis`        | `analysis`                             |
| `content`         | `content`                              |
| `docs`            | `docs`, `docs/site`                    |
| `code`            | `code`, `code/resolver`, `code/schema` |
| `release`         | `release`                              |
| `ops`             | `ops`                                  |
| lifecycle commits | `task`, `close`, `integrate`           |

Optional commit trailers may mirror the resolved route for review tools:

```text
Blueprint: code.branch_pr
Task-Kind: code
Mutation-Scope: code
Risk-Flags: none
```

The commit-message hook reads structured task intent from the task README when available and rejects
task commit scopes that contradict it. This keeps commit policy synchronized with blueprint routing
without making commit subjects the source of truth for route selection.

## Validation Invariants

`validateBlueprint()` should reject invalid definitions before resolver tests run.

Structural invariants:

- `id` is unique in the registry.
- `version` is supported.
- every node id is unique within the blueprint.
- every edge references existing node ids.
- at least one `intake`, one `scope`, one `context_resolve`, one `work_unit`, one `verify_record`,
  and one `finish` node exists unless the blueprint is explicitly marked experimental.
- v0 graphs are acyclic.
- v0 graphs have a single entry node unless a future version adds explicit multi-entry semantics.
- required evidence points to existing producer nodes.
- protected nodes cannot be optional.

Semantic invariants:

- `analysis.light` and `content.light` cannot contain `worktree_start`, `pr_artifact`,
  `hosted_checks`, or `publish_or_integrate`.
- `code.branch_pr` requires `workflowModes: ["branch_pr"]`.
- `code.direct` requires `workflowModes: ["direct"]`.
- `release.strict` requires an approval gate.
- `ops.approval` requires an approval gate and rollback evidence.
- recipe extension points cannot target protected lifecycle replacement.
- every stop rule has stable id, reason, and severity.

Registry invariants:

- built-in blueprint ids are stable.
- built-in blueprint versions start at `1`.
- registry lookup by id is total for built-ins.
- duplicate built-ins fail at module load or registry construction time.

## Adding New Blueprints

The current built-in registry is code-backed. Add a new built-in blueprint with this sequence:

1. Add the id to `BlueprintId` in `packages/agentplane/src/blueprints/model.ts`.
2. Add the blueprint definition to `packages/agentplane/src/blueprints/builtins.ts`.
3. Reuse existing node kinds where possible; add a new node kind only when an existing node cannot
   express the route state.
4. Run validation through `validateBlueprint()` and add focused validator tests for any new semantic
   invariant.
5. Add resolver mapping in `packages/agentplane/src/blueprints/resolve.ts`.
6. Add resolver tests that cover structured task intent, workflow compatibility, risk precedence,
   and recipe preference behavior.
7. Update `docs/developer/blueprints.mdx` and generated CLI reference if command flags change.

Efficient authoring path for the next implementation step:

- introduce data-driven specs under `.agentplane/blueprints/*.yaml` or `.json`;
- add `agentplane blueprint scaffold <id>`;
- add `agentplane blueprint validate <path>`;
- load project-local blueprints after built-ins but before recipe hints;
- generate compatibility fixtures from the registry so every new blueprint gets route, evidence,
  and stop-rule coverage.

Recipes should not define full lifecycle graphs yet. They may request compatible blueprints and add
extension hints, but the resolver and registry remain the authority for lifecycle topology.

## Recipes Relationship

Recipes are domain extensions. They should make a route more specific, not replace AgentPlane's
core lifecycle.

Recipes may provide:

- context hints;
- task templates;
- output schemas;
- deterministic check suggestions;
- artifact templates;
- tool profile hints;
- evidence requirements;
- risk hints.
- preferred blueprint hints.

Recipes must not:

- disable approval gates;
- bypass workflow mode;
- replace `finish`;
- publish, merge, deploy, or access network without core approval;
- convert analysis or content work into a code PR pipeline without resolver selection;
- mutate protected task state directly;
- hide required evidence;
- add policy modules outside resolved route boundaries.

`preferred_blueprint` is a request, not an override. The resolver accepts it only when the requested
blueprint is compatible with the inferred task kind and workflow mode, and only when stronger risk or
explicit user selection has not already chosen a safer route.

The resolver should treat recipe data as input. It should emit a resolved blueprint with selection
reasons, accepted recipe extensions, rejected recipe extensions, and any approval requirements.

Allowed recipe extension targets in v0.5 rc1:

| Extension kind         | Default target          | Meaning                                      |
| ---------------------- | ----------------------- | -------------------------------------------- |
| `context_hint`         | `context_resolve`       | Additional domain context to consider        |
| `risk_hint`            | `context_resolve`       | Domain risk to surface during route choice   |
| `output_schema`        | `work_unit`             | Expected shape for the produced artifact     |
| `artifact_template`    | `work_unit`             | Suggested artifact template                  |
| `check_suggestion`     | `deterministic_check`   | Additional check suggestion                  |
| `evidence_requirement` | `verify_record`         | Expected verification evidence               |
| `preferred_blueprint`  | route selection request | Compatible route preference, never execution |

## Evidence Model

Blueprints define expected evidence before work starts. Evidence is not always a test result.

| Blueprint        | Evidence examples                                                                  |
| ---------------- | ---------------------------------------------------------------------------------- |
| `analysis.light` | sources, assumptions, confidence, weak links, answer                               |
| `content.light`  | audience, source facts, style check, final artifact                                |
| `docs.change`    | changed docs paths, docs IA check, routing check, doctor result                    |
| `code.branch_pr` | changed paths, fast checks, tests, PR artifacts, hosted checks, integration commit |
| `release.strict` | release plan, version gates, publish workflow evidence, post-publish checks        |
| `ops.approval`   | approval source, target system, action log, rollback path                          |

This keeps non-code tasks rigorous without forcing CI semantics onto them.

## Explain Output

The first user-visible surface is explanation, not execution.

CLI shape:

```text
agentplane blueprint list
agentplane blueprint explain <task-id>
agentplane blueprint explain --kind analysis --mutation none
agentplane blueprint explain <task-id> --json
```

Expected explain fields:

```text
blueprint_id
blueprint_version
route
selection_reasons
workflow_mode
required_evidence
accepted_recipe_extensions
rejected_recipe_extensions
stop_reasons
```

Example:

```text
Blueprint: analysis.light@1
Route: intake -> scope -> context_resolve -> work_unit -> artifact_write -> verify_record -> finish
Reasons:
- task has no repository mutation
- task intent is analysis
- workflow mode does not require PR lifecycle for read-only work
Required evidence:
- sources
- assumptions
- weak links
- final output
Not required:
- worktree
- PR
- CI
- merge
```

## ACR Relationship

Blueprints are pre-execution and during-execution contracts. ACR is post-execution evidence.

v0.5 rc1 records a compact `extensions["agentplane.blueprint"]` object:

```text
blueprint_id
blueprint_version
workflow_mode
route
required_evidence
accepted_recipe_extensions
rejected_recipe_extensions
stop_reasons
```

ACR should not store full prompts, full transcripts, or raw context bundles by default. It should
record enough structured evidence to audit which route was used and why the result is reviewable.

## Runner Relationship

Runner support should be incremental.

v0 should support model, registry, validation, resolution, and explanation:

```text
validate built-ins
resolve from task metadata
explain resolved route
surface expected evidence
```

Later runner work can consume the resolved blueprint as a graph. The first implementation should
not attempt a full workflow engine.

## Context Relationship

Blueprints should eventually constrain context loading, but that is not part of the first code PR.

Future context contract:

- `context_resolve` records why each policy module, recipe, file group, or tool profile was loaded.
- runner bundles expose a context budget and loaded context manifest.
- validation rejects policy modules that are unrelated to the resolved blueprint.
- ACR records context manifest references, not full raw context.

This is broader than ACR. ACR records what happened after execution; the runner bundle and resolver
must explain why context was available before execution.

## Stop Rules

Blueprint resolution must stop or require re-approval when:

- route ambiguity changes mutation behavior;
- route ambiguity changes network, publish, merge, deployment, or security behavior;
- recipe data tries to override a protected node;
- required evidence cannot be produced;
- task kind conflicts with workflow mode;
- analysis or content accidentally resolves to a code PR pipeline;
- runner context would include more policy, tools, or files than the resolved route allows;
- a requested route would bypass existing task lifecycle commands.

## Non-goals for v0

Blueprint v0 should not include:

- a full runner executor;
- a visual workflow editor;
- arbitrary recipe-defined lifecycle graphs;
- CI or PR gates for every task;
- full prompt or transcript capture in ACR;
- replacement of `direct` or `branch_pr`;
- replacement of task status transitions;
- cross-repository orchestration;
- automatic policy-module loading enforcement.

## Enterprise Implication

Blueprints can become the foundation for governed custom agents, but only if the core remains
constrained.

Enterprise-relevant properties:

- routes are formalized instead of being hidden in prose prompts;
- custom agents can choose from approved blueprints;
- recipes can extend context and outputs without overriding protected lifecycle gates;
- auditors can inspect route, evidence, approvals, and stop reasons;
- non-code departments can use lightweight analysis or content routes without inheriting developer
  PR ceremony.

The target is not arbitrary agents with arbitrary lifecycles. The target is custom agents assembled
from approved route contracts, policy modules, recipes, and evidence requirements.
