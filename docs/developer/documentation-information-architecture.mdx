## Why this page exists

This page defines where each topic lives, which code paths are the source of truth, and how to avoid duplicate or stale docs.

Use it as the routing table before editing any documentation page.

Do not publish deferred design plans, implementation drafts, or legacy navigation metadata as active
docs. Keep those details in the archive, ADRs, release notes, or task history instead of the public
runtime guide.

## Section boundaries

### User guide

Use `docs/user/*` for runtime behavior and operator workflows:

- How to initialize and run Agent Plane in a repository.
- What commands do in day-to-day use.
- How tasks, approvals, statuses, and backends behave from a user perspective.

### Developer guide

Use `docs/developer/*` for implementation and maintenance details:

- Package boundaries and architecture.
- CLI contracts, parser behavior, and tests.
- Release and publishing pipeline.
- Contribution standards and quality gates.

### Help

Use `docs/help/*` for diagnosis and recovery:

- Symptom-first troubleshooting.
- Common failure patterns and exact fixes.
- Shared terminology.

### Archive

Use `docs/archive/*`, `docs/adr/*`, and clearly marked historical pages for non-current material:

- Stabilization ledgers and retired refactor programs.
- Accepted decision records.
- Historical incident evidence that no longer belongs in active policy.

Archive pages may remain linkable for provenance, but they must not appear inside the current user
or developer paths as if they were active guidance.

## Agent-first navigation model

The public docs entrypoints should mirror the same workflow model the CLI and gateway use:

1. **Start**: repository entry, setup, bootstrap.
2. **Work on a task**: workflow, lifecycle, commands, roles.
3. **Workflow guides**: copy-paste flows for agents, CI, and review modes around AgentPlane.
4. **Recipes**: installable recipe packages and catalog/runtime behavior.
5. **Upgrade & recover**: upgrade state model, migration, recovery runbooks.
6. **Reference & integrations**: backends, config, generated reference.
7. **Developer**: implementation and maintenance details.
8. **Release notes**: historical record only.
9. **Archive**: historical records, ADRs, and retired program notes.

## Duplication rules

1. Put normative command syntax in command specs and generated CLI reference.
2. Keep narrative explanation in user/developer pages, but link to generated reference instead of copying long option lists.
3. Do not duplicate policy text between docs pages and `AGENTS.md`; summarize and link.
4. Treat release notes as historical records only, not as normative behavior docs.

## Update workflow

1. Find the owning page by using the mapping table above.
2. Confirm behavior in code before editing text.
3. Update navigation and site wiring if a page is added, renamed, or removed:
   - `docs/index.mdx`
   - `website/sidebars.ts`
   - `website/docusaurus.config.ts`
4. If command output changed, regenerate `docs/user/cli-reference.generated.mdx`.
5. Run the docs task verification checklist before closing the task.

## Review checklist for this IA

- Every topic has exactly one primary owner page.
- Every owner page has a concrete code source path.
- Navigation and landing pages point to the same structure.
- No page claims behavior that is not present in code.
