---
title: "Recipes specification"
description: "Contract for recipe archives, manifests, and CLI behavior."
---

Recipes are optional extensions. Core workflow must remain usable with zero installed recipes.

Recipes v1 use a vendor-based lifecycle:

1. remote catalog for discovery and distribution;
2. global cache for download and import;
3. project vendor store for runtime authority;
4. generated overlay and asset bundles as derived artifacts only.

After vendoring, runtime reads recipes only from the project-local vendor store.

This page documents the active recipe v1 contract for the current `0.4` line.

The repository-level docs inventory (`docs/recipes-inventory.json`) is a generated projection sourced
from `agentplane-recipes/index.json` plus the checked-in recipe manifests under
`agentplane-recipes/recipes/*/manifest.json`.

## What a recipe contains

A recipe archive (`.zip` or `.tar.gz`) must include `manifest.json` at archive root or in a single top-level directory.

Optional directories:

- `agents/`
- `assets/`
- `policy/`
- `prompt-modules/`
- `prompts/`
- `README.md`
- `scenarios/`
- `skills/`
- `templates/`
- `tools/`

## Manifest contract

Manifest is validated against the recipe manifest schema in runtime code.

Minimum required fields:

- `schema_version` (`"1"`)
- `id`
- `version`
- `name`
- `summary`
- `description`

First-class sections include `compatibility`, `overlay`, `skills`, `tools`, `agents`, `scenarios`,
`prompt_modules`, `prompt_mutation_sets`, and `blueprint_extensions`.

Contract notes:

- `agents/` and `skills/` are markdown-first assets. Their content must survive packaging without
  being reduced to JSON stubs.
- `tools/` remain structured manifests plus executable content where required.
- `scenarios/` are recipe-owned assets, not a separate top-level package kind.
- Public scenario descriptors must be sufficient for resolver-driven selection without reading unrelated internal files.
- `prompt_modules` point at JSON prompt module files owned by the recipe.
- `prompt_mutation_sets` point at JSON mutation sets that are applied only while compiling the
  project prompt graph; failed mutations do not partially publish generated project artifacts.
- Prompt mutations should target named prompt fragments such as
  `agent.coder.workflow.diff.minimal` or `policy.workflow.branch_pr.sequence.start_worktree`, not
  array indexes, line numbers, or whole-file text replacements.
- The target activation model materializes active recipe mutations into Agentplane-managed prompt
  source files, then regenerates the prompt graph as diagnostics. Recipes may change prompt
  behavior through named fragments, but they must not change core lifecycle, workflow, git hygiene,
  or safety invariants.
- User-authored behavior belongs in `gateway.user.instructions`, normally backed by
  the optional user instructions file under `.agentplane/` (`user-instructions.md`), instead of direct edits to managed prompt files.

## Blueprint extensions

`blueprint_extensions` are advisory route inputs. They let a recipe contribute context, evidence,
checks, output shape, artifact templates, risk notes, or a compatible blueprint preference without
owning lifecycle execution.

Supported extension kinds:

| Kind                   | Fields                                  | Resolver behavior                                    |
| ---------------------- | --------------------------------------- | ---------------------------------------------------- |
| `context_hint`         | `value`, optional `target_node_kind`    | Attaches to `context_resolve` when allowed           |
| `risk_hint`            | `risk`, optional `target_node_kind`     | Attaches to `context_resolve` when allowed           |
| `output_schema`        | `value`, optional `target_node_kind`    | Attaches to `work_unit` when allowed                 |
| `artifact_template`    | `value`, optional `target_node_kind`    | Attaches to `work_unit` when allowed                 |
| `check_suggestion`     | `command`, optional `target_node_kind`  | Attaches to `deterministic_check` when allowed       |
| `evidence_requirement` | `evidence`, optional `target_node_kind` | Attaches to `verify_record` when allowed             |
| `preferred_blueprint`  | `blueprint_id`                          | Accepted only when compatible with task and workflow |

Example:

```json
{
  "blueprint_extensions": [
    {
      "id": "market-research.sources",
      "kind": "evidence_requirement",
      "summary": "Require explicit market source evidence.",
      "evidence": ["sources", "assumptions"],
      "when": { "tags_any": ["research", "analysis"] }
    },
    {
      "id": "market-research.route",
      "kind": "preferred_blueprint",
      "summary": "Prefer the lightweight analysis route for research tasks.",
      "blueprint_id": "analysis.light",
      "when": { "tags_any": ["research", "analysis"] }
    }
  ]
}
```

Hard boundaries:

- recipes cannot disable approvals, workflow-mode constraints, verification, finish, or policy gates;
- a `preferred_blueprint` that would turn analysis/content into `code.branch_pr` is rejected;
- stronger risk routing, such as publish or external-system risk, outranks recipe preference;
- accepted and rejected extensions are visible in blueprint explain output and ACR summaries.

## CLI surface

Global cache commands:

- `agentplane recipes list-remote`
- `agentplane recipes list`
- `agentplane recipes info <id>`
- `agentplane recipes install ...`
- `agentplane recipes cache prune`

Project vendoring commands:

- `agentplane recipes add <id|id@version>`
- `agentplane recipes remove <id>`
- `agentplane recipes active`
- `agentplane recipes explain <id>`
- `agentplane recipes explain-active`
- `agentplane recipes update <id>`
- `agentplane recipes detach <id>`

Recipe-owned scenario assets remain internal to vendored packages; they are no longer part of the
public CLI command surface.

There is no public runner entrypoint in the current release. Recipe-owned task materialization
creates normal tasks for repository agents.

## Install sources

Install supports three source modes:

- `--name <id>` from remote index
- `--path <archive>` from local file
- `--url <archive-url>` from direct URL

Legacy conflict flag for export-oriented installs:

- `--on-conflict fail`
- `--on-conflict rename`
- `--on-conflict overwrite`

The flag is retained as a compatibility no-op. Self-contained recipes are not exported into
`.agentplane/agents`.

## Prompt fragment patching

Recipe-owned prompt modules and mutation sets operate on the compiled prompt graph. Framework prompt
assets may expose source-level fragments that compile into addressable modules. The recipe contract
for those fragments is:

- use stable semantic ids, for example `agent.planner.workflow.verify_steps.contract`;
- patch by selector with `fragment_id` or by module address derived from the fragment id;
- prefer `patch_module` for wording or behavior refinements;
- use `replace_module` when the fragment contract is stable but the whole fragment body must change;
- use `disable_module` only when the target fragment declares a compatible mutability policy;
- use `required_module` or `forbidden_module` validators to assert that expected fragments are
  present or absent after all recipe mutations apply;
- never depend on JSON array index, markdown line number, or current prose text;
- fail transactionally when a target is missing, ambiguous, locked, or produces a compiler error.

Example mutation set:

```json
{
  "schema_version": 1,
  "recipe_id": "review",
  "mutations": [
    {
      "id": "review.patch.coder-diff",
      "op": "patch_module",
      "source": {
        "owner": { "kind": "recipe", "recipe_id": "review", "version": "1.0.0" },
        "provenance": {
          "source_kind": "recipe_asset",
          "source_ref": "prompt-modules/mutations.json",
          "recipe_id": "review",
          "recipe_version": "1.0.0"
        }
      },
      "target": { "fragment_id": "agent.coder.workflow.diff.minimal" },
      "patch": { "content": "Keep diffs minimal, task-scoped, and reviewable.\\n" }
    },
    {
      "id": "review.require.coder-diff",
      "op": "add_validator",
      "source": {
        "owner": { "kind": "recipe", "recipe_id": "review", "version": "1.0.0" },
        "provenance": {
          "source_kind": "recipe_asset",
          "source_ref": "prompt-modules/mutations.json",
          "recipe_id": "review",
          "recipe_version": "1.0.0"
        }
      },
      "validator": {
        "id": "review.requires.coder-diff",
        "phase": "compile",
        "kind": "required_module",
        "target": { "fragment_id": "agent.coder.workflow.diff.minimal" },
        "required": true
      }
    }
  ]
}
```

Recipes that need role-specific behavior should target `agent.<role>.*` fragments. Recipes that need
process policy should target `policy.*` or `gateway.*` fragments only when those fragments are
explicitly marked replaceable or extendable.

Authority rules:

- `.agentplane/recipes/registry.json` is the only project authority for vendored recipes.
- `.agentplane/generated/*` are derived outputs; they must not carry independent authority.
- Runtime reads vendored packages from `.agentplane/recipes/packages/*`, not from the global cache.
- `.agentplane/generated/prompt-graph.json` is the compiled prompt module graph for the current
  active recipe set. Treat it as inspectable derived state, not as the place to edit modules.

Runtime execution artifacts are deferred to the v0.7 runner surface. Current recipe docs should
describe task materialization, provenance, and agent-facing instructions only.

## Agent execution workflow

Recipe scenarios and generated tasks should be inspectable through their task README, plan, Verify
Steps, ACR, and PR artifacts. If work stalls or ownership changes, record a task handoff with the
current status, changed files, verification state, and next action before another agent resumes.

Live runtime smoke probes remain internal v0.7 work and are not part of the current public recipe
contract.

## Resolver contract

A deterministic resolver builds runtime context from:

- active Agentplane version
- supported manifest/scenario/runtime API versions
- current platform
- detected project repo types

Resolver output includes:

- selected recipe/scenario
- compatibility result with explicit reasons/failures
- normalized `run_profile`
- explicit selection reasons

`run_profile` now covers future runtime execution hints only. Global network approval remains governed by
`agents.approvals.require_network`, not by a recipe-scoped `run_profile.network` field.
`run_profile.mode` remains part of resolver selection and CLI discovery output, but it is no longer
projected into an active execution env or capability surface because it functions as a recipe
classification label rather than a shared execution-policy field.
The main-repo recipe contract also no longer accepts `run_profile.requires_human_approval`: the
field was removed from runtime execution surfaces instead of being documented as a fake enforcement
guarantee.
The same rule now applies to `run_profile.expected_exit_contract`: the field has been removed from
the accepted main-repo recipe contract after the bundled catalog was cleaned, because current values
are recipe-specific labels without deterministic shared semantics.
`run_profile.writes_artifacts_to` remains advisory until the v0.7 runner surface reintroduces
enforcement.

## Recipe scenario preview and execution contract

`recipes scenario run` resolves a scenario definition, validates recipe-local file references, and
prints a prepared run plan. It is preview-only: no task is created and no runner artifacts are written.

`recipes scenario execute` uses the same resolver and validation path, then:

1. materializes a normal task from the scenario's explicit `task_template`;
2. records recipe provenance on that task (`recipe_id`, `scenario_id`, `recipe_version`, `run_id`);
3. writes the recipe context into the generated task for the assigned agent.

Required scenario definition support:

- manifest-level scenario selection
- recipe-local file validation (`agents`, `skills`, `tools`, `scenarios`)
- internal scenario definition validation (`goal`, `inputs`, `outputs`, `steps`, `task_template`)
- compatibility failure reporting before preview or execution

`task_template` is required in every executable scenario definition. It must provide:

- `title`
- `description`
- `owner`
- optional `priority`
- optional `tags`
- optional `verify`
- optional `doc` seeds for task README sections

Runtime adapter execution is deferred to the v0.7 runner surface. Until then, keep adapter-specific
requirements in task evidence and recipe metadata rather than exposing execution commands.

Scenario definitions can declare evidence requirements:

- `evidence.required` (boolean)
- `evidence.files` (string array, defaults to `["evidence.json"]` when required)

Evidence declarations are bundled into recipe context for the assigned agent and can seed
verification expectations, but they are not a separate CLI orchestration DSL.

## Security model

Recipe permission declarations are explicit and visible, but they are not a full sandbox.
Treat recipes as executable code and review source and manifest before install.

For shared repositories, `copy` is the safe default. `link` is a development mode and should be
considered non-portable until detached back into a project-local copy.

## Compatibility guidance

- Keep manifest fields backward-compatible.
- Additive fields are preferred over breaking schema changes.
- Avoid environment-specific assumptions in manifest-level scenario descriptors.
- Document required env keys and expected outputs in recipe README.
