## Purpose

This document defines the target release mechanism for AgentPlane without treating the current
implementation as the architectural boundary.

It answers three separate questions:

1. What is the correct release mechanism in `local` and `branch_pr` worlds?
2. How should that mechanism be implemented so releases stay correct under retries, recovery, and
   protected-branch constraints?
3. Where does the current implementation diverge from that target, and what is the smallest
   ordered set of changes that closes those gaps?

## Core release invariants

Any correct release process must preserve these invariants:

1. One release identity = one exact Git commit SHA, one version, one tag, one release note file.
2. Release readiness is decided once per exact SHA by a canonical artifact, not by repeated
   re-interpretation of repository state.
3. Publish is idempotent:
   - rerunning detect does not change state;
   - rerunning publish does not republish already published packages;
   - rerunning post-publish tagging/release steps does not corrupt history.
4. Release planning and release publication are separate:
   - planning chooses the intended version/range/notes target;
   - publication acts only on an already fixed release identity.
5. Protected-main repositories must never depend on direct local tag-and-push as the primary
   publication path.
6. Historical recovery must validate and publish the exact historical SHA, not the moving branch tip
   and not the latest workflow implementation by accident.
7. A release pipeline must have one canonical post-publish record that distinguishes:
   - ready but unpublished;
   - partially published;
   - fully published;
   - published with post-publish artifact drift.

## Canonical release objects

The optimal design should treat these as first-class objects:

1. `release intent`
   - chosen version bump
   - commit range
   - release note target
   - initiating task or operator
2. `release candidate`
   - exact target SHA
   - version
   - tag
   - notes file
   - readiness state
3. `release-ready artifact`
   - exact SHA
   - version
   - tag
   - readiness verdict
   - why the SHA is or is not releasable
4. `publish-result artifact`
   - exact SHA
   - version
   - tag
   - package publication outcomes
   - tag outcome
   - GitHub Release outcome
5. `release ledger entry`
   - durable summary of the release attempt and its terminal state

The current repository already has partial forms of `release-ready` and `publish-result`. The target
design should make them the center of the mechanism, not an implementation detail.

## State model

The release process should be a state machine, not a sequence of ad-hoc commands.

Recommended states:

1. `planned`
   - version and notes target chosen
   - no repository mutation yet
2. `candidate_prepared`
   - version bump and notes are committed on an exact SHA
3. `candidate_validated`
   - exact SHA has a canonical `release-ready` artifact
4. `publishing`
   - publish pipeline is executing for that exact SHA
5. `published`
   - both packages, tag, and GitHub Release are complete
6. `partially_published`
   - at least one publish-side effect happened, but the terminal publish set is incomplete
7. `superseded`
   - candidate replaced before publish
8. `aborted`
   - candidate intentionally abandoned before publish

Transitions should be monotonic. A later rerun should never silently reinterpret an older state.

## Target mechanism: local mode

`local` mode should optimize for one checkout and minimal coordination overhead.

### Ideal flow

1. Plan the release.
   - Freeze `prev tag`, `target version`, `target tag`, and `notes path`.
   - Refuse to skip unpublished patches.
2. Prepare the release candidate in the current checkout.
   - Write notes.
   - Apply version bump.
   - Commit one exact candidate SHA.
3. Validate the exact candidate SHA.
   - Run the local release gate.
   - Produce a local candidate manifest.
   - Optionally mint or request the hosted canonical `release-ready` artifact for that same SHA.
4. Publish from the exact candidate SHA.
   - Publish packages.
   - Push tag.
   - Create GitHub Release.
   - Persist publish-result.
5. Record release evidence.

### What local mode should not do

1. It should not choose the next version from the workspace version alone.
2. It should not publish from a moving branch tip after planning.
3. It should not re-run unrelated broad CI to decide historical publishability.
4. It should not require a second synthetic closeout path just to complete release bookkeeping.

## Target mechanism: branch_pr mode

`branch_pr` mode is different in kind, not just in transport.

The protected branch is the publication authority. That means the release process must be based on
promotion of a release candidate into `main`, not on local mutation pretending it can publish
independently.

### Ideal flow

1. Plan the release on base.
   - Freeze intended version, notes target, and release range.
2. Create a dedicated release candidate branch/worktree.
   - This should be a first-class release branch, not just an arbitrary task branch.
3. Prepare the release candidate on that branch.
   - Write notes.
   - Apply version bump.
   - Commit one exact candidate SHA.
4. Open a hosted PR for the release candidate.
   - Hosted checks validate that exact candidate SHA.
   - The release PR body should reference the release intent and exact target version.
5. Merge the hosted release PR into `main`.
   - `main` becomes the only branch from which automatic publish can proceed.
6. Publish from the merged SHA on `main`.
   - `Core CI` produces the canonical `release-ready` artifact for the merged SHA.
   - `Publish to npm` consumes that artifact and publishes from the merged SHA only.
7. Run deterministic closeout.
   - If local release artifacts or task state still need reconciliation, use the same hosted-close
     mechanism as ordinary `branch_pr` work, but keep it outside the publication-critical path.

### Key architectural rule

In `branch_pr`, the release candidate branch prepares a releasable SHA.
It does not itself own the final publish act.

That boundary is what prevents protected-main repositories from drifting into a half-direct,
half-PR-based release flow.

## Target mechanism: historical recovery

Historical recovery is not “rerun current CI on old code until green”.
It is “recover publication of an already chosen exact release SHA”.

### Correct historical recovery flow

1. Identify the intended historical release SHA.
2. Resolve or recreate the canonical `release-ready` artifact for that exact SHA.
3. Validate only release-critical properties for that SHA:
   - version parity
   - notes existence and tag/version match
   - package buildability / packability
   - registry burn checks
4. Publish from that exact SHA using the current orchestration layer.
5. Record a publish-result artifact and release ledger entry.

### Recovery boundary

Historical recovery should use current orchestration code and workflow logic, but the publish
subject must remain the historical SHA.

That means:

1. current runtime may resolve artifacts and drive recovery;
2. historical checkout provides the payload being published;
3. broad modern regression suites must not be the gate for historical release recovery.

## Optimal implementation principles

The best implementation should follow these constraints:

1. Single source of release truth
   - exact SHA is the anchor, not branch name, not current workspace version, not latest plan file
2. Separation of concerns
   - planning
   - candidate preparation
   - readiness validation
   - publication
   - closeout / cleanup
3. Idempotent workflows
   - every workflow can be rerun safely against the same release identity
4. Explicit release routing
   - `local` and `branch_pr` should be modeled as separate routes with shared invariants, not as one
     route with late branching and special cases
5. Stable artifact contracts
   - readiness and publish outcomes must be machine-readable and durable
6. Narrow validation contours
   - infra-only release recovery should not inherit unrelated broad CI surfaces
7. Automatic cleanup after merge
   - merged task/release branches and worktrees should be deleted automatically when safe
   - cleanup must be outside the correctness-critical publication decision

## Comparison with the current implementation

The current implementation has real strengths, but it still mixes architectural layers.

### What is already strong

1. Exact-SHA `workflow_dispatch` recovery exists for `Core CI` and `Publish to npm`.
2. `release-ready` is already treated as the canonical readiness artifact.
3. `publish-result` already exists as a canonical post-publish artifact.
4. Hosted task closure now opens and merges follow-up closure PRs automatically when org policy
   allows it.
5. Merged task branches/worktrees can already be auto-pruned locally.
6. Planner now refuses to skip unpublished patch versions.

### Where the current implementation still diverges from the target

1. Release route semantics are still mixed.
   - The CLI still presents `agentplane release apply --push --yes` as the dominant operator path.
   - In protected-main `branch_pr` repositories, the real publish authority is hosted merge into
     `main`, not direct local push/tag.
2. Release candidate is not a first-class concept.
   - Current `branch_pr_candidate` behavior is implemented as a deferred variation of `release apply`
     instead of a dedicated release-PR route.
3. Release closeout is still partially task-shaped.
   - Release completion still relies on task-style traces and recovery conventions that were designed
     for ordinary work items, not for release promotion.
4. Local PR artifact churn still leaks into the path.
   - `agentplane pr open|update` can rewrite tracked artifacts after new commits, which creates
     avoidable artifact-only commits and extra operator steps.
5. Release docs are still biased toward the current implementation.
   - The current release doc explains the implemented route well, but it does not clearly separate
     architectural intent from present-day command topology.
6. Infra-only release fixes still pay a broad pre-push tax.
   - Workflow/docs-only changes can still trigger a broad `full-fast` local gate even when the
     actual blast radius is much narrower.
7. Cleanup is still distributed.
   - branch/worktree cleanup, hosted closure, release readiness, and publish recovery now work much
     better than before, but the system still spreads closeout behavior across several partially
     coupled paths instead of one explicit release ledger model.

## Target architecture decisions

If the release system were redesigned from first principles, these would be the recommended hard
choices:

1. `branch_pr` release = release candidate PR into `main`
   - not a direct local publish route with deferred publish semantics
2. Release readiness = canonical hosted artifact
   - always anchored to exact SHA
3. Publish = GitHub-only orchestration
   - even when initiated locally
4. Historical recovery = current orchestrator + historical publish subject
5. Cleanup = automatic, deterministic, and outside the publication-critical path
6. Release ledger = first-class record
   - richer than scattered task notes and ad-hoc workflow summaries

## Atomic improvement plan

The improvement plan below is ordered so each step is independently shippable and reduces one class
of release friction without requiring a full redesign first.

### 1. Split release architecture docs from implementation docs

Scope:

- keep `docs/developer/release-and-publishing.mdx` implementation-focused
- add this architecture doc as the target-model reference

Effect:

- future release changes can be judged against an explicit target design instead of the latest
  implementation accident

Weakest link if skipped:

- release fixes keep optimizing local minima and reintroducing route confusion

### 2. Introduce a first-class release candidate route for `branch_pr`

Scope:

- separate “prepare release candidate PR” from “publish release”
- make release PRs explicit, not a deferred side effect of `release apply`

Effect:

- protected-main repositories stop mixing direct-release semantics with PR-promotion semantics

Key check:

- a release candidate can be prepared and reviewed without implying that local push/tag is the final
  publish authority

### 3. Make release identity first-class in CLI and artifacts

Scope:

- expose `release intent`, `candidate`, `ready`, and `published` states explicitly
- standardize one machine-readable release ledger entry per attempt

Effect:

- recovery no longer depends on reconstructing state from scattered files and workflow runs

Key check:

- any operator can answer “what is the state of release X?” from one canonical record

### 4. Remove task-artifact coupling from the release path

Scope:

- keep release metadata separate from ordinary task PR artifacts
- avoid artifact-only commits caused by `pr open|update` when the change is release-only

Effect:

- release flow becomes shorter and less vulnerable to PR metadata churn

Key check:

- preparing a release candidate does not require unrelated review-packet rewrites

### 5. Narrow local pre-push contours for release-infra-only changes

Scope:

- classify release-workflow/docs-only changes into a narrower local gate
- preserve broad/full gates only where the blast radius actually requires them

Effect:

- release-infra fixes iterate faster without lowering release correctness

Key check:

- workflow/docs-only changes no longer fall into the same local gate as broad runtime mutations

### 6. Unify exact-SHA recovery around current orchestration runtime

Scope:

- ensure all recovery resolution happens through current orchestration code
- historical checkout is only the payload source

Effect:

- historical releases stop failing because old repository scripts lacked future recovery behavior

Key check:

- the same resolver logic works for current publish and historical recovery

### 7. Make auto-cleanup a canonical post-merge capability

Scope:

- after successful merge into `main`, automatically remove safe merged branches/worktrees
- preserve current-worktree and outside-repo safety boundaries

Effect:

- no manual tail remains after a successful merge except explicit diagnostics

Key check:

- a merged task or release PR leaves no stale branch/worktree residue when cleanup is safe

### 8. Add an explicit release operations ledger

Scope:

- record plan creation, candidate SHA, readiness source, publish result, and recovery attempts

Effect:

- postmortems and recovery become ledger reads, not archaeology

Key check:

- failed or partial releases can be resumed from the ledger without reconstructing intent manually

## Recommended near-term target

The most effective near-term endpoint is this:

1. `local`
   - prepare and validate an exact candidate SHA locally
   - publish through GitHub orchestration
2. `branch_pr`
   - prepare a release candidate PR
   - merge into `main`
   - let hosted automation publish from merged `main` SHA
3. `historical recovery`
   - recreate readiness for exact SHA
   - publish from exact SHA with current orchestration

That model matches protected-main reality, keeps exact-SHA recovery sound, and reduces the number of
distinct paths operators must reason about.
