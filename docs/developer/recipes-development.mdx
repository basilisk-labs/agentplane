## Authoring checklist

1. Create recipe directory with `manifest.json`.
2. Add `agents/` and `skills/` as markdown-first assets; keep JSON only where structure is required.
3. Add `tools/` and `scenarios/` only when needed.
4. Keep ids stable (`recipe id`, `tool id`, `scenario id`).
5. Document required env keys in recipe README.
6. Keep outputs deterministic and machine-readable.

## Minimal local validation

1. Build archive (`.tar.gz` or `.zip`).
2. Cache locally:
   - `agentplane recipes install --path <archive>`
3. Verify cache visibility:
   - `agentplane recipes list`
4. Vendor the recipe into a project:
   - `agentplane recipes add <id>`
5. Inspect project-local state:
   - `agentplane recipes active`
   - `agentplane recipes explain <id>`
6. Confirm recipe-owned scenario assets stay internal and are not documented as a public CLI flow.
7. Confirm no additional runtime assumptions leaked into docs or tests.
8. Regenerate the published recipes inventory if recipe manifests changed:

- `node scripts/generate-recipes-inventory.mjs`

## Designing scenarios

- Keep the manifest-level descriptor self-sufficient for selection.
- Keep internal scenario files explicit and forward-compatible with future agent execution.
- Explicitly define expected outputs, required inputs, and failure messages.
- Do not hide required inputs behind implicit defaults.
- Keep `use_when` descriptive, but rely on structured fields for deterministic selection.
- Every executable scenario must define `task_template`; do not synthesize task title, owner, or verification steps implicitly.
- Use `task_template.doc.*` only for task seeding. Do not treat it as a second execution language.

## Designing tools

- Declare minimal permissions in manifest.
- Keep tool side effects inside project-local directories when possible.
- Prefer stable stdout/stderr output formats for downstream automation.
- Exit non-zero on failures and include actionable error text.
- Assume live execution happens through normal task materialization and a repository agent; do not promise a separate public scenario CLI.
- Do not rely on CLI-side interpretation of recipe `steps`; the assigned agent receives them as context, not as an imperative DSL.

## Distribution and upgrades

- Bump recipe version on each behavioral change.
- Keep backward-compatible upgrades when possible.
- Avoid removing scenario ids without migration path.
- Keep remote catalog signatures on the production `key_id` trusted by the current Agentplane CLI.
- Store the recipes signing private key only in the `agentplane-recipes` GitHub Actions secret
  `RECIPES_INDEX_SIGNING_PRIVATE_KEY`; do not put it in `.env`, repository files, release assets, or
  long-lived local key paths.
- Rotate signing keys by adding the new public key to Agentplane, publishing a CLI release with that
  trust-root, and only then switching the default remote catalog signature to the new `key_id`.

## Integration notes

- `recipes install` populates only the global cache; it does not make a recipe active for a project.
- `recipes add` vendors a cached recipe into `.agentplane/recipes/packages/<id>/` and records authority in `.agentplane/recipes/registry.json`.
- Recipe-local agents stay inside `.agentplane/recipes/packages/<id>/agents/`; they are not installed into `.agentplane/agents`.
- Runtime reads vendored project-local recipes only; generated overlay/assets files are derived from the project registry.
- Internal recipe-owned task materialization creates a normal task with recipe provenance for an assigned repository agent.
- Validate both cache install and project vendoring flows before release.
- Use extension recipes for non-core integrations (for example Dokploy or custom external APIs).
