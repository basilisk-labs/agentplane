AgentPlane prompt text is assembled from typed modules instead of ad hoc string templates.
The migration boundary covers gateway files, policy modules, agent profiles, runner prompts, and
recipe-owned prompt extensions.

| -------------------- | ------------------------------------------------------------------------------------------------------- | --------------------------------------------- | --------------------------------------------------------------------------------- |
| Policy gateway       | `packages/agentplane/assets/AGENTS.md`                                                                  | framework prompt module registry              | `AGENTS.md` during `agentplane init` and upgrade baseline seeding                 |

## Managed source model

AgentPlane-managed prompt files are the visible behavior surface:

- `AGENTS.md` / `CLAUDE.md` for the policy gateway;
- `.agentplane/policy/**` for canonical policy modules;
- `.agentplane/agents/*.json` for role profiles.

These files are human-readable and git-tracked, but direct edits are drift unless they happen through
an AgentPlane command. Recipe activation should update them through fragment-aware materialization,
then regenerate `.agentplane/generated/prompt-graph.json` as the diagnostic index for the same
behavior.

User-authored instructions are the exception. They belong in the `gateway.user.instructions`
fragment, with `.agentplane/user-instructions.md` as the default editable file. The gateway should
only contain the stable hook that loads this block when it exists.

## Contracts

Prompt module domain types live under `packages/agentplane/src/runtime/prompt-modules/`.
The core contracts are:

- `PromptModule`: addressable prompt content with owner, provenance, load condition, mutability, and merge policy.
- `PromptModuleGraph`: a list of module nodes, with optional replacement/extension relationships.
- `PromptModuleMutationSet`: recipe-owned operations such as add, replace, patch, disable, bind, and validator changes.
- `PromptModuleCompiledGraph`: compiled nodes plus bindings, validators, diagnostics, and `ok`.

## GPT-5.5 prompt contract

Framework-owned prompts should stay outcome-first. Agent profiles define the target result,
success criteria, binding constraints, stop rules, and output shape. Detailed workflow gates remain
in `AGENTS.md` and policy modules so role prompts do not duplicate command procedures.

Use absolute language only for true invariants: safety, approval gates, side effects, required
fields, irreversible actions, and output contracts. Express judgment calls as decision rules or
preferences. This keeps model behavior extensible while preserving enforcement-backed constraints.

The shipped profile shape is:

```text
Goal -> Success criteria -> Constraints -> Stop rules -> Output
```

Runtime-specific behavior belongs in config and runner context, not in duplicated profile prose:

- `execution.reasoning_effort` controls reasoning depth; `medium` is the default, `low` is
  latency-oriented, and `high` / `xhigh` are for conservative profiles or eval-backed work.
- `execution.text_verbosity` controls output length and structure, independent of reasoning depth.
- Retrieval budgets live in gateway/runner rules: load matched policy, task README, Verify Steps,
  and relevant files; use incidents only for analogous scope or tags; stop when evidence is enough
  to satisfy the acceptance contract.
- Tool-heavy work should have a short visible preamble and meaningful progress updates, while the
  final answer should report actions, evidence/checks, blockers or drift, and next approval.

## Prompt fragments

Prompt fragments are the source-level units inside prompt assets. They are smaller than a prompt
file and compile into prompt modules with stable addresses. A recipe must target fragments by id,
not by line number, array index, or prose matching.

Fragment ids use lowercase dot-case:

```text
gateway.agents.load_rules.workflow.branch_pr
policy.workflow.branch_pr.sequence.start_worktree
policy.dod.core.traceability
agent.coder.workflow.diff.minimal
agent.orchestrator.workflow.task_graph.atomic
runner.lifecycle.no_recursive_task_run
```

The id format is:

```text
<surface>.<target-or-role>.<slot>.<semantic-name>
```

Use these surface prefixes:

- `gateway.*` for `AGENTS.md` / `CLAUDE.md` gateway fragments;
- `policy.*` for `.agentplane/policy/**` fragments;
- `agent.<role>.*` for `.agentplane/agents/<ROLE>.json` fragments;
- `runner.*` for bundled runner prompt fragments;
- `recipe.<id>.*` only for recipe-owned prompt assets.

Semantic names must describe the behavioral contract, not the current wording. For example,
`agent.coder.workflow.diff.minimal` is stable across wording edits; `agent.coder.workflow.line7`
is not.

### Markdown marker contract

Markdown prompt assets use HTML comments as source markers:

```md
<!-- ap:fragment id="policy.workflow.branch_pr.sequence.start_worktree" slot="workflow" mutability="replaceable" -->

2. Start work with dedicated task branch + worktree.
<!-- /ap:fragment -->
```

Rules:

- marker ids must be unique inside the compiled graph;
- markers must not be nested;
- rendered installed files should omit source markers unless a task explicitly scopes marker
  preservation;
- unmarked markdown remains a fallback whole-file fragment until the file is migrated;
- source markers are implementation metadata, not policy text.

### Agent profile fragment contract

Agent profile JSON should use compact keyed objects for patchable prompt fragments. Keys are local
semantic names; the loader derives the full fragment id from the role, slot, and key:

```json
"workflow": {
  "diff.minimal": "Keep diffs minimal, task-scoped, and easy to review.",
  "verify.evidence": {
    "text": "Run declared checks and record evidence.",
    "mutability": "append_only"
  }
}
```

The example above compiles to `agent.coder.workflow.diff.minimal` and
`agent.coder.workflow.verify.evidence`. Use object values only when metadata such as `mutability`
or `slot` differs from the slot default.

This is the preferred storage format for bundled agent profiles because it avoids repeating full
ids in JSON while preserving stable patch targets. The full fragment id is derived as
`agent.<role>.<slot>.<local-key>`, so recipe and project overrides can target the compiled graph
without making source profiles verbose.

The loader must continue to accept legacy string arrays and structured `{ "id", "text" }` arrays:

```json
"workflow": ["Keep diffs minimal, task-scoped, and easy to review."]
```

String entries are compatibility input only. They are not stable patch targets unless the loader
derives a deterministic compatibility id and reports that provenance as generated. Structured
arrays remain accepted for backward compatibility, but new bundled profiles should prefer compact
keyed objects for `inputs`, `outputs`, `permissions`, and `workflow`.

### Fragment mutability

Use the narrowest mutability that supports the intended extension point:

- `locked`: recipes may depend on the fragment but must not alter it;
- `replaceable`: one recipe or project override may replace the fragment;
- `extendable`: recipes may append or prepend related content around the fragment;
- `append_only`: recipes may add follow-up content but must preserve the original fragment.

Default framework policy and safety gates should be `locked` or `append_only`. Role behavior,
workflow ergonomics, and recipe-specific context can be `replaceable` or `extendable` when the
contract is stable enough for third-party recipes.

Modules must carry stable provenance:

- framework modules use `source_kind: "framework_builtin"`;
- repo-local overrides use `owner.kind: "project"` or `source_kind: "project_file"`;
- recipe modules use `owner.kind: "recipe"` and `source_kind: "recipe_asset"`;
- generated/runtime context uses `source_kind: "generated"` or `"runtime"`.

This provenance is not decorative. It is what lets `runtime explain`, `doctor`, and future recipe
layers show which source owns a prompt fragment and whether a repo override is involved.

## Init and upgrade path

`agentplane init` compiles gateway, policy, and agent profile files from the framework prompt module
registry. It also seeds `.agentplane/.upgrade/baseline/**` from the same compiled outputs.

The important invariant is that installed files and upgrade baselines come from the same compiled
module graph. Do not update `AGENTS.md`, `.agentplane/policy/**`, or `.agentplane/agents/**`
generation by adding a parallel string assembly path.

When changing framework-owned prompt assets:

1. update the source asset or registry mapping;
2. keep source provenance derivable from `provenance.source_ref`;
3. run the init regression tests;
4. run `agentplane doctor` to catch managed policy tree drift.

## Runner path

The runner still consumes ordered prompt blocks, but those blocks pass through the prompt module
bridge before the bundle is emitted. That bridge assigns module addresses, owner, source kind, and
content hashes to:

- bundled runner prompt;
- repository gateway;
- owner profile;
- execution profile runtime block;
- project skill index;
- overlay prompt fragments;
- recipe agent, skill, tool, and execution-context blocks.

This keeps the runner output stable while making each block explainable as a prompt module.
New runner prompt sources should be added as prompt blocks first, then round-tripped through
`runnerPromptBlocksToModuleGraph` and `compileRunnerPromptModuleGraph` in tests.

The current Codex adapter uses `codex exec --json --output-last-message` and stores the raw CLI
trace plus the assistant last message. It does not replay Responses API output items. If a future
Responses API adapter is added, it must preserve intermediate `phase: "commentary"` and final
`phase: "final_answer"` semantics in both trace capture and final output handling.

## Recipe layer

Recipe manifests may declare:

- `prompt_modules`: JSON prompt module assets owned by the recipe;
- `prompt_mutation_sets`: JSON mutation sets that modify the compiled prompt graph.

Mutation sets can target named fragments directly with `target.fragment_id`. The selector works for
gateway, policy, runner, and agent profile modules because source fragments compile into normal
prompt modules with `provenance.fragment_id`.

```json
{
  "op": "patch_module",
  "target": { "fragment_id": "agent.coder.workflow.diff.minimal" },
  "patch": {
    "content": "Keep diffs minimal, task-scoped, and reviewable.\\n"
  }
}
```

Use the same selector shape for `replace_module`, `disable_module`, `required_module`, and
`forbidden_module`. A recipe that changes a single CODER behavior should patch
`agent.coder.workflow.<name>` rather than replacing the whole `CODER.json` profile or the whole
runner bundle.

JSON prompt profile lines are not line-addressable. New patchable entries should use compact local
keys so the compiled graph can expose a stable derived `fragment_id`; legacy
`{ "id": "...", "text": "...", "mutability": "..." }` entries remain readable.

Project recipe state is transactionally published. When active recipe prompt mutations fail to
compile, the registry and generated prompt graph remain unchanged.

Generated recipe artifacts are:

- `.agentplane/generated/overlay-bundle.json`;
- `.agentplane/generated/recipe-assets.json`;
- `.agentplane/generated/prompt-graph.json`.

These files are derived outputs. The project authority remains `.agentplane/recipes/registry.json`
plus vendored packages under `.agentplane/recipes/packages/**`.

## Diagnostics

Use `agentplane runtime explain` to inspect the local prompt graph state. Its prompt graph section
reports:

- generated artifact state: `not_configured`, `missing`, `current`, `stale`, `invalid`, or `compile_error`;
- active recipe ids;
- module counts by owner and source kind;
- repo override count;
- binding, validator, and diagnostic counts;
- compiled module provenance.

Use `agentplane doctor` for drift detection. Doctor reports:

- missing generated prompt graph when active recipes exist;
- stale generated prompt graph relative to the current recipe inputs;
- invalid generated prompt graph payloads;
- expected prompt graph compile failures;
- compiler diagnostics persisted in the generated graph.

Doctor is read-only for this surface. It reports the refresh command path but does not mutate
recipe artifacts.

## Verification matrix

Use these focused checks when changing modular prompt assembly:

```bash
bun run agents:check
bun test packages/agentplane/src/runner/context/base-prompts.test.ts \
  packages/agentplane/src/cli/run-cli/commands/init/steps/apply.test.ts \
  packages/agentplane/src/commands/recipes.transaction.test.ts \
  packages/agentplane/src/commands/doctor.command.runtime.test.ts
bun run typecheck
bun run framework:dev:bootstrap
agentplane doctor
node .agentplane/policy/check-routing.mjs
git diff --check
```

Run recipe package tests too when changing the manifest schema or recipe validation package:

```bash
bun test packages/recipes/src/index.test.ts packages/recipes/src/overlay.test.ts
```

## Safe-change rules

- Keep generated files derived; do not make `.agentplane/generated/*` an authority source.
- Keep module provenance populated and deterministic.
- Keep recipe mutation failures transactional.
- Keep init/upgrade outputs compiled from prompt modules, not from duplicate templates.
- Keep new agent profile entries in compact keyed form; legacy arrays are accepted for old projects,
  not preferred for new bundled prompts.
- Keep reasoning effort and text verbosity in execution config or runner context instead of copying
  those runtime knobs into every role profile.
- Keep runner behavior stable by round-tripping prompt blocks through modules before changing sort
  order, roles, or prompt content.
- Keep doctor/runtime diagnostics read-only unless a separate task explicitly scopes auto-repair.
