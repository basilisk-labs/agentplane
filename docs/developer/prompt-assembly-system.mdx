This page is a deferred design note, not active runtime guidance.

The current `0.3.23` line still relies on the existing `init`/`upgrade` managed-file model plus the
current recipe overlay behavior. The module-graph compiler described below is preserved as future
design context only and is intentionally off the active foundation execution path until `0.3`
stabilization is complete.

The planned AgentPlane v0.4 prompt system treats prompt files as compiled artifacts from a named
module graph.

This target model is a breaking change from the current install model, where `agentplane init` writes bundled
framework files directly into the project and recipes add runtime overlays beside that base. The
v0.4 model keeps `AGENTS.md` or `CLAUDE.md` as the human and agent entrypoint, but the entrypoint is
assembled from addressable modules that can be added, patched, replaced, or bound by active recipes.

## Goals

- Keep `AGENTS.md` compact and deterministic as the first file an agent reads.
- Preserve lazy loading: additional files are loaded only when routing rules, role activation, task
  phase, or recipe conditions require them.
- Make every prompt behavior addressable by stable module id instead of anonymous markdown text.
- Let recipes mutate global behavior without losing provenance, reversibility, or upgrade safety.
- Unify the internal representation of gateway sections, policy modules, agent instructions,
  validators, and templates while allowing each surface to compile to its own file format.
- Explain the final prompt filesystem through generated manifests.

## Non-goals

- Recipes must not be arbitrary text patchers for `AGENTS.md`.
- Generated manifests do not replace `AGENTS.md` as the entrypoint.
- Recipe modules must not silently override sealed framework safety rules.
- Runner-only prompt blocks must not be the only way recipe behavior reaches agents that read the
  repository files directly.

## Core model

The planned v0.4 compiler resolves this graph:

```text
framework base modules
+ active recipe mutations
+ repo-local overrides
= compiled prompt filesystem
```

The compiled filesystem still contains ordinary files:

- `AGENTS.md` or `CLAUDE.md`
- `.agentplane/policy/*.md`
- `.agentplane/agents/*`
- `.agentplane/recipes/packages/<recipe-id>/**`
- `.agentplane/generated/prompt-manifest.json`
- `.agentplane/generated/prompt-graph.json`

The difference is authority. The source of behavior is the module graph and recipe registry; the
files are compatibility outputs for agents, IDEs, review, and upgrade inspection.

## Module

A prompt module is a named behavior unit. It is not the same thing as a file.

One module can compile into:

- a section inside `AGENTS.md`;
- a policy file or section under `.agentplane/policy/`;
- an agent profile instruction;
- a recipe-local policy or role file;
- a validator;
- a task, PRD, ADR, or roadmap template;
- a runner prompt block.

Minimum module fields:

```json
{
  "schema_version": 1,
  "id": "recipe.tdd.coder.loop",
  "title": "TDD implementation loop for CODER",
  "kind": "role_instruction",
  "surface": "agent",
  "target": "agent:CODER",
  "slot": "agent.CODER.implementation_rules",
  "owner": {
    "type": "recipe",
    "id": "tdd",
    "version": "1.0.0"
  },
  "priority": 430,
  "load": {
    "mode": "conditional",
    "roles": ["CODER"],
    "phases": ["coding"],
    "when": {
      "task_kinds": ["feature", "bugfix", "refactor"],
      "tags_any": ["code", "backend", "frontend"]
    }
  },
  "mutability": {
    "sealed": false,
    "replace": "allowed",
    "patch": "allowed",
    "disable": "allowed"
  },
  "merge": {
    "strategy": "append",
    "conflict_policy": "higher_priority_wins"
  },
  "content": {
    "type": "markdown",
    "body": "For behavior changes, write or update a failing test before implementation."
  },
  "provenance": {
    "source": ".agentplane/recipes/packages/tdd/prompts/coder-loop.md",
    "content_sha256": "<sha256>"
  }
}
```

## Module kinds

| Kind                 | Purpose                                                  | Typical target                         |
| -------------------- | -------------------------------------------------------- | -------------------------------------- |
| `gateway`            | Whole entrypoint file contract.                          | `AGENTS.md`                            |
| `gateway_section`    | Named section inside the entrypoint.                     | `gateway.load_rules`                   |
| `policy`             | Normative process or safety rule.                        | `.agentplane/policy/*.md`              |
| `policy_load_rule`   | Conditional routing from the gateway to a policy module. | `gateway.load_rules.*`                 |
| `role_profile`       | Role identity, boundaries, inputs, and outputs.          | `agent:CODER`                          |
| `role_instruction`   | Role-scoped behavior.                                    | `agent.PLANNER.decomposition_strategy` |
| `workflow`           | Process route or lifecycle behavior.                     | `workflow.direct.sequence`             |
| `task_decomposition` | Strategy for converting goals into task graphs.          | `agent:PLANNER`                        |
| `quality_gate`       | DoD or acceptance rule.                                  | `.agentplane/policy/dod.code.md`       |
| `naming_rule`        | Branch, task, file, artifact, or PR naming rule.         | `policy.workflow.branch_pr`            |
| `artifact_template`  | Structured artifact template.                            | `template:prd`                         |
| `validator`          | Machine-checkable requirement.                           | `validator:verification`               |
| `runner_context`     | Prompt block for runner executions.                      | `runner:task_run`                      |
| `recipe_overlay`     | Recipe-specific prompt addition.                         | `surface:coding`                       |

## Surfaces and targets

`surface` identifies the prompt plane:

- `gateway`
- `policy`
- `agent`
- `runner`
- `validator`
- `template`
- `docs`

`target` identifies where the module applies:

- `AGENTS.md`
- `.agentplane/policy/dod.code.md`
- `agent:ORCHESTRATOR`
- `agent:PLANNER`
- `runner:task_run`
- `validator:start_ready`
- `template:prd`

This split matters because a recipe can affect several surfaces at once. A TDD recipe can add a
`role_instruction` for `CODER`, a `quality_gate` for `dod.code`, and a `validator` for verification.

## Slots

Slots are named attachment points. They prevent recipes from editing arbitrary text.

Gateway slots:

- `gateway.header`
- `gateway.purpose`
- `gateway.sources_of_truth`
- `gateway.scope_boundary`
- `gateway.commands.preflight`
- `gateway.load_rules.always`
- `gateway.load_rules.workflow`
- `gateway.load_rules.recipes`
- `gateway.must`
- `gateway.dod.summary`
- `gateway.canonical_docs`
- `gateway.change_control`

Policy slots:

- `policy.workflow.direct.sequence`
- `policy.workflow.direct.constraints`
- `policy.workflow.branch_pr.sequence`
- `policy.dod.core.required_sections`
- `policy.dod.code.test_requirements`
- `policy.dod.code.verification_evidence`
- `policy.dod.docs.documentation_checks`

Agent slots:

- `agent.<ROLE>.identity`
- `agent.<ROLE>.inputs`
- `agent.<ROLE>.outputs`
- `agent.<ROLE>.permissions`
- `agent.<ROLE>.workflow`
- `agent.<ROLE>.planning_behavior`
- `agent.<ROLE>.decomposition_strategy`
- `agent.<ROLE>.implementation_rules`
- `agent.<ROLE>.testing_rules`
- `agent.<ROLE>.handoff_rules`

## Mutability

Not every base module can be changed by recipes.

| Mutability    | Meaning                                                                    |
| ------------- | -------------------------------------------------------------------------- |
| `sealed`      | Cannot be patched, replaced, or disabled by recipes.                       |
| `append_only` | Recipes can add stricter or narrower modules, but cannot remove base text. |
| `patchable`   | Recipes can add content into declared slots.                               |
| `replaceable` | Recipes can replace the module when compatibility rules allow it.          |
| `optional`    | Recipes or repo overrides can disable the module.                          |

Framework safety modules should usually be `sealed` or `append_only`. Examples:

- source-of-truth ordering;
- approval gates;
- outside-repo and network access restrictions;
- protected path rules;
- task traceability requirements.

Strategy modules can be replaceable. Examples:

- task decomposition;
- PRD-first planning;
- TDD workflow;
- naming conventions;
- artifact templates.

## Recipe mutations

A recipe does not patch files. It declares operations against the prompt graph.

Allowed operations:

- `add_module`: introduce a new module.
- `patch_module`: add content to a declared slot.
- `replace_module`: replace a replaceable module.
- `disable_module`: disable an optional module.
- `bind_module`: attach a module to a target, slot, role, phase, or condition.
- `add_validator`: add a machine-checkable rule.

Example:

```json
{
  "op": "patch_module",
  "target": "policy.dod.code.test_requirements",
  "mode": "append",
  "module": "recipe.tdd.policy.red_green_refactor"
}
```

Invalid operation:

```json
{
  "op": "text_patch",
  "target": "AGENTS.md",
  "find": "## MUST / MUST NOT",
  "insert_after": "Always use TDD."
}
```

The invalid operation is intentionally unsupported because it loses slot semantics, provenance,
conflict detection, and reversibility.

## Bindings

Modules are not automatically active. A binding decides where and when a module participates.

```json
{
  "id": "recipe.tdd.binding.coder_loop",
  "kind": "binding",
  "module": "recipe.tdd.coder.loop",
  "bind_to": {
    "surface": "agent",
    "target": "agent:CODER",
    "slot": "agent.CODER.implementation_rules"
  },
  "when": {
    "tags_any": ["code", "backend", "frontend"]
  }
}
```

Bindings make these cases explicit:

- a recipe ships a module but does not activate it by default;
- one module is attached to several roles;
- a repo-local override rebinds a recipe module to a narrower condition;
- `agentplane prompts explain` can show why a module was included or skipped.

## AGENTS.md hierarchy

The entrypoint remains the first file agents read.

`AGENTS.md` should contain:

1. the gateway metadata header;
2. source-of-truth order;
3. startup/preflight commands;
4. load rules;
5. role activation rule;
6. module manifest pointer;
7. change-control and safety boundaries.

`AGENTS.md` should not contain:

- full recipe documentation;
- every role instruction;
- large templates;
- generated runner context;
- arbitrary recipe text.

Lazy loading keeps the entrypoint compact:

```text
AGENTS.md
  -> load workflow policy only for active workflow mode
  -> load DoD modules only for the task type
  -> load recipe policy modules only when active recipe bindings match
  -> load agent role modules only when that role is active
  -> load validators only when their phase is reached
```

## Agent profiles

Agent profiles become compiled views over agent modules.

Source graph:

```json
{
  "id": "agent.CODER",
  "kind": "role_profile",
  "modules": [
    "framework.agent.CODER.identity",
    "framework.agent.CODER.workflow",
    "framework.agent.CODER.implementation_rules",
    "recipe.tdd.coder.loop",
    "repo.agent.CODER.local_constraints"
  ]
}
```

Compiled compatibility output can still be JSON:

```json
{
  "id": "CODER",
  "role": "Implementation owner",
  "instructions": [
    {
      "module_id": "framework.agent.CODER.workflow",
      "content": "Follow the active task lifecycle before implementation."
    },
    {
      "module_id": "recipe.tdd.coder.loop",
      "content": "For behavior changes, write or update a failing test before implementation."
    }
  ]
}
```

This lets the internal model be uniform without forcing every external prompt consumer to adopt the
same file format.

## Storage hierarchy

Recipes are downloaded before they affect any project.

Global cache:

- `~/.agentplane/recipes-index.json`
- `~/.agentplane/recipes-installed.json`
- `~/.agentplane/recipes-store/<recipe-id>/<version>/...`

The global cache is the package base and download authority. It is not the active behavior
authority for a project.

Project-local authority:

- `.agentplane/recipes/registry.json`
- `.agentplane/recipes/packages/<recipe-id>/manifest.json`
- `.agentplane/recipes/packages/<recipe-id>/modules/**`
- `.agentplane/recipes/packages/<recipe-id>/agents/**`
- `.agentplane/recipes/packages/<recipe-id>/policy/**`
- `.agentplane/recipes/packages/<recipe-id>/templates/**`

Generated artifacts:

- `.agentplane/generated/prompt-graph.json`
- `.agentplane/generated/prompt-manifest.json`
- `.agentplane/generated/overlay-bundle.json`
- `.agentplane/generated/recipe-assets.json`

Generated artifacts are derived. They explain the active graph but do not carry independent
authority.

## Compiler pipeline

The v0.4 compiler should run this sequence:

1. Read framework base modules.
2. Read repo config and workflow mode.
3. Read globally cached recipe package metadata when selecting or installing recipes.
4. Read project-local recipe registry for active behavior.
5. Load active recipe modules from `.agentplane/recipes/packages/<id>/`.
6. Validate mutation targets, mutability, version compatibility, and conflicts.
7. Resolve bindings by role, phase, command, task tags, repo type, and workflow mode.
8. Compile `AGENTS.md`, policy files, agent profiles, templates, validators, and runner prompt blocks.
9. Write generated graph and provenance manifests.
10. Run policy routing and doctor checks.

## Conflict resolution

Conflicts must be explicit and explainable.

Resolution inputs:

- module `priority`;
- owner layer: `framework`, `recipe`, `repo`;
- mutability of the target module;
- binding conditions;
- recipe `requires` and `conflicts`;
- compatibility version ranges.

Default layer order:

1. sealed framework modules;
2. repo-local explicit overrides;
3. recipe required modules;
4. recipe default modules;
5. recipe advisory modules;
6. generated compatibility text.

If two required modules patch the same exclusive slot, compilation fails closed. If two modules append
to an ordered list, deterministic ordering uses priority, recipe id, then module id.

## Examples

### Naming recipe

Purpose: enforce branch, task, and PR naming conventions.

Modules:

- `recipe.naming.policy.branch_names` patches `policy.workflow.branch_pr.branch_naming`.
- `recipe.naming.validator.branch_name` validates branch names before PR creation.
- `recipe.naming.orchestrator.slug_planning` tells `ORCHESTRATOR` to plan slugs before task start.

Effect:

- agents do not create arbitrary branch names;
- PR opening fails when the branch name does not match the policy;
- `AGENTS.md` only adds or references the load rule.

### TDD recipe

Purpose: require red-green-refactor evidence for code changes.

Modules:

- `recipe.tdd.coder.loop` patches `agent.CODER.implementation_rules`.
- `recipe.tdd.policy.evidence` patches `policy.dod.code.verification_evidence`.
- `recipe.tdd.validator.test_evidence` requires verification evidence for code tags.

Effect:

- `CODER` changes implementation order;
- DoD requires test evidence;
- verification records whether TDD applied or why it did not.

### Roadmap decomposition recipe

Purpose: make multi-step work start from a roadmap before executable tasks.

Modules:

- `recipe.roadmap.orchestrator.discovery` patches `agent.ORCHESTRATOR.planning_behavior`.
- `recipe.roadmap.planner.decomposition` replaces `agent.PLANNER.decomposition_strategy`.
- `recipe.roadmap.template.roadmap_doc` adds a roadmap artifact template.

Effect:

- broad requests produce milestones, dependencies, risks, and verification gates first;
- `PLANNER` converts approved milestones into executable tasks;
- implementation remains task-scoped after roadmap approval.

### PRD-first recipe

Purpose: require a product requirements document before feature implementation.

Modules:

- `recipe.prd.orchestrator.product_intake` patches product planning behavior.
- `recipe.prd.template.prd` adds `docs/prd/<slug>.md`.
- `recipe.prd.validator.before_code` requires a PRD artifact before code tasks start.

Effect:

- product work creates acceptance criteria before implementation;
- `REVIEWER` checks diff behavior against PRD criteria;
- product tasks are not treated as generic coding tasks.

### ADR recipe

Purpose: require ADRs for architecture-changing work.

Modules:

- `recipe.adr.policy.required_for_architecture` patches docs or code DoD.
- `recipe.adr.template` adds a standard ADR structure.
- `recipe.adr.validator.architecture_artifact` checks for an ADR path when tags include
  `architecture`.

Effect:

- architecture decisions become explicit artifacts;
- task closure must reference the ADR or explain why it was not required.

## Manifest outputs

`prompt-manifest.json` explains the compiled files:

```json
{
  "schema_version": 1,
  "kind": "prompt_manifest",
  "entrypoint": "AGENTS.md",
  "modules": [
    {
      "id": "recipe.tdd.coder.loop",
      "kind": "role_instruction",
      "surface": "agent",
      "target": "agent:CODER",
      "slot": "agent.CODER.implementation_rules",
      "owner": "recipe:tdd@1.0.0",
      "source": ".agentplane/recipes/packages/tdd/prompts/coder-loop.md",
      "compiled_to": [".agentplane/agents/CODER.json"],
      "content_sha256": "<sha256>"
    }
  ]
}
```

The manifest supports:

- `agentplane prompts explain`;
- upgrade drift detection;
- recipe removal and rebuild;
- policy review;
- debugging why a behavior was active.

## Breaking change notes for v0.4

The planned v0.4 model changes the meaning of managed prompt files:

- `AGENTS.md`, `.agentplane/policy/**`, and `.agentplane/agents/**` become compiled outputs.
- recipes can affect the compiled gateway, policy, and agent profile surfaces through module
  mutations.
- recipe activation must rebuild prompt artifacts.
- upgrade must re-resolve base modules before reapplying recipe mutations and repo overrides.
- manual edits to compiled files are drift unless recorded as repo-local modules or overrides.

Migration must preserve the current entrypoint behavior:

1. existing projects keep a readable `AGENTS.md`;
2. framework-owned policy modules are imported into the base graph;
3. existing recipe overlays are converted into module records where possible;
4. unresolved manual edits are reported as drift instead of being silently discarded.

## Critical constraints

- `AGENTS.md` remains the entrypoint.
- Additional files are loaded only when the gateway, role, phase, or binding conditions require them.
- Recipes mutate module graphs, not raw files.
- Sealed framework safety modules cannot be overridden by recipes.
- Every compiled behavior must have provenance.
- Generated artifacts are inspectable but not authoritative.
