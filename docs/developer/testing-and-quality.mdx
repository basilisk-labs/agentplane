## Local checks

From repo root:

```bash
bun install
bun run format:check
bun run schemas:check
bun run agents:check
bun run lint
bun run typecheck
bun run test:fast
```

Inside this framework checkout, `agentplane` from `PATH` now delegates automatically to the
repo-local binary. The repo-local entrypoint still executes from `dist/`, so rebuild after source
changes:

```bash
bun run --filter=@agentplaneorg/core build
bun run --filter=agentplane build
```

To force the global installed CLI instead of the repo-local handoff during framework debugging, set
`AGENTPLANE_USE_GLOBAL_IN_FRAMEWORK=1`.

When the active runtime is unclear, inspect it explicitly:

```bash
agentplane runtime explain
```

Inside the framework checkout, this output is now the canonical framework-dev workflow surface:
it prints the active runtime facts together with the rebuild commands, the global reinstall helper,
the runtime re-check step, and the explicit force-global override path.

If watched runtime paths are dirty and the repo-local build would otherwise be considered stale,
read-only inspection commands still warn and continue:

```bash
agentplane doctor
agentplane runtime explain
agentplane task list
agentplane task show <task-id>
agentplane task verify-show <task-id>
agentplane config show
agentplane quickstart
agentplane role CODER
```

After you rebuild the local packages, strict commands should be unblocked again as soon as the
recorded watched-runtime snapshot in `dist/.build-manifest.json` matches the current source tree,
even before those runtime file changes are committed.

Mutating commands remain strict and still require a fresh repo-local build.

If you want the globally installed `agentplane` command itself to come from this checkout, use the
developer reinstall helper:

```bash
scripts/reinstall-global-agentplane.sh
```

It rebuilds local `@agentplaneorg/core` and `agentplane`, reinstalls both packages globally from
this checkout, and verifies that the global `agentplane` runtime resolves `@agentplaneorg/core`
from this checkout instead of a published fallback.

For a full run (including slower CLI integration tests):

```bash
bun run test:full
```

To run only the CLI integration tests (without the slower smoke test):

```bash
bun run test:cli
```

To run the critical pre-push suite (fast, real git, separate-process CLI e2e checks):

```bash
bun run test:critical
```

To run the platform-critical cross-runner suite that backs Windows CI and prepublish:

```bash
bun run test:platform-critical
```

The critical suites are now split into canonical named sub-suites so workflow YAML, local CI,
and manual debugging can point at the same contracts:

```bash
bun run test:critical:exit-codes
bun run test:critical:git-edge
bun run test:critical:protected-paths
bun run test:critical:scope-leak
bun run test:critical:symlink-root

bun run test:platform-critical:pr-meta
bun run test:platform-critical:scenario
bun run test:platform-critical:init-upgrade-backend

bun run test:backend-critical
```

To run the canonical workflow/harness and significant-file coverage suites that GitHub workflows
now call via shared scripts instead of inline `vitest` command lines:

```bash
bun run test:workflow-coverage
bun run test:significant-coverage
```

To check the GitHub workflow command contract directly:

```bash
bun run workflows:command-check
```

To run the faster CLI unit-ish suites (core + recipes):

```bash
bun run test:cli:unit
```

To run specific CLI suites:

```bash
bun run test:cli:core
bun run test:cli:recipes
bun run test:cli:scenario
```

To run the CLI smoke test (slowest end-to-end path):

```bash
bun run test:cli:smoke
```

To run the slow CLI suites together (scenario + smoke):

```bash
bun run test:cli:slow
```

To run package-scoped unit tests:

```bash
bun run test:agentplane
bun run test:core
bun run test:recipes
bun run test:testkit
```

## CI scoping (internal)

The repo includes a helper script that runs checks on a single scope:

```bash
node scripts/ci-scope.mjs --scope agentplane
```

Scopes: `agentplane`, `core`, `recipes`, `spec`, `testkit`.

This is intended for CI/local optimization; it is not part of the public CLI.

## Git hooks

Git hooks are a required quality gate for this repo. Do not bypass them.

```bash
bun run hooks:install
```

By default, hooks run:

- `commit-msg`: commit subject enforcement:
  - Task commits: `<emoji> <TASK_SUFFIX> <scope>: <summary>`
  - Non-task commits: `<emoji> <scope>: <summary>` (optional explicit `DEV` suffix allowed)
  - Status/comment-driven task commits derive `<emoji>` from the executor agent id and enforce it via `AGENTPLANE_AGENT_ID`
  - Finish commits must use `✅` (enforced via `AGENTPLANE_STATUS_TO=DONE`)
- `pre-commit`: staged-only format/lint checks + `test:precommit`
- `pre-push`: runs `bun run ci:local` by default. This fast local gate always covers format, schemas, agent templates, policy routing, release parity, build, onboarding, lint, fast tests, and critical CLI checks, but it now chooses a narrower contour when the changed-file scope is isolated:
  - `docs-only`: docs, website, policy, and task-doc changes skip code-heavy checks
  - `targeted(task)`: task command paths run the task-focused fast suite
  - `targeted(doctor)`: doctor command paths run the doctor-focused fast suite
  - `targeted(backend)`: backend/projection paths run the backend-critical fast suite
  - `targeted(hooks)`: hook and CI-routing paths run the hook-focused fast suite
  - `targeted(workflow)`: workflow YAML, actionlint config, and workflow command-contract paths run
    workflow lint + contract enforcement without the broad fallback
  - `targeted(cli-help)`: safe CLI help/spec/guidance paths run the help-focused fast suite
  - `targeted(cli-core)`: isolated `run-cli` execution-plumbing paths run the focused core CLI suite
  - `targeted(cli-runtime)`: isolated bin/runtime freshness, handoff, and stale-dist paths run the focused runtime CLI suite
  - `targeted(release)`: release command/script paths run the release-focused fast suite
  - `targeted(upgrade)`: upgrade command paths run the upgrade-focused fast suite
  - `targeted(guard)`: guard command paths run the guard-focused fast suite
  - `full-fast fallback`: mixed, unclassified, shared command, package metadata, and residual runtime-sensitive CLI paths without a clear focused suite still run the broad fast suite on purpose

  Runtime-sensitive CLI paths are intentionally split along two safe boundaries:
  - `cli-core`: `run-cli` execution plumbing and its focused core tests
  - `cli-runtime`: repo-local handoff, runtime-watch, stale-dist, and global-install verification paths

  The following still stay on the broad fallback by design:
  - docs-generation stability paths such as `run-cli.core.docs-cli.test.ts`
  - init/upgrade integration paths such as `run-cli.core.init-upgrade-backend.test.ts`
  - any mixed CLI change that crosses help, runtime, hooks, guard, or broader infra-sensitive boundaries

  Set `AGENTPLANE_HOOKS_FULL=1` to force `bun run ci:local:full`, which adds docs-site, workflows lint, platform-critical Windows tests, and the significant-file coverage guard. For release-tag pushes (or when `AGENTPLANE_HOOKS_RELEASE=1`/`AGENTPLANE_HOOKS_FULL=1`), it additionally runs release notes validation and `bun run release:prepublish`.

`commit-msg` stale-dist guard is path-aware: it is enforced for staged `packages/agentplane/src/**`, `packages/core/src/**`, and `packages/agentplane/bin/agentplane.js` changes; docs-only/config-only commits skip this guard.

If a hook fails, fix the issue instead of skipping the hook.

## CI

On PRs and on pushes to `main`, GitHub runs:

- `Core CI`:
- Linux: format/schemas/agents/release parity/build/CLI docs check/lint core/`test:fast`/`test:critical`/significant coverage guard
- Linux workflow guards now call canonical named scripts instead of embedding inline `vitest` suites:
  - `bun run test:workflow-coverage`
  - `bun run test:significant-coverage`
- Linux also enforces workflow-contract gates:
  - `agentplane workflow build --validate --dry-run`
  - `agentplane doctor`
  - dedicated workflow+harness branch-coverage thresholds (`check-workflow-harness-coverage.mjs`)
- Windows: format/schemas/agents/release parity/build + `bun run test:platform-critical`
- `Docs CI`: docs generation + site typecheck/build + design-language checks
- `Workflows Lint`: `actionlint` plus the workflow command-contract check, which rejects inline
  `bun test`, `bunx vitest run`, and bare `vitest run` suites in workflow YAML once a canonical
  repository script exists

## Coverage and parity tests

- CLI tests live under `packages/agentplane/src/cli/` and `packages/agentplane/src/commands/`.
- Significant-file coverage policy is checked by `bun run coverage:significant`.
- The canonical workflow wrappers for these guards are:
  - `bun run test:workflow-coverage`
  - `bun run test:significant-coverage`
- Workflow/harness coverage policy is checked by:
  - `node scripts/check-workflow-harness-coverage.mjs`
- Release-time local parity checks are enforced by `bun run release:prepublish`.
