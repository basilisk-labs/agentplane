## Local Turborepo graph

Turborepo is used only as a framework-checkout developer overlay. It is not part of the installed
AgentPlane runtime, generated user-project scaffold, published CLI package surface, or task backend
contract.

Use the Turbo scripts when you need package-level graph execution, cache evidence, or a fast local
affected check before choosing a broader validation route:

```bash
bun run dev:turbo:ls
bun run dev:turbo:graph
bun run dev:turbo:affected
bun run ci:local:turbo
```

`dev:turbo:ls` prints the workspace packages that Turbo can see. `dev:turbo:graph` writes a Mermaid
task graph to `.agentplane/cache/turbo-task-graph.mermaid`. `dev:turbo:affected` runs
`build`, `typecheck`, and `test` for affected `packages/*` tasks. `ci:local:turbo` runs the same
package task graph without `--affected`. Both wrapper commands record the latest local evidence in
`.agentplane/cache/turbo-local-ci-report.json`.

The local CI report records attempted, successful, failed, and cached task counts, package names,
task ids, cache status, log files, and dependency edges from the Turbo summary. Treat it as
execution evidence for local development. It is not a semantic code map and does not replace
AgentPlane task scope, policy routing, context wiki, architecture docs, or human review of domain
impact.

Default Turbo package checks intentionally target `packages/*`. The docs website has separate build
side effects and should be checked explicitly when docs or site files change:

```bash
bun run dev:turbo:docs
```

Inside this framework checkout, `agentplane` from `PATH` now delegates automatically to the


Inside the framework checkout, this output is now the canonical framework-dev workflow surface:

To inspect unused files, exports, dependencies, and unresolved imports:

```bash
bun run knip:check
```

`knip:check` enforces the current unused-code baseline and fails when unused files, value exports,
type exports, enum members, namespace members, or total findings grow. New cleanup work should
reduce the baseline; only update the guard thresholds after reviewed debt changes. Use
`bun run knip:report` for the full local report.

Current enforced baseline:

- 5 unused files.
- 235 unused value exports.
- 298 unused exported types.
- 0 unused enum members.
- 0 unused namespace members.
- 538 total guarded unused-code findings.

The baseline has no unresolved imports and no unused dependencies; those categories should stay
empty.

To inspect structural duplication before or after refactoring:

```bash
bun run clone:report
bun run clone:check
```

`clone:report` runs `jscpd` over runtime package sources and repository scripts, excluding tests,
snapshots, generated files, build output, dependency folders, and baseline artifacts. Treat the
reported clone clusters as refactor candidates, not automatic refactor instructions. Fix a cluster
when the duplicated blocks share the same responsibility and are likely to drift together; leave it
alone when the similarity is only syntactic or a shared abstraction would hide different domain
rules.

`clone:check` compares the current report with `scripts/baselines/clone-baseline.json` and fails
when guarded clone counts, duplicated lines, or duplicated tokens increase. During a refactor, use
this sequence:

```bash
bun run clone:report
# refactor one or two confirmed clone clusters
bun run clone:check
bun run typecheck
bun run test:fast
```

When a reviewed refactor intentionally reduces clone debt, refresh the baseline in the same change:

```bash
bun run clone:baseline:update
```

Record the before/after clone metrics in the PR or task verification note, for example
`clones 88 -> 82, duplicatedLines 1581 -> 1430`. Do not update the baseline merely to accept a
regression; in that case either reduce the new duplication or document why the new duplicate shape
is intentional.

## Test file naming

Vitest test files use the `.test.ts` suffix. `vitest.config.ts` includes
`packages/**/src/**/*.test.ts`, and new tests should follow that convention.

Do not use `.spec.ts` for Vitest tests. In this repository, `.spec.ts` is reserved for
command specification modules such as `packages/agentplane/src/commands/*.spec.ts`; those files
define command parsing/help contracts and are not test suites.

## Git hooks

Git hooks are a required quality gate for this repo. Do not bypass them.

```bash
bun run hooks:install
```

By default, hooks run:

- `commit-msg`: commit subject enforcement:
  - Task commits: `<emoji> <TASK_SUFFIX> <scope>: <summary>`
  - Non-task commits: `<emoji> <scope>: <summary>` (optional explicit `DEV` suffix allowed)
  - Status/comment-driven task commits derive `<emoji>` from the executor agent id and enforce it via `AGENTPLANE_AGENT_ID`
  - Finish commits must use `✅` (enforced via `AGENTPLANE_STATUS_TO=DONE`)
- `pre-commit`: staged-only format/lint checks + `test:precommit`
- `pre-push`: runs `bun run ci:local` by default. This fast local gate is intentionally read-only: it uses `format:check`, fails with an explicit remediation command instead of rewriting files, and blocks if any hook step mutates tracked content. It always covers format, schemas, agent templates, policy routing, release parity, build, onboarding, lint, architecture boundaries, fast tests, and critical CLI checks, but it now chooses a narrower contour when the changed-file scope is isolated:
  - `docs-only`: docs, website, policy, and task-doc changes skip code-heavy checks
  - `targeted(task)`: task command paths run the task-focused fast suite
  - `targeted(doctor)`: doctor command paths run the doctor-focused fast suite
  - `targeted(backend)`: backend/projection paths run the backend-critical fast suite
  - `targeted(hooks)`: hook and CI-routing paths run the hook-focused fast suite
  - `targeted(workflow)`: workflow YAML, actionlint config, and workflow command-contract paths run
    workflow lint + contract enforcement without the broad fallback
  - `targeted(cli-help)`: safe CLI help/spec/guidance paths run the help-focused fast suite
  - `targeted(cli-core)`: isolated `run-cli` execution-plumbing paths run the focused core CLI suite
  - `targeted(cli-runtime)`: isolated bin/runtime freshness, handoff, and stale-dist paths run the focused runtime CLI suite
  - `targeted(release)`: release command/script paths run the release-focused fast suite
  - `targeted(upgrade)`: upgrade command paths run the upgrade-focused fast suite
  - `targeted(guard)`: guard command paths run the guard-focused fast suite
  - `full-fast fallback`: mixed, unclassified, shared command, package metadata, and residual runtime-sensitive CLI paths without a clear focused suite still run the broad fast suite on purpose

  Runtime-sensitive CLI paths are intentionally split along two safe boundaries:
  - `cli-core`: `run-cli` execution plumbing and its focused core tests
  - `cli-runtime`: repo-local handoff, runtime-watch, stale-dist, and global-install verification paths

  The following still stay on the broad fallback by design:
  - bootstrap/init integration paths such as `run-cli.core.init.test.ts`
  - any mixed CLI change that crosses help, runtime, hooks, guard, or broader infra-sensitive boundaries

  The live Codex runner smoke harness stays outside this mandatory local path. Run it manually with
  `bun run runner:codex:smoke -- --live` when you need an on-demand end-to-end probe against the
  current repo-local binary; neither `ci:local:fast` nor the default `pre-push` hook should invoke it.

  Set `AGENTPLANE_HOOKS_FULL=1` to force `bun run ci:local:full`, which adds docs-site, workflows lint, platform-critical Windows tests, and the significant-file coverage guard. For release-tag pushes (or when `AGENTPLANE_HOOKS_RELEASE=1`/`AGENTPLANE_HOOKS_FULL=1`), it additionally runs release notes validation and `bun run release:prepublish`.

`commit-msg` stale-dist guard is path-aware: it is enforced for staged `packages/agentplane/src/**`, `packages/core/src/**`, and `packages/agentplane/bin/agentplane.js` changes; docs-only/config-only commits skip this guard.

If a hook fails, fix the issue instead of skipping the hook.

## CI

On PRs and on pushes to `main`, GitHub runs:

- `Core CI`:
- Linux: format/schemas/agents/release parity/build/CLI docs check/lint core/architecture boundaries/`test:fast`/`test:critical`/significant coverage guard
- Linux workflow guards now call canonical named scripts instead of embedding inline `vitest` suites:
  - `bun run coverage:workflow-suite`
  - `bun run coverage:significant-suite`
- Linux also enforces workflow-contract gates:
  - `agentplane workflow build --validate --dry-run`
  - `agentplane doctor`
  - dedicated workflow+harness branch-coverage thresholds (`check-workflow-harness-coverage.mjs`)
- Windows: format/schemas/agents/release parity/build + `bun run test:platform-critical`
- `Docs CI`: docs generation + site typecheck/build + design-language checks
- `Workflows Lint`: `actionlint` plus the workflow command-contract check, which rejects inline
  `bun test`, `bunx vitest run`, and bare `vitest run` suites in workflow YAML once a canonical
  repository script exists

## Coverage and parity tests

- CLI tests live under `packages/agentplane/src/cli/` and `packages/agentplane/src/commands/`.
- Significant-file coverage policy is checked by `bun run coverage:significant`.
- The canonical workflow wrappers for these guards are:
  - `bun run coverage:workflow-suite`
  - `bun run coverage:significant-suite`
- Workflow/harness coverage policy is checked by:
  - `node scripts/check-workflow-harness-coverage.mjs`
- Release-time local parity checks are enforced by `bun run release:prepublish`.
