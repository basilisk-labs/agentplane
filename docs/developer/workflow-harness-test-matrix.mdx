## Suite gate

CI enforces an explicit workflow-harness suite contract for these modules:

- `packages/agentplane/src/workflow-runtime/validate.ts`
- `packages/agentplane/src/workflow-runtime/file-ops.ts`
- `packages/agentplane/src/harness/state-machine.ts`
- `packages/agentplane/src/harness/retry-policy.ts`
- `packages/agentplane/src/harness/scheduler.ts`

The gate runs the dedicated harness/runtime tests and then checks that the curated source-to-test
mapping still exists in this matrix:

```bash
bunx vitest run \
  packages/agentplane/src/workflow-runtime/*.test.ts \
  packages/agentplane/src/harness/*.test.ts
node scripts/check-workflow-harness-coverage.mjs
```

The repository previously used per-file branch thresholds here, but the current Vitest v8 JSON
reporter no longer emits a usable file-level map under the pinned toolchain. The suite contract is
now the stable release/CI gate until a reliable per-file coverage backend is reintroduced.

## Flaky-test policy

- Use deterministic fixtures only (`mkdtemp`, isolated git repos, no wall-clock assertions).
- Avoid network and outside-repo dependencies in tests.
- For env-based kill-switch tests, always restore env in `finally`.
- Do not assert on full human-formatted messages when a stable code exists; assert on code/path primitives.

## Harness engineering checks

- Favor tests that validate system behavior under scale and repetition, not only one-shot correctness.
- Ensure every failure mode emits enough structured context for fast operator triage.
- Keep recovery paths (`restore`, retry, fallback) tested at the same rigor as happy-path execution.
