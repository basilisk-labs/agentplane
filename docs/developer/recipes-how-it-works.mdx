title: "How recipes work"
description: "Cache, vendor, and inspect recipes in the current CLI runtime."

This page describes runtime behavior behind the public `recipes` surface and the internal
recipe-owned scenario assets it still compiles.

## Cache flow

1. Resolve source (`--name`, `--path`, or `--url`).
2. Download when source is remote.
3. Extract archive and locate `manifest.json`.
4. Validate manifest, recipe-local file references, and scenario definitions.
5. Copy the recipe archive contents into the global cache under `~/.agentplane/recipes-store/<recipe-id>/<version>/`.
6. Update the global cache inventory (`recipes-installed.json`) and preserve the global remote index cache.

`recipes install` stops here. It does not make the recipe active for any project.

## Vendoring flow

`recipes add`:

1. Resolves `<id|id@version>` from the global cache.
2. Materializes the cached package into `.agentplane/recipes/packages/<recipe-id>/` using `copy` by default or `link` when requested.
3. Writes or updates `.agentplane/recipes/registry.json` as the sole project authority for vendored recipes.
4. Recompiles `.agentplane/generated/overlay-bundle.json`,
   `.agentplane/generated/recipe-assets.json`, and `.agentplane/generated/prompt-graph.json`.
5. Leaves runtime pointed at the vendored project-local package, not at the global cache.

## Listing and inspection

- `recipes list` scans the global cache and prints cached entries.
- `recipes info` reads cached recipe metadata.
- `recipes active` and `recipes explain` read project-local vendored state from `registry.json`.
- Recipe-owned scenario assets still live in vendored packages, but they are no longer exposed as a
  first-class public CLI namespace.

## Prompt module assets

Project overlay recipes can contribute modular prompt assets:

- `prompt_modules` add recipe-owned prompt modules to the project prompt graph.
- `prompt_mutation_sets` patch, replace, disable, bind, or validate prompt modules after framework
  and active recipe modules are loaded.

Framework prompt assets may expose named fragments inside gateway, policy, runner, and agent profile
files. Those fragments compile into ordinary prompt modules with stable provenance. Recipe mutation
sets should target those fragment ids when they need to change one behavior, for example a single
CODER workflow instruction, instead of replacing an entire profile or markdown file.

During recipe artifact refresh, active recipe prompt modules and mutation sets are compiled with the
framework prompt module registry. The derived prompt graph is published to
`.agentplane/generated/prompt-graph.json` in the same transaction as the overlay bundle, recipe asset
registry, and project recipe registry. If prompt graph compilation fails, the previously published
generated artifacts remain unchanged.

## Scenario asset preparation

Internal scenario preparation still:

1. Parses `<recipe:scenario>` id.
2. Resolves the scenario through the deterministic recipe resolver.
3. Validates compatibility, recipe-local files, and the internal `scenario.json` contract, including `task_template`.
4. Prints a prepared run plan with selection reasons, normalized `run_profile`, and task-template data.
5. Stops without executing tools or creating `runs/` artifacts.

## Scenario asset materialization

Internal scenario materialization still:

1. Parses `<recipe:scenario>` id.
2. Resolves and validates the scenario through the same deterministic resolver used during internal
   scenario preparation.
3. Materializes a task from the scenario's explicit `task_template` and records recipe provenance on that task.
4. Assembles the shared runner bundle from base prompts, repository context, task context, and recipe context.
5. Creates task-local run artifacts under `.agentplane/tasks/<task-id>/runs/<run-id>/`.
6. Invokes the configured runner adapter and reports task id, run id, artifact paths, and runner status.

`task run` uses the same shared runner contract for ordinary tasks that were not created from recipes.

## Environment behavior

Install and validation flows inherit the current process environment.
Execution flows are delegated to the configured runner adapter. The CLI passes recipe and task context through the runner bundle and does not interpret recipe `steps` as a direct execution DSL.
Recipe `run_profile` no longer declares a separate network policy. Real network approval outside the
runner bundle continues to flow through the global `require_network` gate.
Recipe `run_profile.mode` stays in resolver selection and scenario discovery output, but the main
runner no longer exports it through adapter env or capability display. It is treated as a scenario
classification label, not as an execution-policy field.
Recipe `run_profile` also no longer exposes `requires_human_approval` in the main-repo runtime
contract. The field was removed after a live Codex probe showed that non-interactive approval modes
do not provide a trustworthy barrier.
Recipe `run_profile.expected_exit_contract` is treated the same way: the main-repo runtime no longer
accepts or projects it because current values are scenario-specific labels, not a deterministic
shared contract. The bundled catalog has already been cleaned to match that narrower runtime
surface.
`run_profile.writes_artifacts_to` is now a partial enforcement contract: the adapter still receives
the declared prefixes through env, invalid prefixes are rejected before spawn, and after execution
the shared runner rejects external `result.json` manifest paths outside those prefixes after
normalized relative-path checks. The check only applies to recipe-controlled manifest paths such as
`artifacts[].path` and `evidence.evidence_paths[]`; traversal forms and absolute paths are rejected,
while runner-owned control artifacts and preserved source manifests are exempt.

## Cache and runtime behavior

- Remote index cache and recipe archives are stored globally under `AGENTPLANE_HOME` (default `~/.agentplane`).
- Vendored project packages live under `.agentplane/recipes/packages/` and are tracked by `.agentplane/recipes/registry.json`.
- Generated recipe runtime artifacts live under `.agentplane/generated/`; this includes
  `overlay-bundle.json`, `recipe-assets.json`, and `prompt-graph.json`.
- After vendoring, runtime reads only project-local recipes; deleting the global cache does not break execution for already vendored recipes.
- `recipes list-remote --refresh` forces index refresh.
- `recipes cache prune` removes stale cached entries and supports `--dry-run`.

## Failure behavior

Common hard failures:

- missing or invalid manifest
- unsupported archive structure
- cache miss during `recipes add`, `recipes update`, or `recipes detach`
- unknown scenario id
- missing recipe-local files referenced by manifest
- incompatible runtime context for the selected scenario
- network/download errors for remote sources

On failure, CLI exits non-zero and reports an error with standard error code mapping.
