## Release policy

- Versioning follows SemVer.
- `agentplane`, `@agentplaneorg/core`, and `@agentplaneorg/recipes` release together at the same version.
- `packages/agentplane` dependencies `@agentplaneorg/core` and `@agentplaneorg/recipes` must match the same release version exactly.
- Patch bump is default automation path.
- Minor/major bump requires explicit approval.

## Notes policy

Every release requires notes in `docs/releases/vX.Y.Z.md`.

Requirements:

- English text
- at least one bullet per change from the selected release range
- no fixed maximum bullet count
- human-readable summary, not raw git log dump

## Standard flow

1. Generate release plan:

```bash
agentplane release plan
```

If the workspace version is already ahead of the latest published/tagged patch release, the planner
now fails instead of proposing a skipped future patch. Recover and publish the missing patch chain
first, then plan the next release.

2. Draft notes from plan artifacts.
3. Run local release gate (must match publish prerequisites):

```bash
bun run release:prepublish
```

Optional fast parity check before the full gate:

```bash
bun run release:parity
```

4. Prepare the release commit:

```bash
AGENTPLANE_RELEASE_DRY_RUN=1 agentplane release apply
```

In `branch_pr` mode, do not use `release apply` as the candidate route. Prepare the release
candidate on a dedicated non-base branch instead:

```bash
AGENTPLANE_RELEASE_DRY_RUN=1 agentplane release candidate
```

5. Real release:

```bash
agentplane release apply --push --yes
```

`release apply --push --yes` now fails in an immutable preflight stage if the working tree is dirty, the local or remote release tag already exists, the target npm version is already burned, the configured remote is missing, or package versions drifted away from the prepared plan baseline. Only after that preflight passes does it run the release gate, bump versions, refresh generated package-reference docs, and push with `git push --no-verify`, so it does not recursively re-enter the local `pre-push` hook after the release gate already passed.

In `branch_pr` repositories, the explicit candidate route is:

```bash
agentplane release candidate --push --yes
```

That command prepares and optionally pushes only the candidate branch. Final publication remains
gated on merge to the protected base branch and hosted publish from `main`.

## What `release:prepublish` covers

`bun run release:prepublish` includes:

- `release:ci-check`:
- format check
- schema sync check
- agent template sync check
- build
- lint
- release-focused Linux test base
- workflow/harness coverage guard
- significant-file coverage guard
- release-critical smoke/recovery checks
- `release:check`:
- package builds
- npm dry-run pack for both packages

This is intentionally narrower than every workflow in the repository, but it should match the
release-relevant Linux gate that can block or invalidate a publishable SHA.

## Publishing policy

npm publish is GitHub-only.
Local `npm publish` is blocked by `prepublishOnly` through `scripts/enforce-github-publish.mjs`.

GitHub workflow `Publish to npm`:

- auto-publishes only from successful `Core CI` `workflow_run` events on `main`
- `workflow_dispatch` recovery remains available for explicit reruns
- manual recovery should pass an exact `sha` input; `ref` is only a fallback when `sha` is omitted
- serializes publish runs by release identity (`workflow_run.head_sha` or explicit dispatch `sha`) instead of branch ref, and queues duplicates instead of canceling an in-flight release
- uses OIDC trusted publishing
- falls back to `NPM_TOKEN` when a package is not yet configured for npm trusted publishing
- sets provenance
- consumes the `release-ready` artifact from the exact `Core CI` run as the only readiness source
- emits a `publish-result` artifact from the publish job as the canonical post-publish outcome record
- validates the exact publish payload with `bun run release:check` inside the workflow itself after
  the matching `release-ready` artifact already proved CI readiness for that SHA
- validates notes and version/tag parity
- validates workspace parity (`core`, `recipes`, `agentplane`, and both CLI dependency pins)
- runs release-specific checks (`release:check`, release notes/version validation, origin tag existence)
- publishes `@agentplaneorg/core` first, then `@agentplaneorg/recipes`, then `agentplane`
- uses detect-and-skip logic for already-published versions to tolerate reruns and workflow-dispatch recovery
- resolves the same `release-ready` artifact contract for `workflow_dispatch` recovery instead of deciding release readiness independently

`Pre-publish Checks` workflow remains available as a manual matrix gate (`workflow_dispatch`) for explicit validation runs.

## Distribution contract

The release commit is the source of truth. Every downstream distribution channel must derive its
version, tag, source URL, checksum, and human-readable status from the same release distribution
manifest rather than recalculating those values independently.

Required release distribution artifacts:

- `release-distribution.json`: machine-readable summary of the release version, tag, source SHA,
  npm tarball URLs, npm package status, GitHub Release asset names, external channel status, and
  recovery hints.
- `SHA256SUMS`: checksums for generated GitHub Release assets and installer payloads.
- `install.sh`: POSIX installer entrypoint for users who want a shell install path.
- `install.ps1`: PowerShell installer entrypoint for Windows users. The script must install a
  versioned release payload and verify checksums before executing or linking installed files.
- `agentplane-upgrade.tar.gz`: upgrade bundle consumed by `agentplane upgrade`.

The install scripts are convenience entrypoints, not update sources of truth. Package managers and
scripts must install an exact release version and must not implement self-update behavior that
bypasses the hosted release pipeline.

Blocking release channels:

- npm packages: `@agentplaneorg/core`, `@agentplaneorg/recipes`, and `agentplane`
- GitHub Release tag/body/assets
- GitHub Release checksum manifest
- GHCR image for the exact version and commit
- AgentPlane-owned Homebrew tap update when tap credentials are configured
- AgentPlane-owned Scoop bucket update when bucket credentials are configured
- `setup-agentplane` release/update when action repository credentials are configured

Credentials-gated channels must be explicit in the manifest. If a credential is absent, the channel
must record `skipped_missing_credentials` with the expected secret name and a recovery command. It
must not fail an otherwise valid npm/GitHub release unless that channel was explicitly marked
required for the run.

External moderated registries such as `homebrew/core`, winget, Chocolatey, AUR, and nixpkgs are
follow-up publication targets. They should be opened as PR jobs or manual recovery tasks after the
release is already published, because their review queues are outside the AgentPlane release SLA.

## Distribution modules

Keep publication modules independent and evidence-backed:

- npm module: publishes packages, runs published-package smoke checks, and records package outcomes.
- GitHub assets module: creates release assets, checksum manifest, and upload evidence.
- GHCR module: builds the exact-version image, tags `vX.Y.Z`, commit SHA, and `latest`, then runs a
  container CLI smoke check.
- Homebrew tap module: renders `Formula/agentplane.rb` from `release-distribution.json`, verifies
  the formula in dry-run/check mode, then opens or updates a tap PR when credentials exist.
- Scoop bucket module: renders `bucket/agentplane.json` from `release-distribution.json`, verifies
  checksum and URL fields in check mode, then opens or updates a bucket PR when credentials exist.
- setup action module: updates the `setup-agentplane` release path so CI users can resolve an exact
  AgentPlane version from the manifest.

Each module must emit a compact JSON evidence file under `.agentplane/.release/publish/`. The final
`publish-result.json` should reference those module files instead of flattening every channel into
one shell step.

GitHub workflow `Core CI` also supports `workflow_dispatch` recovery:

- prefer `sha=<release commit>` to validate the exact historical release candidate
- use `ref` only when `sha` is omitted
- the dispatched run checks out that exact target and can mint the canonical `release-ready` artifact
  for the same commit, which `Publish to npm` then consumes

## Provenance requirements

All published packages set:

- `publishConfig.access = public`
- `publishConfig.provenance = true`

Trusted publishing should be configured in npm for repository workflow identity. If a package has not
been onboarded there yet, provide `NPM_TOKEN` in the `npm` GitHub Actions environment so the same
workflow can publish it without forking the release route.

## Failure points and fixes

- `release apply --push` blocked:
- run `bun run release:prepublish` and fix failures
- Tag/version mismatch:
- ensure `vX.Y.Z` matches package versions and run `bun run release:parity`
- Package already published:
- bump to next patch and regenerate notes
- Publish workflow fails after partial publish:
- rerun workflow, idempotent checks should skip already-published packages
- Manual publish/recovery needs an exact commit:
- prefer `workflow_dispatch` with `sha=<release commit>` so the workflow resolves a fixed `release-ready`
  artifact source instead of following a moving branch ref
- If the exact release SHA never produced `release-ready` historically:
- first dispatch `Core CI` for that same `sha`
- then dispatch `Publish to npm` for that same `sha`
- If only some packages are trusted publishers:
- keep OIDC enabled and supply `NPM_TOKEN` as fallback for the remaining packages until npm-side
  trusted publishing is enabled for them too

## Recovery tool

For read-only recovery diagnostics against the latest release plan, use:

```bash
bun run release:recover
```

This reports:

- local tag created but remote tag missing
- release-note drift for the planned tag
- version bump drift versus the prepared plan
- optional GitHub workflow state for the release SHA when you want to distinguish
  `missing release-ready artifact`, `publish still missing`, `publish succeeded`, and `sibling Core CI failed` on the same commit:

```bash
bun run release:recover -- --check-github
```

This uses the latest apply report commit hash when available and treats the `release-ready` artifact
from `Core CI` as the canonical readiness source. For local use, provide GitHub auth and repo context
via `GITHUB_TOKEN` plus `GITHUB_REPOSITORY` or `--github-repo`.

The publish workflow now also writes `publish-result.json` as a separate artifact so post-publish
diagnostics can distinguish a completed release from a partially failed publish run without relying
only on workflow status inference.

For distribution recovery, use the manifest first:

- if npm succeeded but a package-manager channel was skipped, rerun only that channel with the same
  `release-distribution.json`
- if GitHub Release assets are missing, regenerate and upload assets for the same tag before running
  package-manager modules
- if GHCR publish failed after npm succeeded, rerun the GHCR module against the exact release SHA
- if a tap or bucket PR already exists, update that PR rather than creating a duplicate
- if credentials were missing, add the secret and rerun the credentials-gated module; do not bump the
  release version

## Local release E2E harness

For a local end-to-end validation of the exact release SHA against the real GitHub `release-ready`
artifact, use:

```bash
GITHUB_TOKEN=... \
GITHUB_REPOSITORY=basilisk-labs/agentplane \
bun run release:e2e:local -- --sha "$(git rev-parse HEAD)"
```

Contract:

- run it from a checkout that is already at the exact release SHA
- keep `gh` installed locally; the harness uses `gh run download` to fetch the canonical artifact
- by default it runs `bun run release:prepublish`, generates a local `release-ready` manifest,
  resolves the matching successful `Core CI` run, downloads the `release-ready` artifact, and
  validates `ready/sha/version/tag` parity
- use `--skip-prepublish` only when you intentionally want to debug the GitHub artifact path
  without rerunning the local gate

- optional burned-version checks when you explicitly opt in:

```bash
bun run release:recover -- --check-registry
```

The tool is intentionally read-only: it suggests exact next actions, but it does not rewrite tags,
versions, or release notes for you.

## Post-release checks

- verify npm versions for all published packages
- verify GitHub release body matches `docs/releases/vX.Y.Z.md`
- verify upgrade bundle artifacts exist in release assets
- verify `release-distribution.json` matches the published tag and release SHA
- verify `SHA256SUMS` covers all install and upgrade assets
- verify GHCR image tags resolve and `agentplane --version` works in the container
- verify Homebrew tap, Scoop bucket, and setup-action outcomes are either published or explicitly
  recorded as `skipped_missing_credentials` with recovery instructions
