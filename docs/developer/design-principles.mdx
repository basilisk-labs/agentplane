## Goals

Agent Plane is designed to make agent-driven development predictable and auditable. The system favors explicitness over convenience and makes workflow state visible and recoverable.

## Principles

### 1) Single source of truth

- Task state lives in `.agentplane/tasks/`.
- Avoid manual edits; use the CLI for changes.

### 2) Explicit workflow transitions

- Start, block, verify, and finish are explicit operations.
- Status comments are structured and traceable.
- Status commits are allowed for major transitions and follow task primary-tag semantics.

### 3) Deterministic artifacts

- PR artifacts and task docs are structured and stable.
- Exports are reproducible from source data.

### 4) Minimal core, extensible edges

- Core provides task, workflow, and guardrails.
- Optional automation ships as recipes, not core behavior.

### 5) Offline-first by default

- No network use unless explicitly required.
- Approval requirements can escalate by execution profile, but capabilities do not change.

### 6) Auditability

- Every task has a doc, verify steps, and rollback plan.
- Commits are tied to task IDs and include changelog detail.
- Release flow is GitHub-only publish with provenance and repeatable CI gates.

### 7) Harness engeneering over prompt tuning

- Optimize the execution harness first: contracts, gates, retries, recovery, and observability.
- Treat prompt text as one layer of the system, not the system itself.
- Prefer changes that improve operator legibility (`what`, `why`, `next action`) across all commands.

### 8) Throughput-aware integration

- High agent throughput favors short-lived slices and frequent integration.
- Avoid large, delayed merges when smaller deterministic increments are possible.
- Keep verification close to each increment to reduce rollback cost.

### 9) Entropy management as a default operation

- Agent output can increase repository entropy quickly.
- Bake in routine cleanup: detect drift early, restore known-good state fast, and keep docs/contracts synchronized with behavior.

## What this means for contributors

- Prefer explicit configs and clear CLI output over implicit behavior.
- Keep new functionality behind recipes when it is optional.
- Keep docs aligned with command behavior in the same task where code changes are made.
