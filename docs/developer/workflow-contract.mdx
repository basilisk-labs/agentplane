## Scope

This contract defines the versioned format for `.agentplane/WORKFLOW.md` used by the workflow runtime.

- Zod source of truth: `packages/core/src/config/workflow-contract.ts`
- Generated public schema: `schemas/workflow.schema.json`
- Published schema copies: `packages/core/schemas/workflow.schema.json` and
  `packages/spec/schemas/workflow.schema.json`
- Canonical fixtures: `packages/spec/examples/workflow-v1.json` and
  `packages/spec/examples/workflow-v2.json`
- Prompt body is required and interpreted as a template.

This is the active contract used by `agentplane init`, `agentplane workflow build`, `agentplane workflow restore`, `agentplane doctor`, and `agentplane preflight`.

## Harness engineering intent

`.agentplane/WORKFLOW.md` is not documentation only; it is an executable control surface.

- **Legibility:** make state and decisions inspectable (`Prompt Template`, `Checks`, `Fallback`).
- **Enforcement:** push correctness into contract + validators, not informal prompt style.
- **Recovery:** preserve and restore known-good workflow state deterministically.
- **Throughput safety:** keep fast iteration possible without sacrificing gates and rollback.

## File shape

`.agentplane/WORKFLOW.md` is composed of:

1. YAML front matter validated against `schemas/workflow.schema.json`
2. Markdown body with required sections:
   - `## Prompt Template`
   - `## Checks`
   - `## Fallback`

If front matter exists but does not decode into a map/object, the file is invalid.

## Supported versions

The runtime accepts exactly two input shapes and normalizes both to v2 before use:

- v1 uses the flat `mode` field.
- v2 uses `workflow.mode` and is the canonical persisted form.
- Any other integer version, including a future version greater than 2, fails with
  `WF_UNSUPPORTED_VERSION`. Writers must not silently downgrade or overwrite it.

Both versions require:

- `owners.orchestrator`: non-empty string
- `approvals.require_plan`: boolean
- `approvals.require_verify`: boolean
- `approvals.require_network`: boolean
- `retry_policy.normal_exit_continuation`: boolean
- `retry_policy.abnormal_backoff`: currently only `exponential`
- `retry_policy.max_attempts`: integer in `[1, 100]`
- `timeouts.stall_seconds`: integer in `[1, 86400]`
- `in_scope_paths`: non-empty array of non-empty strings

The version-specific mode key is required:

- v1: `mode: direct | branch_pr`
- v2: `workflow.mode: direct | branch_pr`

Unknown root keys are rejected. Known optional sections such as `workspace`, `tasks`, `framework`,
`scheduler`, `evaluator`, and `observability` are accepted in both supported versions and retain
typed extension fields without data loss. The `workflow` section remains v2-only; v1 uses the flat
`mode` discriminant.

`observability.runs_dir` is an informational, reserved description of the enforced supervisor
layout. It is not an executable storage override: production run artifacts always resolve through
the repository Git common directory.

## Failure behavior

Invalid `.agentplane/WORKFLOW.md` must fail validation with explicit diagnostics:

- missing required key
- invalid type
- invalid enum/range
- unknown key (when `additionalProperties: false`)
- missing required body section
- malformed/empty prompt template (runtime strict template checks are enforced in later tasks)

Existing non-migration writers and the restore helper do not define the versioned migration
contract:

- `publishWorkflowCandidate` rejects candidates with any `ERROR` diagnostics, atomically replaces
  the active workflow (`tmp + rename`), then copies it to `last-known-good`;
- `writeWorkflowConfigRaw` atomically replaces the active workflow first and `last-known-good`
  second;
- neither two-file sequence is a multi-file atomic transaction or receipt-backed operation;
- `restoreWorkflowFromLastKnownGood` validates `last-known-good`, then atomically replaces only the
  active workflow.

Migration and receipt rollback instead use the `receipt -> last-known-good -> active` commit-point
ordering described below.

## Reason code contract

Mutation and preflight failures should expose stable `reason_code` values. CLI guidance uses these codes to output deterministic remediation hints.

Current reconcile and guidance-critical codes:

- `usage_help`
- `sync_backend_mismatch`
- `reconcile_git_state_unreadable`
- `reconcile_task_scan_failed`
- `reconcile_task_scan_incomplete`
- `git_branch_state`
- `git_index_state`
- `git_context`
- `network_gate`
- `backend_sync_config`
- `backend_config`
- `validation_preflight`

The canonical code map lives in `packages/agentplane/src/cli/reason-codes.ts`.

## Compatibility policy

Versioning policy:

- `version` is the contract version for front matter schema.
- Backward-compatible additions require either:
  - optional fields, or
  - new schema version with migration path.
- Breaking changes require a major contract bump and migration tooling.

Compatibility guarantees:

- v1 remains readable during the compatibility window and has the same normalized meaning as its
  v2 migration result.
- Supported writers persist canonical v2 and preserve v2-only and extension fields.
- Unsupported versions are rejected before config, build, doctor/fix, restore, or upgrade writers
  can overwrite them.
- The v1-to-v2 migration is deterministic, idempotent, and has an exact-byte rollback receipt.

Generate or verify schema copies and fixtures with:

```bash
bun run schemas:sync
bun run schemas:check
```

## Migration contract

Preview the canonical v2 candidate without writing files:

```bash
agentplane workflow migrate --dry-run
```

Apply the migration with receipt-backed recovery. The receipt and last-known-good target are
persisted first; each file uses atomic replacement, and the active workflow is replaced last as the
commit point. This is not a multi-file atomic transaction:

```bash
agentplane workflow migrate
```

Re-running the command on v2 is a no-op. Roll back only while the active workflow still matches the
receipt target hash:

```bash
agentplane workflow migrate \
  --rollback .agentplane/workflows/migrations/workflow-v1-to-v2-<digest>.json
```

Rollback prepares the exact source as last-known-good first, then atomically replaces the active
workflow as its commit point.

## Canonical valid example

```md
---
version: 2
workflow:
  mode: direct
owners:
  orchestrator: ORCHESTRATOR
approvals:
  require_plan: true
  require_verify: true
  require_network: true
retry_policy:
  normal_exit_continuation: true
  abnormal_backoff: exponential
  max_attempts: 5
timeouts:
  stall_seconds: 900
in_scope_paths:
  - "**"
---

## Prompt Template

Use approved task scope and policy gates from this workflow contract.

## Checks

- preflight
- verify
- finish

## Fallback

last_known_good: .agentplane/workflows/last-known-good.md
```

## Canonical invalid example

```md
---
version: 3
workflow:
  mode: fast
owners:
  orchestrator: ""
approvals:
  require_plan: "yes"
retry_policy:
  abnormal_backoff: linear
  max_attempts: 0
timeouts:
  stall_seconds: -1
in_scope_paths: []
extra_field: true
---

## Prompt

This section name is invalid for the contract.
```

Why invalid:

- `version` is unsupported; only 1 and 2 are accepted
- `workflow.mode` enum mismatch
- empty `owners.orchestrator`
- `approvals.require_plan` must be boolean
- missing `approvals.require_verify` and `approvals.require_network`
- `retry_policy.abnormal_backoff` enum mismatch
- `retry_policy.max_attempts` below minimum
- `timeouts.stall_seconds` below minimum
- `in_scope_paths` must be non-empty
- `extra_field` is not allowed
- missing required sections (`Prompt Template`, `Checks`, `Fallback`)
