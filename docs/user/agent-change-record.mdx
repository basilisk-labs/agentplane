## Status

Agent Change Record, abbreviated ACR, is a proposed AgentPlane evidence format. The current
AgentPlane lifecycle already records task, plan, approval, verification, finish, policy, and Git
state. ACR v0.1 turns that lifecycle into one portable, machine-readable record.

ACR is not implemented as a CLI command yet. This page defines the standard AgentPlane should
produce and validate.

## Purpose

ACR is a repo-local evidence manifest for AI-authored software changes.

It answers seven review questions in one structured artifact:

1. What task was executed?
2. Which plan was approved or waived?
3. Which agent and toolchain performed the work?
4. Which Git base and work commits define the change?
5. Which policy decisions were applied?
6. Which verification checks ran, passed, failed, or were waived?
7. Is the change merge-ready under the repository policy?

The default AgentPlane path is:

```text
.agentplane/tasks/<task-id>/acr.json
```

The path is task-local because ACR is per-task evidence. Export paths may exist for integrations,
but the task-local record is the default producer location.

## Source-of-truth model

ACR is a derived projection, not a new canonical task source.

Canonical inputs stay where they already belong:

- Policy: `AGENTS.md` or `CLAUDE.md`.
- Workflow and configuration: `.agentplane/WORKFLOW.md`.
- Task state and task documentation: `.agentplane/tasks/<task-id>/README.md`.
- Verification records: `agentplane verify ...` output appended to the task record.
- Finish metadata: `agentplane finish ... --commit <git-rev>`.
- Git history: local commits, branches, PRs, and merge records.

ACR must be generated from those inputs and then validated against them. A hand-edited ACR that
does not match local task evidence is invalid.

## Non-goals

ACR v0.1 deliberately stays narrow:

- It does not replace AgentPlane task records.
- It does not require a hosted dashboard.
- It does not store raw prompts by default.
- It does not store raw model transcripts by default.
- It does not store secrets, environment variables, credentials, or private tokens.
- It does not embed full shell output by default.
- It does not require the ACR file to know the commit hash of the commit that contains the ACR
  itself.

## Git integrity model

ACR distinguishes the actual work commit from the evidence commit.

```text
base_commit -> work_commit -> evidence_commit
```

- `base_commit` is the effective base before the task work.
- `work_commit` is the commit or revision containing the product/docs/code change.
- `evidence_commit` is any later commit that adds or refreshes task evidence, including ACR.

The required field is `repository.work_commit`. Requiring "the commit that contains this ACR" would
create a self-reference problem: the hash is unknown until after the ACR is committed.

## Minimal CLI surface

The proposed AgentPlane command group is:

```bash
agentplane acr generate <task-id> --work-commit <rev> --write
agentplane acr validate <task-id> --mode local
agentplane acr check <task-id> --mode ci
agentplane acr explain <task-id>
agentplane acr schema --version 0.1
```

Expected behavior:

- `generate` builds ACR from task, policy, verification, and Git state.
- `validate` checks schema and local evidence invariants.
- `check` acts as a CI/PR merge gate.
- `explain` prints a concise reviewer summary.
- `schema` emits the bundled JSON Schema.

The first implementation should be opt-in. A later release can wire ACR into `finish` through an
explicit `--write-acr` flag and then a config-controlled `write_on_finish` default.

## Validation modes

### `schema`

Checks only the file shape:

- file exists
- file is valid JSON
- `acr_version` is supported
- JSON matches the ACR v0.1 schema

### `local`

Checks schema plus repository evidence:

- task exists
- plan artifact exists when `plan.status=approved`
- approvals are structurally valid
- `base_commit` and `work_commit` resolve locally
- `work_commit` descends from `base_commit`
- evidence paths are repository-relative
- evidence hashes match the referenced files
- verification state is internally consistent
- record digest matches canonicalized content

### `ci`

Checks local mode plus merge expectations:

- ACR lives at `.agentplane/tasks/<task-id>/acr.json` unless a path override is explicit.
- current branch or PR contains `work_commit`
- required plan approval exists
- required verification is passed
- no policy decision is `fail`
- manual override and verification waiver are rejected unless explicitly allowed and approved

## Record shape

Required top-level fields for ACR v0.1:

```text
acr_version
record_type
record_id
created_at
producer
repository
task
agent
plan
permissions
policy
changes
verification
approvals
evidence
result
integrity
```

Optional:

```text
extensions
```

Unknown top-level fields are rejected in v0.1. Extension keys must use reverse-DNS naming and must
not override core fields.

## Merge readiness

`result.merge_ready=true` requires all of the following:

- plan is approved or explicitly waived with approval
- verification passed or was explicitly waived with approval
- no policy decision is `fail`
- no unapproved `manual_override` exists
- referenced evidence is present and hash-valid in `local` and `ci` validation modes

This keeps ACR useful as a merge gate rather than only as a retrospective report.

## Privacy baseline

ACR must be safe to commit to Git.

Forbidden by default:

- raw prompts
- raw model transcripts
- secrets
- environment variables
- access tokens
- private keys
- full shell output
- authenticated URLs
- unnecessary personal data

Allowed by default:

- task intent and short summaries
- repository-relative artifact paths
- artifact hashes
- command names
- exit codes
- tool names and versions
- policy rule ids
- verification status
- redacted log artifact references

Private evidence bundles may be added in a future version, but they must stay separate from the
public ACR v0.1 record.

## Integration with AgentPlane lifecycle

ACR maps onto existing lifecycle stages:

| AgentPlane stage                  | ACR projection                                                 |
| --------------------------------- | -------------------------------------------------------------- |
| `task new`                        | `task.task_id`, `task.title`, `task.intent`                    |
| `task plan set`                   | `plan.artifact` and plan evidence                              |
| `task plan approve`               | `approvals[]` entry with `type=plan_approval`                  |
| `work start` / `task start-ready` | `repository.base_commit`, agent identity, permissions snapshot |
| implementation commit             | `repository.work_commit`, `changes`                            |
| `verify`                          | `verification`, verification evidence                          |
| `finish`                          | `result`, finish evidence, merge readiness                     |

The producer should never fabricate missing lifecycle data. Unknown model identity, unavailable
remote metadata, or skipped checks must be represented explicitly as `unknown`, `not_run`,
`partial`, or `waived` with approval, depending on the field.

## Market-standard criteria

For ACR to work beyond AgentPlane, the format needs these properties:

1. Git-native: a record can live in any Git repository without a hosted control plane.
2. Evidence-first: claims are backed by hashes, local paths, Git revisions, and policy decisions.
3. Privacy-safe: the default public artifact is commit-safe.
4. Producer-neutral: AgentPlane can produce ACR, but other tools can implement the same schema.
5. Validator-friendly: schema validation is not enough; local and CI modes verify evidence.
6. Extensible: reverse-DNS extensions allow vendors to add fields without changing core semantics.
7. Reviewable: a human reviewer can read `acr explain` and inspect referenced artifacts.
