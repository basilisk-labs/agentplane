## workflow_mode

Configured in `.agentplane/WORKFLOW.md`.

### direct

- Single-checkout workflow.
- Task branches and PR artifacts are optional.
- Tasks can be implemented and closed on the current branch.
- `agentplane work start` records the active task (single-stream) for the current checkout in direct mode.

### branch_pr

- Task branches and worktrees are required.
- `agentplane work start` must run on base branch and must include `--worktree`.
- `agentplane work start` now fails fast when the pinned base branch is behind its upstream tracking ref; refresh the base branch first instead of opening a hosted PR from stale history.
- Implementation commits happen in the task worktree.
- Lifecycle/status commits are task-state checkpoints and are not the implementation commit recorded at finish.
- INTEGRATOR performs merge/integration and finish from the base checkout after the task branch is verified.
- Local `agentplane pr ...` commands manage tracked PR artifacts only. They do not open or merge a hosted GitHub PR.

## Task Branches and Worktrees

- Branch naming: `task/<task-id>/<slug>` by default (config: `branch.task_prefix`).
- Worktrees live under `.agentplane/worktrees/` by default (config: `paths.worktrees_dir`).

## Umbrella and Batch Tasks

Use one branch/worktree per independent task by default. If several tasks are tightly coupled
parts of one delivery theme, create one primary task as the umbrella implementation branch and
link the dependent tasks into that PR with `agentplane pr open|update --include-task <task-id>`.

The primary task owns the branch, worktree, PR artifacts, merge, and cleanup. Included tasks keep
their own plan, start, verification, and finish evidence, but they do not need separate branches
unless they can be reviewed, merged, or reverted independently.

Keep implementation out of `main` until the umbrella PR is integrated. The top-level task may be
created from the base branch, but all code/docs changes for the batch still happen inside the
primary task worktree.

## PR Artifacts

- Location: `.agentplane/tasks/<task-id>/pr/`.
- Purpose: local PR simulation for review and integration.

Artifact roles:

| File               | Role                                                            | Drift rule                                                                                                                                   |
| ------------------ | --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `meta.json`        | Canonical PR identity and lifecycle record for the task branch. | Must be preserved and updated by `agentplane pr ...`, `integrate`, or hosted-close reconciliation. Do not regenerate it from Markdown prose. |
| `diffstat.txt`     | Evidence snapshot for the current rendered task branch diff.    | May be regenerated by `agentplane pr update`; stale content means the review packet is stale, not that task state changed.                   |
| `review.md`        | Local review projection for agents and maintainers.             | May be regenerated from canonical task data plus PR evidence. It must not be treated as the source of task lifecycle truth.                  |
| `github-title.txt` | Hosted PR title projection.                                     | May be regenerated from task metadata. Use it as the exact hosted title payload.                                                             |
| `github-body.md`   | Hosted PR body projection.                                      | May be regenerated from task metadata and review evidence. Use it as the exact hosted body payload.                                          |
| `notes.jsonl`      | Optional handoff note stream.                                   | Preserve entries once present. Absence is equivalent to no handoff notes.                                                                    |
| `verify.log`       | Optional verification command log.                              | Preserve entries once present. Absence is equivalent to no branch-local verify log.                                                          |

Only `meta.json` is a canonical PR lifecycle record. The Markdown/text files are either evidence
snapshots or rendered projections; agents may refresh them, but they must not infer task status
from projection prose when the task README/frontmatter or `meta.json` says otherwise.

## Local PR artifacts vs hosted PR checks

Treat these as two separate layers:

1. `agentplane pr open|update|check` maintains the repository-local review packet under `.agentplane/tasks/<task-id>/pr/`.
2. `gh pr ...` is the hosted-review layer that triggers server-side required checks using the derived `github-title.txt` and `github-body.md` projections from that same local packet.

If this repository requires remote checks before `main` moves, do both:

- keep local PR artifacts current with `agentplane`;
- push the task branch and wait for hosted required checks with the workflow helper or raw `gh`.

If a hosted GitHub PR is merged directly into `main` instead of going through local
`agentplane integrate`, the repository now pushes a deterministic closure branch for the
task artifacts automatically. If the organization allows GitHub Actions to create PRs, the
workflow also opens the follow-up closure PR and enables auto-merge after the normal
required checks pass. If the organization blocks Actions PR creation, the workflow leaves a
manual PR link on the merged task PR instead of failing silently.

## Handoff Notes

- Reviewers and executors record handoff notes in `review.md`; the same note stream is projected into `github-body.md`.
- INTEGRATOR appends notes into the task closure record.

## Commands

See the [generated CLI reference](cli-reference.generated) for the public command surface.

Typical branch_pr flow:

```bash
agentplane task plan approve <task-id> --by ORCHESTRATOR
agentplane work start <task-id> --agent CODER --slug <slug> --worktree
cd .agentplane/worktrees/<task-id>-<slug>
agentplane task start-ready <task-id> --author CODER --body "Start: ..."
# edit files
git add <changed-files>
git commit -m "Implement <task>"
agentplane pr open <task-id> --branch task/<task-id>/<slug> --author CODER
agentplane pr update <task-id>
git push -u origin task/<task-id>/<slug>
gh pr create --base main --head task/<task-id>/<slug> --title "$(cat .agentplane/tasks/<task-id>/pr/github-title.txt)" --body-file .agentplane/tasks/<task-id>/pr/github-body.md
bun run workflow:wait-remote-checks
agentplane verify <task-id> --ok --by CODER --note "Focused checks passed on the task branch."
cd <base-checkout>
agentplane pr check <task-id>
agentplane integrate <task-id> --branch task/<task-id>/<slug> --run-verify
agentplane finish <task-id> --author INTEGRATOR --body "Verified: ..." --result "<one line>" --commit <implementation-git-rev> --close-commit
agentplane cleanup merged --finalize
```

`finish --commit-from-comment` is intentionally not part of `branch_pr`. Finish runs on the base
checkout, so it records the explicit implementation commit hash that already exists on the task
branch. Use comment-driven status commits only where the specific lifecycle command supports them
from the task worktree.

Underlying hosted-check command:

```bash
gh pr checks --watch --required --fail-fast
```

Hosted checks are the gate. Do not integrate or finish on base branch until `bun run workflow:wait-remote-checks` returns green.
`agentplane integrate` defaults to `merge` so task branch commits stay in base history; pass
`--merge-strategy squash` only when compacting history is intentional.

Hosted merge shortcut:

1. Keep `.agentplane/tasks/<task-id>/pr/*` current with `agentplane pr open` and `agentplane pr update`.
2. Merge the hosted GitHub PR after required checks are green.
3. Let the `Task Hosted Close` workflow push the deterministic closure branch for the task artifacts.
4. If the organization allows Actions PR creation, the workflow opens the follow-up closure PR automatically. Otherwise use the manual PR URL that the workflow comments onto the merged task PR.
5. Pull the updated `main` after the closure PR merges; no local finish-only tail commit should remain necessary.

When GitHub UI shows `Expected — Waiting for status to be reported` for one of the required checks even though
PR workflow jobs already look green, audit branch protection directly:

```bash
bun run workflow:github-protection:check
```

For this repository the canonical merge gate is exactly:

- `Core CI / test`
- `Core CI / test-windows`
- `Docs CI / docs`

Each required check must be bound to a concrete GitHub app in branch protection. Unbound
`app_id=null` entries can leave PRs stuck in `Expected` even when GitHub Actions already emitted
successful `pull_request` check-runs.
