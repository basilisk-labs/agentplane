## Overview

The cloud backend is a remote task backend option for projects that want AgentPlane task state to
sync through a cloud service.

The local backend stores the canonical task state in the repository. The cloud backend keeps the
same local projection model, but the canonical state and connector-specific sync state live behind a
remote service.

AgentPlane still reads ordinary task commands from the local projection by default:

- `agentplane task list`
- `agentplane task show <task-id>`
- `agentplane doctor`

Network refresh is explicit through backend sync commands.

## What the cloud service owns

The cloud service owns connector-specific details:

- connector authorization;
- connector webhooks;
- connector field mapping;
- sync cursors;
- conflict records;
- remote freshness state.

AgentPlane owns the CLI workflow:

- local projection cache;
- task lifecycle commands;
- explicit pull/push sync;
- freshness warnings before using stale projection data;
- local task rendering and export behavior;
- safe application of service-approved operational fields to known local tasks.

## Connect flow

Initialize with the cloud backend:

```bash
agentplane init --backend cloud
```

Connect the local project to a cloud project:

```bash
agentplane backend connect cloud \
  --endpoint https://sync.agentplane.cloud \
  --project-id proj_456 \
  --provider github-projects \
  --token apc_...
```

The cloud service handles workspace selection, project selection, and connector setup outside the
repository. For example, if the selected connector is GitHub Projects, the cloud service handles the
GitHub App installation and project selection. Paste the returned `token` into `--token`; the CLI
writes it only to the ignored project `.env`.

After connect, the backend JSON should contain non-secret connection metadata:

```json
{
  "id": "cloud",
  "version": 1,
  "settings": {
    "endpoint": "https://sync.agentplane.cloud",
    "project_id": "proj_456",
    "provider": "github-projects",
    "cache_dir": ".agentplane/tasks",
    "stale_after_seconds": 300
  }
}
```

Secrets do not belong in backend JSON. The connect command stores the CLI token in ignored `.env` as
`AGENTPLANE_CLOUD_TOKEN`. You can also provide it through the shell:

```bash
export AGENTPLANE_CLOUD_TOKEN=...
```

## Endpoint selection

Set the endpoint explicitly when connecting:

```bash
agentplane backend connect cloud --endpoint https://sync.agentplane.cloud --project-id proj_456
```

Recommended precedence for implementations:

1. command flag;
2. environment variable;
3. backend JSON setting;

## Sync

Refresh the local projection from the cloud service:

```bash
agentplane backend sync cloud --direction pull
```

Push local dirty task state to the cloud service:

```bash
agentplane backend sync cloud --direction push --yes
```

For large task archives, the CLI automatically uploads the push projection in
byte-limited batches. The cloud service stages those chunks and only replaces
the canonical projection after the final batch is received, then enqueues one
publish job for connected views. This keeps `backend sync cloud --direction
push --yes` as the user-facing command while avoiding oversized HTTP request
bodies.

Operational rule:

- use `pull` before local task mutations when the projection is stale;
- use `push --yes` when local task state should become visible to connected remote views;
- ordinary task reads do not implicitly refresh from the network.

### Watch mode (pull)

To keep the local projection updated while you change GitHub issue state or Project fields, run a
long-lived pull loop:

```bash
agentplane backend sync cloud --direction pull --watch --yes
```

Defaults:

- `--interval-ms 30000`
- runs until interrupted (or use `--max-iterations N` for bounded runs/tests)

### Optional auto-push on status mutations

If the cloud backend is configured with `auto_push_on_mutation=true` (or the environment variable
`AGENTPLANE_CLOUD_AUTO_PUSH_ON_MUTATION=1` is set), AgentPlane will push after successful status
mutations (for example `task start-ready`, `task block`, `task set-status`, `task finish`) so GitHub
issue state stays in sync without an extra manual push step.

Cloud pull writes the local projection cache. In a git repository, that means a successful pull can
produce ordinary working-tree changes under `.agentplane/tasks/**`, as if those task files had been
edited locally. Review and commit those projection changes through the same repository workflow as
other task-state changes.

Access levels:

- `publish_only`: AgentPlane publishes local operational task state to the connected remote view.
  Pull does not apply edits from that view.
- `bidirectional`: the cloud service may apply remote edits to selected operational fields
  (`title`, `status`, `priority`, `owner`, `tags`). AgentPlane still owns the full task contract,
  including plan approval, verification, rollback, findings, comments, events, and README sections.

Conflict modes for pull:

- `--conflict=fail` stops before writing projection changes when the service reports open
  conflicts through either `sync/state` or the pull response.
- `--conflict=diff` is read-only. It prints a summary of known local tasks that would change,
  remote-only tasks that were ignored, and open conflicts; it does not write remote changes into the
  local projection or mark a changed projection as fresh.
- `--conflict=prefer-remote` applies only service-approved operational fields.

Remote-only items are ignored until a local task creation policy is explicitly configured.

## Inspection

Inspect the active connection:

```bash
agentplane backend inspect cloud --yes
```

Expected output should include:

- endpoint;
- project id;
- provider name;
- connection status;
- missing environment keys;
- last checked timestamp;
- projection freshness;
- local state path.

The current cloud backend records `last_checked_at` in the ignored local backend state file after
successful sync. If that timestamp is older than `stale_after_seconds`,
`backend inspect cloud --yes` marks the projection stale. Task commands still read the local
projection; sync remains explicit.

## Connected views

The cloud backend is connector-neutral from the CLI perspective. Connector setup happens in the
cloud service. A connected view can be GitHub Projects, Redmine, or another task system without
requiring a new AgentPlane backend id.

For GitHub Projects, the cloud service can map AgentPlane tasks to GitHub Issues and Project fields.
AgentPlane still treats the cloud backend as `cloud`, not `github-projects`.
