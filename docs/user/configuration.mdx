## Overview

Agent Plane reads `.agentplane/WORKFLOW.md` and validates the YAML front matter as the single
repo-local workflow/config source of truth.
The CLI still accepts legacy `config.json` as an import fallback for older workspaces, but saves
changes back to `WORKFLOW.md`.

`agentplane config show` is the runtime readback of that file. `agentplane quickstart` is startup
guidance assembled from the configured route; it is not a second configuration source.

Use CLI commands to read and update values safely:

```bash
agentplane config show
agentplane config set workflow_mode branch_pr
```

## High-impact keys

### workflow.mode

- `direct`: single checkout workflow.
- `branch_pr`: per-task branches and PR artifacts.

### status_commit_policy

- `off`: do not gate status/comment-driven commits.
- `warn`: allow, but warn when no explicit confirmation.
- `confirm`: require explicit confirmation.

### finish_auto_status_commit

- `false` by default.
- `init` sets it to `true` for `branch_pr`, `false` for `direct`.

### approvals

- `require_plan`
- `require_network`
- `require_verify`
- `require_force`

These keys control approval gates; they do not add or remove CLI capabilities.

### workspace

- `workspace.agents_dir`
- `workspace.workflow_dir`
- `workspace.worktrees_dir`

### branch

- `branch.task_prefix`: namespace for branch_pr implementation branches. Default: `task`.
- `branch.task_close_prefix`: namespace for hosted/base close-tail branches. Default: `task-close`.

Branch names are deterministic:

- Task branch: `<task-prefix>/<task-id>/<slug>`.
- Close branch: `<task-close-prefix>/<task-id>/<sha12>`.
- Worktree path: `<workspace.worktrees_dir>/<task-id>-<slug>`.

Only prefixes and the worktree root are configurable. The `<task-id>/<slug>` and `<task-id>-<slug>`
suffixes stay fixed so task ownership, cleanup, hooks, and hosted-close recovery can parse them.

### tasks.tags (primary tag model)

- `tasks.tags.primary_allowlist`
- `tasks.tags.strict_primary`
- `tasks.tags.fallback_primary`
- `tasks.tags.lock_primary_on_update`

Task policy resolves exactly one primary tag per task. Secondary tags remain free-form.

### tasks.verify

- `tasks.verify.require_steps_for_primary`
- `tasks.verify.require_verification_for_primary`
- `tasks.verify.spike_tag`
- `tasks.verify.enforce_on_plan_approve`
- `tasks.verify.enforce_on_start_when_no_plan`

Compatibility note: `tasks.verify.required_tags` still exists in schema and may be present in older configs.
Current enforcement should be documented and configured through primary-based keys.

### tasks.backend

- `tasks.backend.config_path` points to the active backend config file.

### execution

- `execution.profile`
- `execution.reasoning_effort`
- `execution.text_verbosity`
- `execution.tool_budget`
- `execution.stop_conditions`
- `execution.handoff_conditions`
- `execution.unsafe_actions_requiring_explicit_user_ok`

`execution.reasoning_effort` controls depth, not answer length. `medium` is the balanced default;
`low` is latency-oriented; `high` and `xhigh` should be reserved for conservative profiles or
eval-backed work. `execution.text_verbosity` controls output length/structure in execution profile
prompts.

### runner

Runner configuration controls the local `agentplane task run <task-id>` execution path. The default
adapter is `codex`; it runs Codex non-interactively against the prepared AgentPlane bundle and starts
the prompt with `/goal ...`. Production run state, events, receipts, traces, and stderr are stored in
supervisor-owned local Git administration storage. Native Codex `read-only` and `workspace-write`
sandboxes cannot write that storage. `danger-full-access` and host-authority custom or Hermes
runners can address it, so their filesystem authority remains unverified. Inspect artifacts with
`agentplane task run status`, `agentplane task run inspect`, and `agentplane task run logs`; their
internal filesystem location is not a public mutation surface.

`status` and `inspect` expose active-claim ownership plus `projection_pending` and
`reconcile_required`. `reconcile_required=true` means the selected terminal supervisor run still
owns the task's active claim. `projection_pending=true` additionally means TaskData does not yet
contain that exact run ID, status, and update timestamp. `logs` prints the same retained-claim
condition as a warning. Treat these fields as degraded lifecycle state that must reconcile before a
new provider run can start. `status` and `inspect` remain read-only diagnostics, so automation must
test these JSON booleans rather than interpret a successful inspection exit as reconciliation.

A terminal run retains its active claim when cleanup of the declared process scope reports a live
residual, fails, or cannot establish an observation. A later invocation does not auto-retire that
claim merely because the supervisor owner exited. Confirmed cleanup of the declared scope can retire
the claim even when `containment_state=limited`; the claim serializes provider authority but does
not prove that detached descendants outside that scope are absent. Failed, unknown, or residual
cleanup stays fail-closed and blocks the next provider run.

`direct_child_only` with `residual_alive=false` confirms only the immediate child. Its
`runner.process_group_cleanup` and `runner.process_containment` checks are `not_run`, not failed,
because no process-group or bounded descendant-lifetime claim was established. Under
`workspace-write` or a custom host-authority runner, `success_policy` remains `unverified` and names
the residual descendant-lifetime risk while the mechanical run result may still be `success`. An
exact native inherited `read-only` sandbox with no writable scope may instead establish overall
`observed_success` through the passed filesystem-effect boundary; the receipt still records limited
process-lifetime containment separately.

Use `agentplane task run reconcile <task-id>` only after `status` or `inspect` reports a stale
reconcilable owner. The command starts no provider and sends no process signal. It refuses active,
unverified, running, spawn-authorized, cleanup-unverified, or residual claims rather than forcing
their retirement.

New run directories are created only in supervisor storage. Public inspection commands expose
`storage=supervisor|legacy_task`; an explicit run ID falls back to a historical 0.6.24 task-local
`.agentplane/tasks/<task-id>/runs/<run-id>/` artifact only when that supervisor run is safely absent.
Implicit latest selection uses legacy storage only when supervisor storage has no entries. Invalid,
unsafe, or incomplete supervisor storage is reported as an error and never hidden by the fallback.
A current supervisor-owned running run is cancelled cooperatively: the lifecycle command publishes
an immutable intent in that run directory, and only the supervisor that owns the child-process
handle may send TERM or KILL. The lifecycle command never signals a persisted PID. Historical
0.6.24 task-local supervisors do not poll this intent protocol, so a live or unverifiable running
legacy run fails immediately with `legacy_running_cancellation_unsupported` and receives no signal.
The refusal also applies when its persisted child appears absent: AgentPlane neither signals the PID
nor synthesizes a terminal state for any running legacy artifact.

For supervisor storage, a running orphan is terminalized only when the child is confirmed gone, a
matching active claim has a stale owner, and an exclusive generation-scoped recovery lease is held.
Without that authority the lifecycle remains fail-closed and no persisted PID is signaled. If a
start owner dies before child-spawn authorization, cancellation can similarly recover a prepared run
only after the owner lease expires and under its matching stale active claim and recovery lease. If
immutable spawn authorization already exists but no running state was recorded, AgentPlane keeps the
run and its active claim fail-closed with `spawn_authorized_but_unconfirmed`. This bounded crash
window is not auto-recovered because a provider child may already exist.

Stale active claims are automatically recoverable only for an absent run, a terminal run after
TaskData reconciliation, or an incomplete pre-provider run. An existing recovery lease blocks other
recovery attempts even when its owner appears stale or unverified, and an invalid lease fails with
`E_RUNTIME`; AgentPlane does not age out or steal recovery authority.

Task outcome projection preserves each run's immutable creation time. A delayed terminal update for
an older run is retained in history instead of replacing a newer current run. The current active
claim remains authoritative if the wall clock moves backwards, and same-run lifecycle progress
outranks timestamps. Non-local task backends must support revision-guarded writes; a revision
conflict leaves the terminal active claim in place so a later invocation can reconcile the
supervisor state before another provider run.

Atomic task and runner control-file writes fsync their contents. AgentPlane also fsyncs the parent
directory where the operating system and filesystem support it. On platforms that reject directory
fsync, this step is best-effort and does not provide a universal power-loss guarantee.

The workflow field `observability.runs_dir` is informational and reserved. AgentPlane enforces the
Git-common-dir runner layout; changing this field does not override production artifact storage.

### runner.custom

Custom runner configuration controls the `custom` adapter command and enforcement settings. `init
--tool hermes` selects the custom adapter and seeds `runner.custom.command` as `["hermes",
"agentplane", "run"]`; the custom adapter also exports `AGENTPLANE_RUNNER_TASK_ID` for task-targeted
Hermes launchers.

### feedback

- `feedback.github_issues.enabled`
- `feedback.github_issues.repository`
- `feedback.github_issues.transport`
- `feedback.github_issues.cloud_endpoint`
- `feedback.github_issues.allow_anonymous_cloud`
- `feedback.github_issues.prompt_on_internal_error`
- `feedback.github_issues.include_insights_report`
- `feedback.github_issues.dedupe`
- `feedback.github_issues.labels`

GitHub issue feedback defaults to disabled. Enable it when you want internal Agentplane errors to
offer a privacy-bounded issue path for maintainers. `transport=github` uses the user's GitHub auth,
`transport=cloud` sends an anonymous intake to Agentplane Cloud, and `transport=auto` tries GitHub
first before falling back to cloud. Cloud intake requires
`feedback.github_issues.allow_anonymous_cloud=true`.

Preview the issue payload before creating anything:

```bash
agentplane insights issue --error-code E_INTERNAL --dry-run
```

To opt in, ask your agent to enable feedback issue prompts or run:

```bash
agentplane config set feedback.github_issues.enabled true
```

For a one-shot issue without changing project config, pass an explicit command opt-in:

```bash
agentplane insights issue --allow-disabled-feedback --error-code E_INTERNAL
```

To allow anonymous cloud fallback:

```bash
agentplane config set feedback.github_issues.allow_anonymous_cloud true
agentplane config set feedback.github_issues.transport auto
```

### acr

Agent Change Record support is active. The default config surface is:

```json
{
  "acr": {
    "enabled": true,
    "version": "0.1.0",
    "write_on_finish": true,
    "require_for_pr_check": false,
    "default_validation_mode": "local",
    "include_model_identity": "when_known",
    "include_prompts": false,
    "include_tool_outputs": false
  }
}
```

This config remains privacy-safe by default. ACR records evidence pointers, hashes, Git revisions,
policy decisions, and verification status; it does not commit prompts, transcripts, or raw tool
output by default. `write_on_finish=true` refreshes `.agentplane/tasks/<task-id>/acr.json` during
successful `finish`; `require_for_pr_check=true` turns ACR into a PR/CI gate that requires
merge-ready semantic validation, not just schema validation. See [Agent Change Records](../reference/acr).

## Examples

Set primary tags that require Verify Steps:

```bash
agentplane config set tasks.verify.require_steps_for_primary '["code","data","ops"]'
```

Set primary tags that require recorded verification before finish:

```bash
agentplane config set tasks.verify.require_verification_for_primary '["code","data","ops"]'
```

Switch to branch workflow:

```bash
agentplane config set workflow_mode branch_pr
```

Enable stricter force approvals:

```bash
agentplane config set agents.approvals.require_force true
```
