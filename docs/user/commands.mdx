## Help and version

```bash
agentplane --help
agentplane help task
agentplane help task plan
agentplane help task doc set --compact
agentplane help --all
agentplane --version
```

`agentplane --help` and `agentplane help` show the normal installed-project surface by default.
The first screen is intentionally biased toward commands an agent needs for project work: startup,
task lifecycle, branch/PR workflow, backend sync, recipes, diagnostics, and setup.
For context management, `agentplane` keeps the human task-factory surface visible: `context init`,
`context learn files`, `context learn changes`, `context learn tasks`, `context search`,
`context show`, and `context check`.

Advanced recovery and maintenance commands stay available through direct help or the full registry:

```bash
agentplane help --all
```

Framework-maintainer commands for developing AgentPlane itself are intentionally outside the public
user command path and live in the direct-link developer docs.

## Init

```bash
agentplane init
```

If the directory is not a git repository, `agentplane init` initializes git in that directory.
Use `agentplane init --dry-run --yes` to preview the setup plan without creating `.git`,
`.agentplane`, backups, hooks, or commits. Add `--output json` for a machine-readable plan.

Interactive init now starts with a profile menu:

- `light`
- `normal` (default)
- `full-harness`

Hooks default by profile:

- `light`: disabled
- `normal`: enabled
- `full-harness`: enabled

`execution.profile` is stored in `.agentplane/WORKFLOW.md`.

Non-interactive:

```bash
agentplane init --setup-profile normal --workflow direct --backend local --execution-profile balanced --strict-unsafe-confirm true --hooks true --require-network-approval true --yes
agentplane init --dry-run --yes --output json
```

Workflow diagnostics and repair:

```bash
agentplane doctor --fix
```

## Runtime diagnostics

```bash
agentplane runtime explain
agentplane runtime explain --json
```

Use this when you need to prove which binary is active, whether you are on
`repo-local`, `repo-local-handoff`, or a global install, and which `agentplane` /
`@agentplaneorg/core` package roots are actually being resolved at runtime.

If the repository expects a newer CLI than the active runtime, `runtime explain` and `doctor` now
print a concrete recovery path and a follow-up `agentplane runtime explain` verification step.

## Tasks

```bash
agentplane task list
agentplane help task
agentplane help task plan
agentplane task show <task-id>
agentplane task new --title "..." --description "..." --priority med --owner CODER --tag nodejs
agentplane task run <task-id> --dry-run
agentplane task run <task-id>
agentplane task run show <task-id> [run-id]
agentplane task run tail <task-id> [run-id] --lines 20
agentplane task doc set <task-id> --section Summary --text "..."
agentplane task doc set <task-id> --full-doc --file ./task-readme.md
```

`task run` creates run artifacts under `.agentplane/tasks/<task-id>/runs/<run-id>/`.

- Start with `task run show`: it prints the selected run, status, adapter, effective policy,
  refusal metadata, timeout reason, metrics, capability summaries, and artifact paths from
  `run-state.json` / `result.json`.
- Use `task run tail` when you only need the last N raw trace lines from `agent-trace.jsonl`.
- Advanced raw-trace export remains available through the hidden help surface when you need the full
  artifact unchanged. When `run-id` is omitted, `show`, `tail`, and advanced trace export all select
  the latest run for the task.
- Use `agent-trace.jsonl` and `stderr.log` when you need the raw runner trace.
- Use `codex-last-message.md` only as a side artifact for human inspection when the adapter writes it.
- Task-facing runner logs stay compact and English-only: they expose status, metrics, and artifact paths, but
  they do not copy assistant prose from raw trace artifacts back into the task README.
- If `task run show` reports `timeout_reason`, inspect `task run tail` first to see the last raw activity
  before the timeout. If it reports `policy_refusal`, inspect `policy_requested`, `policy_effective`,
  and `policy_fields` before changing recipe or runner settings.
- `task run --dry-run` and `task run show` now print both raw JSON blocks and compact
  `capability[...]` / `policy_field[...]` lines. Use these summaries to see whether a field is
  `native`, `wrapper`, `advisory`, or `unsupported` without opening `bundle.json`.
- For `runner.default_adapter=custom`, the `capability[sandbox]` note is the quickest way to tell
  whether the adapter is still env-only or is configured to wrap execution through
  `runner.custom.enforcement.mode=codex_sandbox_full_auto`. When wrapper mode is active, the same
  line also shows the currently supported sandbox values.

## Advanced task projections

Optional projection and maintenance commands are intentionally outside the normal public happy path.
They remain available for integrations, legacy recovery, and framework maintenance through the
hidden help surface (`agentplane help --all`).

The exported snapshot is derived from the current backend projection. It is not the canonical
source of truth.

The Obsidian projection command writes a Markdown navigation projection under
`.agentplane/generated/obsidian/` for compatibility with Obsidian and plain Markdown browsers:

- `.agentplane/generated/obsidian/index.md`
- `.agentplane/generated/obsidian/tasks.md`
- `.agentplane/generated/obsidian/by-status/*.md`
- `.agentplane/generated/obsidian/by-tag/*.md`
- `.agentplane/generated/obsidian/by-owner/*.md`

Open `.agentplane/generated/obsidian/` as an Obsidian vault to browse task IDs, statuses, owners,
tags, and dependency links while keeping `.agentplane/tasks/<task-id>/README.md` as the canonical
task record.
The companion cleanup command removes only the generated Obsidian projection files; it does not
delete canonical task READMEs or non-generated Markdown files. Inspect its exact syntax with
the hidden help surface.

This surface is a generated task-wiki/navigation adapter, not a separate storage authority. Legacy
generated files in the `.agentplane/` root are still cleaned when they carry the generated marker.

## Agent Change Record

ACR commands generate and validate the task-local Agent Change Record:

```bash
agentplane acr generate <task-id> --work-commit <rev> --write
agentplane acr validate <task-id> --mode local
agentplane acr check <task-id> --mode ci
agentplane acr explain <task-id>
agentplane acr schema --version 0.1
```

These commands derive `.agentplane/tasks/<task-id>/acr.json` from existing
task lifecycle evidence. `finish` refreshes the task-local ACR automatically when
`acr.write_on_finish=true`; use `--no-write-acr` only for explicit recovery cases. ACR remains a
derived projection, not a second source of truth. `acr validate --mode schema` checks only JSON
shape; `acr check --mode ci` enforces merge-ready semantics. See
[Agent Change Record](agent-change-record).

## Local context

Local context commands manage repository-owned memory artifacts and the generated search
projection:

```bash
agentplane context init
agentplane context doctor
agentplane context reindex
agentplane context search "query"
agentplane context show context/wiki/page.md#section=section-slug
agentplane context ingest --changed
agentplane context ingest ./notes.md --run
agentplane context harvest tasks --tag release --limit 20 --dry-run
agentplane context harvest tasks --tag branch_pr --batch-size 25
agentplane context harvest tasks --tag branch_pr --write-proposals
agentplane context verify-task <task-id>
```

Source artifacts live under `context/**`. Derived context artifacts live under
`.agentplane/context/derived/**`, and the fast SQLite projection is stored in the shared cache
database at `.agentplane/cache.sqlite`. Use `context harvest tasks` to convert completed task
history into raw evidence scaffolds or default `CURATOR` extraction tasks before promoting reusable
knowledge. Semantic extraction is agent-owned by default for non-write broad harvest runs: standard
`CURATOR` tasks read task READMEs and ACR evidence in bounded oldest-first batches, then update
wiki/facts/graph outputs through the normal task lifecycle. `--write-proposals` is the deterministic
raw/proposal scaffold path, not the semantic truth path. Write modes also mark harvested task
READMEs with `extensions.context_harvest` for raw ingestion and
`extensions.context_task_extraction` for queued semantic extraction, so later broad runs can skip
unchanged ingested or queued tasks by source digest instead of guessing from wiki files. Write modes
require `agentplane context init` first. Use `context verify-task` before closing work that creates
or relies on context artifacts. See [Local context](local-context).

There is no public `context list` command by design. Search is the primary discovery surface; use
`context graph summary`, `context graph show`, and `context graph neighbors` when you need structured
graph inspection instead of text search.

## Backend sync (remote backends)

For remote backends, use explicit sync:

```bash
agentplane backend sync cloud --direction pull
agentplane backend sync cloud --direction push --yes
agentplane sync redmine
agentplane backend sync redmine --direction pull
agentplane backend sync redmine --direction push --yes
```

`sync` and `backend sync` default to `--direction push`; prefer explicit `--direction pull` when
refreshing the local projection from a remote source.

Ordinary task reads (`task list`, `task show`, `doctor`) use the current projection by default and
do not implicitly refresh from the network.

## Workflow

Configured branch_pr path in this repository:

```bash
agentplane work start <task-id> --agent CODER --slug <slug> --worktree
agentplane task start-ready <task-id> --author CODER --body "Start: ..."
git add <changed-files>
git commit -m "Implement <task>"
agentplane task verify-show <task-id>
agentplane pr open <task-id> --branch task/<task-id>/<slug> --author CODER
agentplane pr update <task-id>
agentplane verify <task-id> --ok --by CODER --note "Focused checks passed on the task branch."
bun run workflow:wait-remote-checks
agentplane integrate <task-id> --branch task/<task-id>/<slug> --run-verify
agentplane finish <task-id> --author INTEGRATOR --body "Verified: ..." --result "One-line outcome" --commit <git-rev> --close-commit
```

`agentplane pr ...` manages local PR artifacts under `.agentplane/tasks/<task-id>/pr/`, including `review.md`, `github-title.txt`, and `github-body.md`. A normal `agentplane pr open` run also publishes the task branch and creates or links the hosted GitHub PR; use `--sync-only` only when you intentionally want local artifacts without remote side effects. The canonical required-check wait step is `bun run workflow:wait-remote-checks`, which delegates to `gh pr checks --watch --required --fail-fast`. `agentplane integrate` defaults to `merge` so task branch commits stay in base history; pass `--merge-strategy squash` only when compacting history is intentional.

GitHub protection audit for this repository:

```bash
bun run workflow:github-protection:check
```

Use this when a hosted PR shows `Expected — Waiting for status to be reported` for the required
checks even though the `pull_request` workflows appear green. The audit validates the canonical
required-check set on `main` and rejects unbound `app_id=null` status entries.

Direct-mode alternative:

```bash
agentplane task start-ready <task-id> --author CODER --body "Start: ..."
agentplane task verify-show <task-id>
agentplane verify <task-id> --ok --by REVIEWER --note "Looks good"
agentplane finish <task-id> --author CODER --body "Verified: ..." --result "One-line outcome" --commit <git-rev>
```

In `direct`, `finish` creates the deterministic close commit by default. In this repository,
`direct` is an explicit alternative rather than the default guarded route; use `--no-close-commit`
only when close handling must stay manual.

Fallback/manual paths:

```bash
agentplane block <task-id> --author CODER --body "Blocked: ..."
agentplane finish <task-id> --author CODER --body "Verified: ..." --result "..." --commit <git-rev> --no-close-commit
agentplane finish <task-id> --author CODER --body "Verified: ..." --result "..." --commit <git-rev> --close-commit
agentplane finish <task-id> --author CODER --body "Verified: ..." --result "..." --commit <git-rev> --close-commit --close-unstage-others
```

For any commit-capable command (`commit`, `guard commit`, `start/block/finish ... --commit-from-comment`):

- Always pass explicit allowlist prefixes (`--allow` / `--commit-allow` / `--status-commit-allow`).
- Top-level `agentplane commit` can auto-stage changed paths that match the provided allowlist when the git index starts empty.
- `--allow-tasks` is intentionally task-scoped: it covers the optional exported tasks snapshot plus artifacts under the active task subtree, not unrelated `.agentplane/tasks/**` directories.
- Path-scoped protected overrides can also stand alone without a duplicate explicit prefix: `--allow-policy`, `--allow-config`, `--allow-hooks`, and `--allow-ci` each admit their own protected path family for `commit` auto-stage and `guard commit` policy checks.
- `--allow-base` is different: it is only a branch-level override for base-branch edits and does not select any file paths by itself.
- `agentplane guard commit` remains staged-only; it validates the current index and never stages files for you.
- If `commit.dco.enabled=true`, AgentPlane-created commits append the configured `Signed-off-by` trailer. The `commit-msg` hook rejects manual commits without a valid `Signed-off-by` trailer, but it does not require every manual commit to use the configured AgentPlane identity.
- If commit stderr reports `git_pre_commit_format` or `git_pre_commit_lint`, run `bun run format` or `bun run lint:core` respectively, then retry the original `commit` / `finish` command.
- Repo-wide prefix `.` is rejected by policy.
- Fast path to build safe prefixes from current index:

```bash
agentplane guard suggest-allow --format args
```

Status transitions (normal mode) are strict:

- `TODO -> DOING|BLOCKED`
- `DOING -> DONE|BLOCKED`
- `BLOCKED -> TODO|DOING`

Status/comment-driven commits are allowed only on major transitions:

- `TODO -> DOING`
- `DOING -> BLOCKED`
- `BLOCKED -> DOING`
- `DOING -> DONE`

## Recipes

```bash
agentplane recipes list-remote --refresh --yes
agentplane recipes install code-map --refresh --yes
agentplane recipes list
agentplane recipes info <id>
agentplane recipes add <id|id@version>
agentplane recipes active
agentplane recipes explain <id>
agentplane recipes update <id>
agentplane recipes detach <id>
```

`recipes list-remote` inspects the signed remote catalog. `recipes install` imports a recipe into
the global cache. `recipes list` and `recipes info` inspect the global cache. `recipes add` vendors a
cached recipe into the current project, and `recipes active` / `recipes explain` inspect the
project-local runtime surface.

After a recipe is vendored, runtime reads project-local packages from `.agentplane/recipes/` rather
than from the global cache.

Recipe packages may still contain scenario assets, but they are no longer exposed as a first-class
public CLI surface. Treat them as recipe-owned internals until a replacement entrypoint is
introduced.

Recipe `run_profile` no longer supports `requires_human_approval` in the main repo runtime. The
current runner accepts only fields it can explain honestly in `task run show`.

## Blueprint catalogs

```bash
agentplane blueprints catalog refresh --index <path-or-url>
agentplane blueprints catalog list
agentplane blueprints catalog info <id>
agentplane blueprints install <id>
agentplane blueprints install pack:<id> --kind pack --activate
```

Blueprint catalogs are optional extension catalogs for route blueprints. They are separate from
recipes: recipes add domain guidance and prompt/runtime assets; blueprints add task-route
definitions that can become project-local trusted routes.

`blueprints catalog refresh` caches an external catalog index under `~/.agentplane/blueprints`.
`blueprints install` caches blueprint package contents under `~/.agentplane/blueprint-catalog` and
materializes the selected definition files into the project-local `.agentplane/blueprints` store.
Installing a blueprint does not activate it unless `--activate` is passed. Activation updates the
project-local blueprint trust config with an explicit allowlist.

Blueprint packs are lightweight catalog entries that expand to multiple blueprint installs. A pack
does not own extra runtime behavior by itself; it only groups blueprint ids that should be installed
together.

During advanced/full-harness init, cached catalog entries can be installed in one step:

```bash
agentplane init --setup-profile full-harness --blueprints pack:enterprise-baseline --yes
```

Use `blueprint:<id>` or `pack:<id>` when a catalog contains overlapping ids.
