## Init

```bash
agentplane init
```

If the directory is not a git repository, `agentplane init` initializes git in that directory.

Interactive init now starts with a profile menu:

- `light`
- `normal` (default)
- `full-harness`

Hooks default by profile:

- `light`: disabled
- `normal`: enabled
- `full-harness`: enabled

`execution.profile` is stored in `.agentplane/config.json`.

Non-interactive:

```bash
agentplane init --setup-profile normal --workflow direct --backend local --execution-profile balanced --strict-unsafe-confirm true --hooks true --require-plan-approval true --require-network-approval true --require-verify-approval true --yes
```

Workflow contract operations:

```bash
agentplane workflow build --validate --dry-run
agentplane workflow build --validate
agentplane workflow restore
agentplane doctor --fix
```

## Runtime diagnostics

```bash
agentplane runtime explain
agentplane runtime explain --json
```

Use this when you need to prove which binary is active, whether you are on
`repo-local`, `repo-local-handoff`, or a global install, and which `agentplane` /
`@agentplaneorg/core` package roots are actually being resolved at runtime.

Inside the Agent Plane framework checkout, the same command now acts as the canonical
framework-dev workflow surface: it prints rebuild steps, the reinstall helper, the follow-up
runtime check, and the optional force-global override in one place.

If watched runtime paths are dirty inside the framework checkout, `agentplane runtime explain` and
`agentplane doctor` warn and continue as read-only diagnostics; stricter commands still keep the
stale-build hard-fail.

## Backend sync (remote backends)

For remote backends (for example Redmine), use explicit sync:

```bash
agentplane sync redmine
agentplane backend sync redmine --direction pull
agentplane backend sync redmine --direction push --yes
```

`sync` and `backend sync` default to `--direction push`; prefer explicit `--direction pull` when refreshing local state from Redmine.

## Workflow

Default direct happy path:

```bash
agentplane task start-ready <task-id> --author CODER --body "Start: ..."
agentplane task verify-show <task-id>
agentplane verify <task-id> --ok --by REVIEWER --note "Looks good"
agentplane finish <task-id> --author CODER --body "Verified: ..." --result "One-line outcome" --commit <git-rev>
```

In `direct`, `finish` creates the deterministic close commit by default. Use `--no-close-commit`
only when close handling must stay manual.

Fallback/manual paths:

```bash
agentplane block <task-id> --author CODER --body "Blocked: ..."
agentplane finish <task-id> --author CODER --body "Verified: ..." --result "..." --commit <git-rev> --no-close-commit
agentplane finish <task-id> --author CODER --body "Verified: ..." --result "..." --commit <git-rev> --close-commit
agentplane finish <task-id> --author CODER --body "Verified: ..." --result "..." --commit <git-rev> --close-commit --close-unstage-others
```

For any commit-capable command (`commit`, `guard commit`, `start/block/finish ... --commit-from-comment`):

- Always pass explicit allowlist prefixes (`--allow` / `--commit-allow` / `--status-commit-allow`).
- Repo-wide prefix `.` is rejected by policy.
- Fast path to build safe prefixes from current index:

```bash
agentplane guard suggest-allow --format args
```

Status transitions (normal mode) are strict:

- `TODO -> DOING|BLOCKED`
- `DOING -> DONE|BLOCKED`
- `BLOCKED -> TODO|DOING`

Status/comment-driven commits are allowed only on major transitions:

- `TODO -> DOING`
- `DOING -> BLOCKED`
- `BLOCKED -> DOING`
- `DOING -> DONE`
