## Purpose

This page defines the public website structure used by the Docusaurus deployment on `agentplane.org`.
It keeps marketing, technical docs, and blog content aligned to one routing model.

## Top-level routes

1. `/` — product narrative and conversion entry point.
2. `/docs` — canonical documentation sourced from repository `docs/`.
3. `/docs/workflow-guides` — copy-paste flows for common coding-agent and review stacks.
4. `/docs/recipes` — installable signed recipe packages.
5. `/blog` — editorial stream for releases, implementation notes, and migration guides.

## Docs navigation model

The docs shell should mirror the same workflow model that `AGENTS.md` and CLI guidance use:

1. **Start**
2. **Work on a task**
3. **Workflow guides**
4. **Recipes**
5. **Upgrade & recover**
6. **Reference & integrations**
7. **Developer**
8. **Release notes**

## Home page section contract

The home page must map each section to concrete docs pages:

1. **Hero**
   Links: GitHub repository, `npm i -g agentplane`, and the 90-second terminal flow.
2. **Problem**
   Explain what task, plan, diff, verification, and finish evidence answer for coding-agent work.
3. **Demo**
   Links: GitHub repository and `docs/workflow-guides`.
4. **Repo artifacts**
   Names: `AGENTS.md` or `CLAUDE.md`, `.agentplane/`, task records, verification records, finish record.
5. **How it works**
   Commands: `agentplane init`, `task new`, `task plan set`, `verify`, `finish`.
6. **Workflow guides**
   Links: `docs/workflow-guides/claude-code`, `docs/workflow-guides/codex`,
   `docs/workflow-guides/cursor`, `docs/workflow-guides/aider`,
   `docs/workflow-guides/github-actions`, `docs/workflow-guides/branch-pr`.
7. **Recipes**
   Links: `docs/recipes`, `docs/recipes/code-map`.
8. **Getting Started**
   Links: `docs/user/overview`, `docs/user/prerequisites`, `docs/user/setup`.
9. **Workflow Model**
   Links: `docs/user/workflow`, `docs/user/task-lifecycle`, `docs/user/agents`.
10. **Reference**
    Links: `docs/user/commands`, `docs/user/configuration`, `docs/reference/generated-reference`.
11. **Developer Track**
    Links: `docs/developer/architecture`, `docs/developer/cli-contract`, `docs/developer/release-and-publishing`.
12. **Support**
    Links: `docs/help/troubleshooting-by-symptom`, `docs/help/glossary`.

## Blog scope

Blog posts should be limited to:

1. Release announcements with migration notes.
2. Workflow deep dives and architecture rationale.
3. Operational guidance for CI, policy, and verification changes.

Posts must link back to canonical docs pages and avoid duplicating full reference material.

## SEO and discovery requirements

1. Every top-level section must have stable title and meta description.
2. Home page claims must link to verifiable docs artifacts.
3. Blog posts must include author, date, tags, and canonical URL.
4. Search and crawler endpoints (`sitemap.xml`, `robots.txt`, `llms.txt`) must be kept in the deploy artifact.
