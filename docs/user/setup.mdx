## Install CLI

```bash
npx agentplane --help
```

Or install globally:

```bash
npm i -g agentplane
# or
bun add -g agentplane
agentplane --help
```

If the shell says `agentplane` is not recognized (common on fresh Windows setup), use:

```bash
npx agentplane --help
```

Quick checks on Windows:

```powershell
where agentplane
npm config get prefix
```

If `where agentplane` returns nothing, reopen the terminal after global install or use `npx agentplane ...`.

## Initialize repository

Run from repository root:

```bash
agentplane init
```

`init` creates:

- `AGENTS.md` in repo root.
- `.agentplane/config.json`.
- `.agentplane/agents/*`.
- `.agentplane/backends/<selected-backend>/backend.json`.
- `.agentplane/tasks/` workspace directories.
- `.agentplane/WORKFLOW.md` (validated contract).
- `.agentplane/workflows/last-known-good.md` (rollback snapshot).

If `.git` does not exist in that folder, `init` creates a new git repository there.

## Interactive choices in init

`agentplane init` now starts with a setup profile:

- `Light`: compact flow, minimum enforcement, maximum flexibility.
- `Normal` (default): balanced approvals and checks for most teams.
- `Full Harness`: strict guardrails and full questionnaire.

Hooks default by profile:

- `Light`: hooks disabled by default.
- `Normal`: hooks enabled by default.
- `Full Harness`: hooks enabled by default.

You can override the default explicitly with `--hooks true|false`.

Use `--setup-profile full-harness` to force the full questionnaire.

In the interactive happy-path, `plan` and `verify` approvals now follow the selected setup profile instead of being asked as separate questions. Keep using `--require-plan-approval` and `--require-verify-approval` only when you want to override the profile defaults explicitly.

Non-interactive setup is available through flags:

```bash
agentplane init \
  --setup-profile normal \
  --workflow direct \
  --backend local \
  --execution-profile balanced \
  --require-network-approval true \
  --hooks true \
  --ide codex \
  --yes
```

Legacy aliases still work and are mapped automatically:

- `vibecoder -> light`
- `manager -> normal`
- `developer|enterprise -> full-harness`

You can switch profile later without re-running init:

```bash
agentplane profile set light
agentplane profile set normal
agentplane profile set full-harness
```

## Redmine env files on init

If `--backend redmine` is selected, `init` writes a template to `.env.example` and creates `.env` when missing.

Credentials are read from env at runtime; secrets are not stored in backend JSON.

## Keep agent files out of git (optional)

```bash
agentplane init --gitignore-agents
```

This appends ignores for `AGENTS.md` and `.agentplane/agents/` and skips the init commit.

## Upgrade a workspace

```bash
agentplane upgrade
```

Default mode applies the managed framework files from the installed CLI bundle and creates a dedicated upgrade commit.

### Managed ownership contract

Treat these files as framework-owned:

- `AGENTS.md`
- `.agentplane/policy/**`
- other managed framework files written by `agentplane init` and `agentplane upgrade`

Treat this file as the sanctioned local override area:

- `.agentplane/policy/incidents.md`

Operational rule:

- record situational directives in `incidents.md`;
- do not treat `incidents.md` as part of the normal startup reading path for agents; consult it only during recovery, incident handling, or direct work on that file;
- do not hand-edit the rest of the managed policy tree in normal project work;
- expect `agentplane upgrade` to replace or reconcile framework-owned files on apply.

### Upgrade state model

Think about upgrade in three states:

1. **Clean managed state**
   `AGENTS.md`, `.agentplane/policy/**`, and other managed framework files match the installed CLI bundle.
   Normal path:

   ```bash
   agentplane upgrade
   ```

2. **Partial upgrade state**
   The gateway or part of the managed tree changed, but not all managed files were updated.
   Typical recovery:

   ```bash
   agentplane doctor
   agentplane upgrade --yes
   ```

3. **Manual drift state**
   Managed files were edited by hand, or the workspace was updated with an older CLI than expected.
   Recovery starts with diagnostics and may require reinstalling/updating the CLI first:

   ```bash
   agentplane doctor
   agentplane upgrade --remote --yes
   ```

- managed files are replace-first on apply, except `.agentplane/policy/incidents.md`, which is append-only when local content already exists;
- upgrade prints the bundle source, resolved version label, and selected mode before doing any work.

Non-default modes:

```bash
agentplane upgrade --dry-run
agentplane upgrade --agent
```

`--dry-run` is the preview path. `--agent` keeps the older review-plan mode without writes.

If a legacy project ends up with a new gateway but missing policy modules after upgrade, use the
[Legacy upgrade recovery](../help/legacy-upgrade-recovery) runbook.

If `agentplane doctor` reports that active tasks still use legacy README v2 docs or that the
workspace is in an active v2/v3 mixed state after an upgrade, normalize task docs before
continuing active work:

```bash
agentplane task migrate-doc --all
agentplane doctor
```

Use this after the managed framework files are already up to date. `agentplane upgrade` updates the
framework-owned files; `task migrate-doc --all` updates task-local README docs to the README v3
contract.

## Preflight modes

Quick preflight is now the default and is optimized for fast readiness checks:

```bash
agentplane preflight
agentplane preflight --json
```

For full backend/task-list probing, use full mode:

```bash
agentplane preflight --mode full
# or
agentplane preflight --full
```

## Verify setup quickly

```bash
agentplane config show
agentplane quickstart
agentplane task list
agentplane workflow build --validate --dry-run
agentplane doctor
```

Then read [Agent bootstrap](agent-bootstrap.generated) for the canonical agent startup path.

## Working inside the Agent Plane framework checkout

If you run `agentplane` from `PATH` inside this framework repository, the global wrapper now
detects the checkout and delegates automatically to the repo-local binary from this checkout.

The repo-local binary still checks whether `dist/` matches the current source tree.

- To inspect which runtime is currently active, run:

```bash
agentplane runtime explain
```

It prints the active binary path, runtime mode, and the resolved `agentplane` /
`@agentplaneorg/core` package roots.

- Inside the framework checkout, a stale build is a hard stop. Rebuild first:

```bash
bun run --filter=@agentplaneorg/core build
bun run --filter=agentplane build
```

- If you intentionally want to keep using the global installed CLI inside the framework checkout,
  set:

```bash
AGENTPLANE_USE_GLOBAL_IN_FRAMEWORK=1
```

- Outside the framework checkout, the linked global binary continues to use its own installed
  package and does not hand off.

The gateway preflight commands are still the correct commands. Inside the framework repository
they now resolve to the repo-local binary automatically and therefore assume a fresh local build.

## Core workflow operations

```bash
agentplane workflow debug
agentplane workflow sync
agentplane workflow land
```

Each command writes evidence to `.agentplane/workflows/evidence/`.

## Install optional recipes (extensions)

```bash
agentplane recipes install --name viewer
agentplane scenario run viewer:viewer
```
