## Task record model

Each task record contains:

- core metadata (`id`, `status`, `owner`, `priority`, `tags`, deps)
- README doc sections
- events and comments
- optional commit metadata

For local backend, records are persisted under:

- `.agentplane/tasks/<task-id>/README.md`

## Source, projection, and snapshot

Task backends now follow one install-first contract:

- **canonical source**: where the authoritative task state lives
- **local projection**: the task state that ordinary CLI reads use
- **sync boundary**: explicit refresh/push operations between projection and canonical source
- **optional export snapshot**: generated tasks JSON for integrations

For `local`, the canonical source and projection are the same repository state.

For `cloud`, the cloud service is the canonical remote source and `.agentplane/` holds the local
projection. Connector setup happens in the cloud service, so AgentPlane still uses the `cloud`
backend id even when the connected view is GitHub Projects, Redmine, or another task system.
Cloud projection refreshes are git-visible: `agentplane backend sync cloud --direction pull` may
update `.agentplane/tasks/**` in the working tree. In `publish_only`, pull does not apply remote
view edits. In `bidirectional`, pull may update only service-approved operational fields
(`title`, `status`, `priority`, `owner`, `tags`) on known local tasks; AgentPlane-owned plan,
verification, rollback, findings, comments, events, and README sections remain local task state.
Remote-only items are ignored unless a future creation policy enables them.

For `redmine`, Redmine is the canonical source and `.agentplane/` holds the local projection.
`task list`, `task show`, and `doctor` read the projection by default; `backend sync <id>
--direction pull` refreshes it from the remote source.

## README contract versions

- `doc_version=2` is the legacy task README layout and is no longer accepted by the current
  runtime validators.
- `doc_version=3` is the only active layout for task READMEs.
- Frontmatter remains a machine-oriented state layer.
- Body sections remain the human working document for planning, verification, and findings.

### README v3 body contract

`doc_version=3` uses this body structure:

- `## Summary`
- `## Scope`
- `## Plan`
- `## Verify Steps`
- `## Verification`
- `## Rollback Plan`
- `## Findings`

Section semantics:

- `## Summary` states the problem, target outcome, and material constraint.
- `## Scope` separates `In scope` from `Out of scope`.
- `## Plan` contains executable task steps only.
- `## Verify Steps` contains acceptance-oriented checks. These may be commands, manual checks, artifact checks, state checks, or review criteria.
- `## Verification` is the append-only result log written by `agentplane verify`.
- `## Rollback Plan` records the shortest safe rollback path.
- `## Findings` records local observations, deviations, follow-up items, and incident candidates.

### Findings vs incidents

- `## Findings` is task-local and may contain raw incident candidates.
- `.agentplane/policy/incidents.md` stores only curated strong incidents with repository-wide value.
- Plain prose in `## Findings` stays task-local and never updates `.agentplane/policy/incidents.md` by itself.
- Structured reusable external findings are promoted during `agentplane finish` or `agentplane incidents collect <task-id>`.

### Frontmatter boundaries

- `plan_approval` stores only the approval state of the plan, not the plan text.
- `verification` stores only the latest verification status, not the full verification log.
- Plan text stays in `## Plan`.
- Detailed verification evidence stays in `## Verification`.

## Verification sections

- `## Verify Steps` is the planned verification contract.
- `## Verification` is the append-only result log written by `agentplane verify`.

## Migrating older projects to README v3

Older repositories may still contain legacy `doc_version=2` task READMEs after the framework files
were upgraded. Migrate those READMEs before running normal task lifecycle commands with the current
runtime.

Use this order:

```bash
agentplane upgrade --yes
agentplane doctor
agentplane task migrate-doc --all
agentplane doctor
```

Operational split:

- `agentplane upgrade --yes` updates framework-managed files.
- `agentplane task migrate-doc --all` is an advanced maintenance command that updates task-local
  README docs to `doc_version=3` and
  refreshes the projection snapshot for the configured backend.
- `agentplane doctor` tells you whether active work is still blocked by a mixed v2/v3 state or
  whether only the historical archive remains on the old format.

## Export snapshot

An optional export-only snapshot can be generated for integrations (gitignored by default).

The snapshot is derived from the current backend projection. It is not the canonical source of truth.
Its exact maintenance command is intentionally hidden from the public happy path; use
`agentplane help --all` when an integration explicitly needs it.

Do not edit generated tasks export snapshots manually.

## Task navigation projection

An optional Markdown navigation projection can be generated for humans and agents:

This writes gitignored Markdown files under `.agentplane/generated/obsidian/` for compatibility
with Obsidian and plain Markdown browsers:

- `.agentplane/generated/obsidian/index.md`
- `.agentplane/generated/obsidian/tasks.md`
- `.agentplane/generated/obsidian/by-status/*.md`
- `.agentplane/generated/obsidian/by-tag/*.md`
- `.agentplane/generated/obsidian/by-owner/*.md`

Open `.agentplane/generated/obsidian/` as an Obsidian vault, or read the files as a lightweight
task wiki, to browse tasks without renaming every task README.
Task IDs link back to `.agentplane/tasks/<task-id>/README.md`, and dependency cells link to the
same canonical README paths.

Remove the generated navigation layer with:

The cleanup command removes only files generated by the Obsidian projection command. It preserves
canonical task READMEs and non-generated Markdown files. Projection generation and cleanup are both
available through the hidden help surface (`agentplane help --all`).

This projection can make manual navigation and agent context gathering faster because a reader can
open or grep a compact grouped index instead of scanning thousands of task README files. It does not
replace backend reads, projection caches, or task README frontmatter as runtime authority.

Conceptually, this is a generated wiki/navigation output adapter over the task projection, not a
separate Obsidian-owned storage model. Legacy generated files under `.agentplane/index.md`,
`.agentplane/tasks.md`, and `.agentplane/by-*` are removed by the advanced cleanup path when they
carry the generated marker.

## Active backend

Backend routing comes from `.agentplane/WORKFLOW.md`:

```json
{
  "tasks_backend": {
    "config_path": ".agentplane/backends/local/backend.json"
  }
}
```

`tasks_backend.config_path` points to backend config (`local`, `cloud`, or another configured
backend).

## Sync

```bash
agentplane sync redmine
agentplane sync redmine --direction pull
agentplane sync redmine --direction push --yes
agentplane backend sync cloud --direction pull
agentplane backend sync cloud --direction push --yes
agentplane backend sync <id> --direction pull
agentplane backend sync <id> --direction push --yes
```

`sync`/`backend sync` default to `--direction push` when omitted.

Conflict modes: `diff`, `prefer-local`, `prefer-remote`, `fail`.

Use explicit `agentplane backend sync` in scripts to avoid ambiguity with other `sync` commands.
