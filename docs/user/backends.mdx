## Overview

A backend defines task storage and sync behavior.

`agentplane` resolves backend config from `.agentplane/WORKFLOW.md` and routes task commands through that backend.

## Available backends

- [Local backend](./backends/local)
- [Cloud backend](./backends/cloud)
- [Redmine backend](./backends/redmine) for direct Redmine setups

## Remote backends

Remote backends are optional and must be configured explicitly.

The local/backend choice is intentionally small:

- `local` keeps the canonical task state in the repository;
- `cloud` connects the project to a cloud sync service and keeps local reads on a projection;
- direct backend integrations such as `redmine` may remain available for explicit setups that need
  them.

## Sync model

Use explicit sync commands:

```bash
agentplane backend sync <id> --direction pull
agentplane backend sync <id> --direction push --yes
```

Remote backends follow the same install-first model as the local backend:

- the backend keeps a local projection under `.agentplane/`;
- ordinary task reads (`task list`, `task show`, `doctor`) use that projection by default;
- `agentplane backend sync <id> --direction pull` refreshes the projection from the remote source;
- `agentplane backend sync <id> --direction push --yes` pushes local dirty state back to the
  remote source;
- advanced export commands can write an integration snapshot from the current projection when an
  integration still needs a local JSON file.

Do not assume ordinary task reads will hit the network implicitly.

Prefer `agentplane backend sync` in docs and automation.

For the cloud backend, connector selection happens in the cloud service. AgentPlane uses `cloud` as
the backend id even when the connected view is GitHub Projects, Redmine, or another task system.

## Conflict strategies

- `diff` — report differences without overwriting.
- `prefer-local` — local state wins.
- `prefer-remote` — remote state wins.
- `fail` — abort when conflicts exist.

## Backend configuration

Backend config path lives in `.agentplane/WORKFLOW.md` at `tasks.backend.config_path`.

### Backend config format (standardized)

Each backend config uses this shape:

```json
{
  "id": "local",
  "version": 1,
  "settings": {
    "dir": ".agentplane/tasks"
  }
}
```

Legacy fields like `module`/`class` are ignored by the loader.
