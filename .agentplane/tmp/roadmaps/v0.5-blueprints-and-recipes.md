# v0.5 Blueprint and Recipe Roadmaps

Status: temporary planning artifact.
Scope: implementation planning for the AgentPlane `0.5` blueprint layer.
Non-goal: this document does not replace `ROADMAP.md` and does not claim shipped behavior.

## Current Baseline

- Blueprint core exists: typed model, built-in blueprints, registry helpers, validation, and focused tests.
- Recipes already exist as project overlays: agents, skills, tools, scenarios, prompt modules, validators, and templates.
- `0.5` should make blueprints visible and useful before runner execution grows more autonomous.
- Recipe work for `0.5` must be narrow: recipes may contribute bounded hints to blueprint resolution, but the resolver remains the authority.

## Product Boundary

Blueprints are task routes. Recipes are advisors. The resolver is the gatekeeper.

Allowed recipe influence:

- context hints;
- evidence requirements;
- deterministic check suggestions;
- output schemas;
- artifact templates;
- risk hints;
- compatible preferred blueprint hints.

Forbidden recipe influence:

- workflow mode override;
- approval bypass;
- protected node removal;
- finish semantics replacement;
- direct task-state mutation;
- network, publish, merge, or deploy authority without core approval;
- converting lightweight analysis/content tasks into a code PR pipeline by default.

## Recipe Roadmap to v0.5

### Epic R1: Recipe Blueprint Extension Contract

Goal: add a formal manifest contract for recipe hints that can safely influence blueprint resolution.

Atomic tasks:

1. Add `RecipeBlueprintExtension` types to `packages/recipes/src/manifest-contracts.ts`.
2. Add allowed extension kinds: `context_hint`, `evidence_requirement`, `check_suggestion`, `output_schema`, `artifact_template`, `risk_hint`, and `preferred_blueprint`.
3. Add forbidden lifecycle override validation for approval bypass, workflow override, protected node removal, finish override, and direct task-state mutation.
4. Extend recipe manifest validation so `blueprint_extensions` is optional and backward-compatible.
5. Add tests for valid extensions, invalid ids, empty summaries, and forbidden override attempts.

Acceptance:

- existing recipe manifests still validate;
- valid blueprint extensions validate;
- lifecycle override attempts fail with actionable errors;
- recipe hints are modeled as suggestions, not execution authority.

### Epic R2: Recipe Hint Normalization

Goal: convert active recipe extension data into deterministic resolver input with provenance.

Atomic tasks:

1. Add `packages/recipes/src/blueprint-extensions.ts`.
2. Collect extensions from active project recipes only.
3. Apply `when` filters using task kind, tags, commands, and repo type.
4. Preserve recipe provenance: recipe id, version, name, extension id, and selection reason.
5. Sort normalized extensions deterministically by recipe id and extension id.
6. Ignore or report incompatible recipe extensions without silently accepting them.

Acceptance:

- inactive recipes do not contribute hints;
- incompatible recipes do not affect resolution by default;
- normalized output is stable across runs;
- resolver can consume recipe hints without reading unrelated recipe internals.

### Epic R3: Recipe Hint Safety Tests

Goal: prevent recipe hints from becoming a second policy language.

Atomic tasks:

1. Add tests where a recipe adds analysis evidence requirements.
2. Add tests where a recipe suggests a deterministic local check for docs/code tasks.
3. Add tests where a recipe tries to disable approval gates.
4. Add tests where a recipe tries to force `code.branch_pr` for read-only analysis.
5. Add tests where two active recipes contribute compatible evidence hints.
6. Add tests where incompatible/conflicting recipes are surfaced as rejected input.

Acceptance:

- safe hints are accepted;
- unsafe hints are rejected with reasons;
- analysis/content tasks do not inherit CI/PR ceremony through recipe hints;
- rejection data is available for explain output.

### Epic R4: Recipe Authoring Documentation

Goal: make the recipe-to-blueprint contract understandable for recipe authors.

Atomic tasks:

1. Update `docs/developer/recipes-spec.mdx` with `blueprint_extensions`.
2. Add examples for analysis, docs, and code-oriented recipes.
3. Document allowed and forbidden extension kinds.
4. Document that recipe hints are accepted or rejected by the blueprint resolver.
5. Add a fixture-style manifest example that can later be reused in tests.

Acceptance:

- recipe authors can see how to add evidence/context/check hints;
- docs explicitly state that recipes cannot override protected lifecycle gates;
- examples match the implemented schema.

## Blueprint Roadmap to v0.5

### Epic B1: Blueprint Resolver

Goal: select a blueprint for a task from task metadata, workflow mode, mutation class, risk, explicit requests, and recipe hints.

Atomic tasks:

1. Add `packages/agentplane/src/blueprints/resolve.ts`.
2. Define `BlueprintResolveInput` with task kind, workflow mode, mutation class, risk, tags, explicit blueprint id, and recipe extensions.
3. Define `ResolvedBlueprint` with blueprint, active route, selection reasons, required evidence, accepted extensions, rejected extensions, stop reasons, and not-required gates.
4. Implement the default selection matrix:
   - read-only analysis -> `analysis.light`;
   - read-only content -> `content.light`;
   - docs mutation -> `docs.change`;
   - code mutation in `direct` -> `code.direct`;
   - code mutation in `branch_pr` -> `code.branch_pr`;
   - release/publish risk -> `release.strict`;
   - external-system or approval-heavy operations -> `ops.approval`.
5. Treat explicit blueprint ids as validated requests, not absolute commands.
6. Merge safe recipe hints into resolved evidence/context/check/output data.
7. Reject unsafe recipe hints with stable reasons.

Acceptance:

- resolver is pure and unit-tested;
- non-code tasks do not receive CI/PR nodes by default;
- workflow mode constrains mutation topology;
- recipes can add bounded hints but cannot bypass protected gates.

### Epic B2: Blueprint Explain CLI

Goal: expose selected routes before execution or verification.

Atomic tasks:

1. Add `agentplane blueprint list`.
2. Add `agentplane blueprint explain <task-id>`.
3. Add synthetic explain input, for example `agentplane blueprint explain --kind analysis --mutation none`.
4. Add `--json` output for machine-readable use.
5. Print route, workflow mode, selection reasons, required evidence, accepted recipe extensions, rejected recipe extensions, stop reasons, and not-required gates.
6. Add snapshot or structured output tests.

Acceptance:

- explain performs no task mutation;
- analysis explain explicitly says worktree, PR, hosted checks, and CI are not required;
- code branch-pr explain shows worktree, fast local checks, PR artifact, hosted checks, and integration evidence;
- JSON output is parseable and stable enough for downstream tooling.

### Epic B3: Task Evidence Surface

Goal: make expected blueprint evidence visible in existing task workflows without adding runner execution.

Atomic tasks:

1. Include resolved blueprint summary in `task verify-show`.
2. Show expected evidence for the selected route.
3. Keep non-code tasks free from CI/PR expectations.
4. Optionally write `.agentplane/tasks/<task-id>/blueprint.json` as a snapshot artifact.
5. Include accepted/rejected recipe extension summaries in the snapshot.
6. Avoid storing full prompts or transcripts.

Acceptance:

- `verify-show` displays expected evidence for resolved blueprints;
- code PR tasks show fast local checks and PR evidence;
- analysis/content tasks show sources, assumptions, confidence, weak links, and final artifact expectations;
- blueprint snapshot can be regenerated or compared.

### Epic B4: Minimal ACR Bridge

Goal: connect blueprint resolution to audit evidence without rewriting ACR or introducing runner coupling.

Atomic tasks:

1. Add optional ACR fields for blueprint id and version.
2. Add optional route summary.
3. Add expected evidence summary.
4. Add accepted/rejected recipe extension summary.
5. Add stop reason when resolution stops or requires approval.
6. Keep transcript and prompt capture out of scope.

Acceptance:

- task closure can reference the selected blueprint;
- auditors can see route, expected evidence, and stop reasons;
- ACR remains lightweight and does not become a prompt archive.

### Epic B5: Documentation and Release Readiness

Goal: make `0.5` shippable as a visible harness layer rather than an internal library.

Atomic tasks:

1. Update `docs/developer/blueprints.mdx` from implementation reality.
2. Update recipe docs to match the implemented extension contract.
3. Add release note draft for `0.5`.
4. Review public/internal export boundaries for blueprint resolver and recipe extension types.
5. Update generated reference if CLI commands are added.
6. Run package/install smoke for `agentplane blueprint explain`.

Acceptance:

- docs do not claim runner execution;
- release notes describe resolver and explain surfaces as the first user-visible layer;
- package includes blueprint modules and CLI commands;
- install smoke proves explain works outside the source checkout.

## Suggested PR Sequence

1. PR R1: recipe blueprint extension schema and validation.
2. PR R2: recipe hint normalization and provenance.
3. PR B1: blueprint resolver with recipe hint acceptance/rejection.
4. PR B2: `agentplane blueprint list/explain`.
5. PR B3: `task verify-show` expected evidence and optional `blueprint.json`.
6. PR B4: minimal ACR bridge.
7. PR B5/R4: docs, examples, release notes, generated reference, and install smoke.

## Minimum Bar for v0.5

`0.5` should not ship only internal blueprint types. It should ship the first visible harness behavior:

- resolver selects a route;
- recipes can contribute bounded hints;
- explain shows why the route was selected;
- verify surfaces expected evidence;
- non-code tasks remain lightweight;
- code tasks can use development gates without making the code PR pipeline the default for all work.

## Deferred Beyond v0.5

- full runner executor;
- visual workflow editor;
- arbitrary recipe-defined lifecycle graphs;
- automatic policy-module loading enforcement;
- cross-repository orchestration;
- full prompt or transcript capture in ACR;
- replacing `direct` or `branch_pr` workflow modes.

## Weakest Link

The main risk is allowing recipe hints to become hidden policy. The implementation must keep the authority boundary explicit:

- recipes propose;
- resolver accepts or rejects;
- protected lifecycle remains owned by AgentPlane;
- explain output makes every accepted and rejected contribution visible.
